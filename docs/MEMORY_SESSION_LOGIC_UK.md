# Як працює памʼять сесій (підсумки, промт, прапори) — UA

Коротко й по суті: коли створюються підсумки сесій, як працюють `LUMI_SESSION_DAYS`,
`LUMI_SESSION_DETAIL_N` і `LUMI_SESSION_FORMAT`, що відбувається при старті/завершенні сесії та на
кожному ході.

> **Головна ідея (одне речення):** підсумок сесії створюється **рівно один раз — коли сесія
> завершується**; кожен наступний хід лише **читає** вже збережені підсумки. Прапори нічого
> **не перегенеровують** — вони лише вирішують, **скільки** сесій додати й у **якій формі**.

---

## 1. Коли запускається підсумовування

Підсумок сесії — це запис `ShortSummary{user_id, session_id, summary, gist, ts}`, де:
- **`summary`** — детальний підсумок (довгий),
- **`gist`** — однорядковий «заголовок» (рядок після «СТИСЛО:»).

**Обидва рівні створюються одним викликом моделі** у `Core.end_session()`. Це стається, **коли сесія
завершується**:

| Тригер | Що це |
|---|---|
| Вихід із TUI | `Ctrl+Q` / `Ctrl+C` |
| `/new` | нова сесія; попередня зберігається |
| Аварійне завершення | запасний шлях `on_unmount` |

- Під час **звичайного повідомлення в чаті підсумовування НЕ відбувається.**
- Збережений підсумок лягає у `store.json` і **більше ніколи не перегенеровується.**

---

## 2. Як використовуються `LUMI_SESSION_DAYS` + два нові прапори (v0.35)

Усі три діють на етапі **збирання промту** (кожен хід), а **не** на генерації.

| Прапор | Типово | Що означає |
|---|--:|---|
| `LUMI_SESSION_DAYS` | 2 | **Вікно в днях**: які закриті сесії взагалі можуть потрапити у блок `## Останні розмови` (ті, що закрились за останні N днів). |
| `LUMI_SESSION_DETAIL_N` | *(не задано = всі)* | **СКІЛЬКИ** найновіших сесій із цього вікна додати: не задано → **усі**; `0` → **жодної**; `N` → **останні N**. |
| `LUMI_SESSION_FORMAT` | `summary` | **У ЯКІЙ ФОРМІ** показати кожну додану сесію: `summary` (повний) або `gist` (один рядок). |

**Два прапори — незалежні** (кількість × форма). Типово (обидва не задано) = **усі сесії, повним
`summary`** → як було до v0.35.

### Що поставити, щоб отримати потрібне

| Хочу | Постав |
|---|---|
| **усі сесії гістами** (короткий промт) | `LUMI_SESSION_FORMAT=gist` |
| останні 5, гістами | `LUMI_SESSION_DETAIL_N=5` + `LUMI_SESSION_FORMAT=gist` |
| останні 2, повними | `LUMI_SESSION_DETAIL_N=2` |
| жодної сесії | `LUMI_SESSION_DETAIL_N=0` |
| як було (без змін) | нічого не задавати |

> ⚠️ `LUMI_SESSION_DETAIL_N` — це **кількість сесій**, а **не** «кількість рядків у гісті». Зробити сесію
> «на 10 рядків» неможливо: у сесії є рівно **дві** форми — повний `summary` і однорядковий `gist`.

---

## 3. Що відбувається при **завершенні** та **старті** сесії

**Завершення сесії (`end_session`)** — один раз: позначити закритою → згенерувати `summary` + `gist`
(один виклик моделі) → витягти факти → **зберегти** у `store.json`.

**Старт сесії (`start_session`)** — нову сесію **не** підсумовуємо. Просто відкривається порожня сесія; її
повідомлення йдуть у «живе вікно».

---

## 4. Що виконується на **кожному ході** (turn)

**Жодного підсумовування сесій.** На кожне твоє повідомлення:

1. **Читання** збережених підсумків (`summaries_since`) → збирання блоку `## Останні розмови`: береться
   **останні `LUMI_SESSION_DETAIL_N`** сесій (або всі), і для кожної — поле `summary` **або** `gist`
   (за `LUMI_SESSION_FORMAT`). Це **вибір уже наявних полів**, без виклику моделі.
2. Підтягуються вже збережені **денні/тижневі** дайджести.
3. **Авто-RAG**: вхідне повідомлення кодується у вектор → дістаються релевантні фрагменти **минулого**.
4. **Живе вікно**: останні повідомлення **поточної** сесії — дослівно.
5. **Один** виклик моделі — власне відповідь Лілі.
6. Наприкінці нове повідомлення індексується у вектор (для майбутнього RAG).

---

## 5. Що ще важливо

- **Генерація vs вибір.** Генерація (`summary` + `gist`) — один раз, при закритті. Вибір (скільки/яка форма)
  — кожен хід, без перегенерації й без запису в `store.json`.
- **Поточна сесія ніколи не «гіститься»** — вона завжди дослівно в живому вікні. Прапори стосуються лише
  **минулих (закритих)** сесій.
- **Деталі не втрачаються.** Навіть коли сесія показана гістом (чи взагалі прибрана з тіра), її дослівні
  повідомлення дістаються через **авто-RAG** (push) або **`recall`** / **`messages_on`** /
  **`messages_between`** (pull).
- **Денні/тижневі дайджести — окрема історія.** Їхній **формат** керує `LUMI_MEMORY_INDEX` (v0.34); після
  зміни формату треба **`/regen-summaries`**. *(Для сесійних `summary`/`gist` — ні: обидва вже збережені.)*
- **`.env` читається ОДИН раз при старті** → після зміни будь-якого прапора **перезапусти `./lumi`**.

## Приклад: перша сесія `[2026-06-21]`

Та довга сесія `[2026-06-21] Розмова почалася з тестування… (RAG)` має у `store.json` **два готові поля**
(обидва записані при її закритті):

| Поле | Розмір |
|---|--:|
| `summary` | ~1376 символів (детальний абзац) |
| `gist` | 1 рядок (~91 символів) |

- Якщо `LUMI_SESSION_FORMAT=summary` (типово) **і** ця сесія потрапляє у `LUMI_SESSION_DETAIL_N` найновіших →
  у промт іде її **повний `summary`** (той самий ~1376-символьний абзац).
- Якщо `LUMI_SESSION_FORMAT=gist` → у промт іде її **`gist`** (1 рядок).
- Якщо вона старіша за `LUMI_SESSION_DETAIL_N` найновіших → **взагалі не додається**.

**Коли:** перезапуск `./lumi` → на наступному ході. **Як:** код просто бере потрібне **вже наявне** поле з
`store.json` — без виклику моделі, без «генерації N рядків», без запису.

> Щоб змінити **довжину самого `summary`**, його треба **перегенерувати** — а це стається **тільки** коли
> та сесія закривається (`/new` / вихід), не від зміни прапора.

---

## Часті питання (швидко)

- **«Чи перегенерується підсумок, коли я змінюю прапор?»** → **Ні. Ніколи.** Просто вибирається інша
  кількість/форма вже збережених полів.
- **«Чи підсумовуються сесії на кожному повідомленні?»** → **Ні.** Тільки **читаються**.
- **«Як зробити всі сесії гістами?»** → `LUMI_SESSION_FORMAT=gist` (і не задавати `LUMI_SESSION_DETAIL_N`,
  щоб узяти всі), потім перезапуск.
- **«`LUMI_SESSION_DETAIL_N=0`?»** → блок `## Останні розмови` зникає (жодної сесії).
