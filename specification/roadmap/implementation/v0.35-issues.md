# v0.35 — GitHub Issues

Issues for phase **v0.35 — Lean memory II: gist the conversation tier** (version **v0 — TUI**), derived from
[ROADMAP.md](../../ROADMAP.md) (§v0.35) and
[PROMPT_OPTIMIZATION_II.md](../../../docs/PROMPT_OPTIMIZATION_II.md) (P1). IDs continue from v0.34
(LUMI-138 → **LUMI-139…140**).

The **detailed-conversations** block (`## Останні розмови (детально)`) is the single biggest slice of the
system prompt (~26% — the last *N* sessions injected **verbatim**, every turn). Extending v0.34's
index-vs-pull pattern to its largest and **riskiest** tier: keep only the **most recent** conversation in full
(the live thread) and render the rest of the `LUMI_SESSION_DAYS` window as their one-line **`gist`** (a dated
index) — the older verbatim text is **retrieved on demand** by the v0.17 auto-RAG "push" + the v0.31 `recall`
/ `messages_between` "pull", not re-sent each turn.

**No new seam** — it reuses the shipped session store, the v0.9 per-session `gist`, v0.17 auto-RAG, and the
v0.31 pull tools; it only changes **how many sessions render verbatim** in `Core._system_prompt` (the
`summaries` list built from `summaries_since` → injected by `core/prompt.py`). Today every session in the
window injects its detailed `summary`; this phase injects the last `LUMI_SESSION_DETAIL_N` as `summary` and
the rest as their dated `gist`.

**Hard rules (unchanged):** never drop a session to **zero** — drop it to **its dated gist** (so she still
knows it exists and can `messages_on(date)` it); the **immediate thread** (the most recent conversation)
stays verbatim so live continuity is intact; **`N` = the full window → byte-identical to today** (opt-in);
per-user isolation + the `{reply, emotion, intensity}` contract untouched. **No paid calls** in tests
(embedder + model mocked).

Depends on **v0.34** (the index-vs-pull pattern + flags + the reconstruction test), **v0.9** (the per-session
gist), **v0.17** (auto-RAG push + dedup), **v0.31** (`recall` + `messages_between` pull).

## v0.35 — Lean memory II: gist the conversation tier

### LUMI-139 — Gist the conversation tier

**Description:**
The `## Останні розмови (детально)` tier injects every session in the `LUMI_SESSION_DAYS` window as its
**detailed `summary`** (~26% of the prompt). Cap the verbatim part to the **last `LUMI_SESSION_DETAIL_N`**
conversations and render the rest of the window as their one-line **`gist`** (a dated `[date] gist` index she
can `messages_on(date)` into). The most recent conversation always stays verbatim (live continuity). (`/core`.)

**What needs to be done:**
- In `Core._system_prompt` (where `summaries` is built from `summaries_since`, ~`core/agent.py:1317`), split
  the window: the **last `N`** sessions → `[date] {s.summary}` (verbatim); the **older** sessions → `[date]
  {s.gist}` (the dated index). **Never drop a session** — if a session has no `gist` (pre-v0.9), fall back to
  a truncated `summary`.
- Keep the list **dated + ordered** (chronological) so the gisted entries carry the `messages_on(date)` /
  `messages_between` key.
- Add **`LUMI_SESSION_DETAIL_N`** (config → `Config` → `Core`), **default = the full window** (a large
  sentinel) so the change is **opt-in** and byte-identical until set; document it in `.env.example`.
- The prompt builder (`core/prompt.py`, `## Останні розмови (детально)`) renders the list as-is — no builder
  change beyond the (still-detailed) header wording if needed.

**Dependencies:** none new (over the shipped v0.9 session summaries + gist).

**Expected result:**
With `LUMI_SESSION_DETAIL_N` small, only the last *N* conversations inject verbatim; the rest of the window
renders as a **dated gist index** (~−20–26% of the prompt). The most recent conversation stays verbatim. `N`
≥ the window size → byte-identical to today.

**Acceptance criteria:**
- [ ] The builder injects the last `N` sessions as `summary` and the older ones as their dated `gist`; the most recent conversation is always verbatim.
- [ ] `LUMI_SESSION_DETAIL_N` (default = full window) gates it; **`N` ≥ window → byte-identical** to today (a unit test pins this).
- [ ] A gisted session is **never empty** (gist, or a truncated summary fallback); entries stay dated + ordered.
- [ ] Per-user isolated (the window is the requesting user's sessions only).
- [ ] Integration: a representative dump's conversation tier shrinks **~−20–26%** with small `N`; identical with `N` = full. **No paid calls** (model mocked).

### LUMI-140 — Auto-RAG covers the gisted window + the reconstruction test

**Description:**
When a session leaves the verbatim tier (becomes a gist), its lines must still be **reachable** — the v0.17
auto-RAG "push" should now surface them (they're no longer verbatim in the prompt), and `recall` /
`messages_between` "pull" them on demand. The key correctness point: **auto-RAG dedup must key off the
*verbatim* tier**, not the gisted index — otherwise a gisted session's relevant line is wrongly deduped out
and becomes invisible. (`/core`.)

**What needs to be done:**
- Make the auto-RAG **dedup set** (the v0.17 block's "already in the prompt" guard) include only the **live
  window + the verbatim (last-`N`) sessions**, **not** the gisted entries — so a relevant line from a
  now-gisted session is surfaced by auto-RAG instead of suppressed.
- Confirm the **auto-RAG floor / top-K** still returns a relevant line from a gisted older session (the push
  path covers what left the verbatim tier).
- The **reconstruction** path: a reference to an older (gisted) conversation's detail is served either by
  auto-RAG (unprompted) **or** by `recall(query)` / `messages_between(after, before)` (she pulls it).

**Dependencies:** LUMI-139 (the gisted tier must exist).

**Expected result:**
With the conversation tier gisted, a turn that needs an **older** conversation's verbatim detail still gets it
— auto-RAG surfaces the line (because dedup no longer suppresses gisted sessions), or `recall` /
`messages_between` fetches it — and the answer is correct. No double-injection of a verbatim session's lines.

**Acceptance criteria:**
- [ ] Auto-RAG dedup keys off the **verbatim** tier + live window only; a gisted session's relevant line is **surfaced**, not suppressed (a unit test on the dedup set).
- [ ] **Reconstruction (contract):** gist an older session, plant a reference, assert auto-RAG **or** a pull tool (`recall` / `messages_between`) serves the detail and the answer is correct.
- [ ] No double-injection: a still-verbatim session's lines are **not** re-surfaced by auto-RAG (dedup holds for the detailed tail).
- [ ] Isolation over the gisted + pulled sessions (A↔B — retrieval never crosses users).
- [ ] **No paid calls** (embedder + model mocked).
