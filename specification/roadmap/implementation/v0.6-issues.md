# v0.6 — GitHub Issues

Issues for phase **v0.6 — Mood of the day (temperament)** (version **v0 — TUI**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v0.6) and the contract in [ARCHITECTURE.md](../../ARCHITECTURE.md) (§Mood and temperament). This file is scoped to a single phase; IDs continue from v0.5 (LUMI-024 → **LUMI-025…027**).

v0.6 gives Лілі a **mood of the day** — a daily backdrop that colors which emotions and tone she trends toward, **never her competence**. After a real-ephemeris test (skyfield) confirmed the model writes a vivid, useful daily reading but **cannot compute accurate transits**, the design uses the **model itself, no astronomy engine** (precision isn't the goal — daily *variation* is). Once per local day the core makes a **mood call** (through the `LLMClient` seam) that writes a horoscope-flavored reading from Лілі's fixed natal chart + today's date, ending in a short **resolution** (what she'll want / won't want / her mood, energy, tone). The **full reading is logged**; **only the resolution** is injected into the system prompt — as a **prominent, prioritized block** (mirroring the v0.5 style header) — so it actually colors the turn. It rides the **locked v0.3 emotion channel** (the model still emits `{reply, emotion, intensity}`, the core still validates it) and reuses the **v0.4 injected clock** for the once-per-day cache. **On by default.** A **`/mood`** command shows the current resolution.

## v0.6 — Mood of the day

### LUMI-025 — Daily mood engine: natal chart + mood call + once-per-day cache + logged reading

**Description:**
The mood subsystem's data layer. A **fixed natal chart** + a **once-per-local-day mood call** (through the `LLMClient` seam) that writes a horoscope-flavored reading from the natal chart + today's date, ending in a short **resolution**. Cached per local day (the v0.4 injected clock), recomputed at local midnight; the **full reading is logged**, the **resolution** is held for injection. **No astronomy engine; mock the model in tests.** (`/core`.)

**What needs to be done:**
- A **fixed natal snapshot** in config (e.g. `core/natal.md` or a `NatalChart` constant), seeded from the canon's natal section (verified accurate against a real ephemeris) — date/time/place + positions.
- A **mood-call** prompt + a `Core` path (through the `LLMClient` seam, like the housekeeping/summary call): once per local day, ask the model for a detailed reading + a short **resolution** (wants / doesn't want / mood / energy / tone). **Parse the resolution** out of the reading.
- A **once-per-local-day cache** keyed by the local date from the **injected clock**; recomputed at local midnight; a turn keeps the mood it started with even across a day boundary.
- **Log the full reading** (keyed by date) — never shown to the user, never in the prompt.
- Expose the current **resolution** (e.g. `Core.mood`) for the prompt injection (LUMI-026) and `/mood` (LUMI-027).
- **On by default** with an off-switch (`LUMI_MOOD`); a model/clock failure → **no mood** (`None`), never blocks a turn.

**Dependencies:** None hard (builds on the **locked** v0.3 emotion channel + the v0.4 injected clock).

**Expected result:**
A cached daily resolution (different per day, stable within a day), the full reading in the logs, degrading gracefully when off or on failure.

**Acceptance criteria:**
- [ ] A fixed natal snapshot in config, seeded from the canon.
- [ ] A once-per-local-day mood call (`LLMClient` seam) → reading + resolution; **cached per local date** (injected clock), recomputed at local midnight; a turn keeps its mood.
- [ ] The **full reading is logged** (keyed by date); the **resolution** is parsed + held.
- [ ] **On by default** (`LUMI_MOOD` off-switch); a mood-call failure → no mood, never raises.
- [ ] Unit tests (**mock model + fixed clock**): cache stable within a day, recompute across local midnight; resolution extraction; full reading logged; failure degrades.

### LUMI-026 — Inject the resolution into the system prompt as a prominent block

**Description:**
Add the day's **resolution** to the system prompt as a **prominent, prioritized "today's mood" block** (mirroring the v0.5 style-header importance directive), biasing the emitted `emotion`/`intensity` and tone — **never competence**. **Only the resolution** rides in the prompt (the full reading stays logged). Graceful when there's no mood. (`/core`.)

**What needs to be done:**
- `build_system_prompt(…, mood=…)` injects the resolution as a **prominent prioritized block** (an importance directive, like the v0.5 style header), placed to actually color the turn.
- `Core` threads the cached resolution from LUMI-025 into `_system_prompt`.
- The block frames the resolution as **background that biases emotion + tone, never competence** (the model still emits `{reply, emotion, intensity}`, the core still validates it).
- No mood (off / failed / not yet computed) → **no block**; the canon-verbatim behavior is preserved.

**Dependencies:** LUMI-025 (the cached resolution).

**Expected result:**
On a normal day a turn's system prompt carries the resolution as a prominent block that shifts her tone/emotion; with no mood there's no block.

**Acceptance criteria:**
- [ ] `build_system_prompt(mood=…)` injects the resolution as a **prominent prioritized block**; off/empty → no block.
- [ ] `Core` threads the cached resolution; **only the resolution** is in the prompt (never the full reading).
- [ ] The block biases emotion + tone, framed as **never affecting competence**.
- [ ] Integration test (mock model + fixed clock): a turn's system prompt carries the resolution as a prominent block; no-mood → none; the full reading is absent from the prompt.

### LUMI-027 — `/mood` command (+ optional status line)

**Description:**
A **`/mood`** command shows Лілі's current day resolution on demand; optionally the mood also appears in the small TUI status line. (`/tui`.)

**What needs to be done:**
- A **`/mood`** command (TUI) that shows the current day's **resolution** (the held one from LUMI-025); a friendly line when no mood is available yet.
- (Optional) show the mood in the small TUI **status line**.

**Dependencies:** LUMI-025 (the current resolution).

**Expected result:**
Typing `/mood` prints today's resolution; no-mood shows a clear message.

**Acceptance criteria:**
- [ ] `/mood` shows the current day's resolution; no-mood → a clear message.
- [ ] (Optional) the status line shows the mood.
- [ ] Integration test: `/mood` renders the resolution (mock model + fixed clock).
