# Version v0.39 — Issue Breakdown

**Phase:** v0.39 — Gemini engine: Google Gemini (3.1 Pro) as a switchable backend

Add **Google Gemini** as a third frontier engine behind the one `LLMClient` seam — chat + the structured
emotion field + the **function-calling tool-loop** + **thinking → the think-box** — switchable via `/model`
like Opus 4.8 ↔ GPT-5.5. It **reuses the Gemini plumbing already in the repo** (`GEMINI_API_KEY`, the stdlib
`urllib` caller, the `generateContent` endpoint — proven by the v0.23 image tool + v0.27 web lookup), so the
transport is essentially free. A **port, not new design** — it mirrors the v0.37 OpenAI tool-loop onto
Gemini's wire format. **Two risks are designed in up front** (the gpt-5.5 lesson): **safety filters** on an
intimate companion (a go/no-go **probe** is the first issue) and the **structured-output-vs-tools conflict**
(the loop never sends `responseSchema` + tools in one request). The Anthropic path is **untouched**; off →
unchanged. Full design + risk analysis: [GEMINI_ENGINE.md](../../../docs/GEMINI_ENGINE.md).

### LUMI-151 — Gemini safety probe (go/no-go)

**Description:**
Gemini runs content-safety classifiers that can **block** an intimate companion reply (returning no text,
`finishReason: SAFETY`). Before investing in the port, confirm cheaply that Лілі's register (tender /
vulnerable / the closeness levels) survives the most permissive `safetySettings`. This is the **gate**: a
no-go here stops the phase.

**What needs to be done:**
- A small **`scripts/gemini_probe.py`** (stdlib `urllib`, like `imagegen.py`): POST a representative
  tender-register Лілі prompt to `generateContent` with `safetySettings: BLOCK_NONE` on the disablable
  categories; print the returned text **or** the `finishReason`/block reason.
- Run it **manually** with a real `GEMINI_API_KEY` + the actual model id (`gemini-3.1-pro` — verify it) and
  **record the decision** (go / no-go) in [GEMINI_ENGINE.md](../../../docs/GEMINI_ENGINE.md) (a short result note).
- A **mock-transport unit test** of the request shape (`safetySettings` present, the prompt + model wired) —
  **no paid calls in CI**; the live run is operator-only.

**Dependencies:** none — reuses the existing `urllib` + `GEMINI_API_KEY`.

**Expected result:**
A clear go/no-go signal: Gemini returns clean text for an intimate Лілі prompt (go), or it sanitises/blocks
(no-go → reconsider the phase) — at the cost of one script, not the whole port.

**Acceptance criteria:**
- [ ] `scripts/gemini_probe.py` builds the `generateContent` request with `safetySettings` + the prompt; runnable with a real key.
- [ ] A **mock-transport test** asserts the request shape (no paid calls).
- [ ] The go/no-go result is recorded in `GEMINI_ENGINE.md`.

### LUMI-152 — `GeminiClient`: chat + structured emotion + wiring

**Description:**
The base `GeminiClient` behind the `LLMClient` seam — `reply` / `reply_structured` via `generateContent`
(stdlib `urllib`, an injected `_transport` for tests, like `MiniMaxClient`), the structured
`{reply, emotion, intensity}` via Gemini's JSON mode, safety settings + graceful block-degrade, and the
`build_llm` / `/model` wiring. The "works, tool-less" base (the v0.18 OpenAI-base analogue); the tool-loop +
thinking land in LUMI-153/154.

**What needs to be done:**
- **`GeminiClient`** (`core/llm.py`): `reply` / `reply_structured` → `generateContent`; build `contents`
  (role translation `assistant`→`model`) + top-level `systemInstruction`; `generationConfig.maxOutputTokens`;
  `last_stats` from `usageMetadata` (`promptTokenCount`/`candidatesTokenCount`/`cachedContentTokenCount`).
- **Structured output:** `generationConfig.responseMimeType="application/json"` + `responseSchema` (the
  `{reply, emotion(enum), intensity, relation?}` shape) → parse the text part → `parse_emotion_json` → the
  v0.3 gate.
- **Safety + graceful degrade:** `safetySettings: BLOCK_NONE` (disablable categories); a blocked/empty
  candidate (`finishReason: SAFETY`, no text) returns `{"reply": ""}` → the gate fills `calm` (never
  crashes/hangs).
- **Wire `build_llm`:** `provider="gemini"` (urllib + `GEMINI_API_KEY`) + `KNOWN_PROVIDERS += ("gemini",)`;
  a default `/model` alias (`gemini-3.1-pro`, from config). Config: the Gemini model id + key already exist.

**Dependencies:** gate on **LUMI-151** (build after the probe says go); reuses v0.18 (the provider seam),
v0.23/v0.27 (the existing Gemini `urllib` caller + key).

**Expected result:**
On `LUMI_PROVIDER=gemini` + `LUMI_MODEL=<id>`, a turn returns a valid `{reply, emotion, intensity}` (the v0.3
gate validating); `/model gemini-3.1-pro` ↔ `/model opus` swaps mid-session; a safety-blocked response
degrades to a reply, never a crash.

**Acceptance criteria:**
- [ ] `reply_structured` → a valid `EmotionState` via the JSON schema; `last_stats` from `usageMetadata`.
- [ ] A **blocked/empty candidate** degrades to the gate's `calm` (never raises); `safetySettings` are sent.
- [ ] `build_llm` builds `GeminiClient` for `provider="gemini"`; `/model` swaps to/from `gemini`; the **Anthropic path is unchanged**.
- [ ] **No paid calls** (a mock `_transport` scripts `generateContent`).

### LUMI-153 — Gemini tool-loop (function-calling port)

**Description:**
Give `GeminiClient` the bounded tool-loop via **Gemini function calling**, mirroring the v0.37 OpenAI port,
so the file / wiki / news / web / journal / image tools + `%`-thought-tools work on Gemini. Bakes in the
**schema-vs-tools split** (Gemini restricts `responseSchema` + tools in one request — the gpt-5.5 lesson).

**What needs to be done:**
- **`_to_gemini_tools`** — Lumi's Anthropic-shaped tools → `tools:[{functionDeclarations:[{name, description,
  parameters}]}]`.
- **`_tool_loop`** (+ a `_text_tool_loop` twin for the think path): intermediate rounds offer tools and **no**
  `responseSchema`; the **forced final round** drops tools and sets `responseSchema` (always terminates).
  **Terminal** = a turn with **no `functionCall`** parts → parse the text as the emotion JSON.
- **Result framing:** a `functionResponse` part wrapping the result **untrusted** (recall → **recollection**).
- **Image divergence:** a tool returning an image → a `functionResponse` ack **plus** a follow-up `user` turn
  with an `inlineData:{mimeType,data}` part (Gemini takes images inline, not in a `functionResponse`).
- **Parallel `functionCall`s** — execute all; append one `functionResponse` part each.

**Dependencies:** **LUMI-152** (the base client it extends); mirrors v0.19/v0.37 (the loop it ports).

**Expected result:**
On Gemini, a turn that needs a file/wiki tool **fires it** and the bounded loop terminates with a valid
`{reply, emotion, intensity}`; no request ever sends `responseSchema` + tools together.

**Acceptance criteria:**
- [ ] The loop **executes tools** (`tool_executor` called) and frames the `functionResponse` **untrusted** (recall → **recollection**).
- [ ] The **forced final round** sets `responseSchema` + drops tools and terminates; **parallel `functionCall`s** all execute; the **image divergence** is handled (`inlineData` follow-up turn).
- [ ] The **schema-vs-tools split** holds (no request sends both); the **no-tools** path is unchanged.
- [ ] **No paid calls** (mock `_transport`).

### LUMI-154 — Gemini thinking → the think-box

**Description:**
Surface Gemini's reasoning into the v0.38 think-box: `thinkingConfig:{includeThoughts:true}` → the `thought`
parts → `last_thinking` (the inner-voice seam — the three-voice torg would show). A plus over OpenAI: Gemini
surfaces thought summaries more readily than the Responses API's often-withheld `reasoning.summary`.

**What needs to be done:**
- **`thinkingConfig`** in `generationConfig` (`includeThoughts: true` + a budget for 3.x); join the response
  `parts` flagged `thought: true` → `last_thinking`; set `self._thinking` so the status bar shows
  `thinking: on`.
- **`LUMI_EFFORT` → a thinking budget** (map the Lumi tiers to a Gemini thinking budget; omit when unset).
- **Docs:** a **Gemini section** in [MODELS_SETUP.md](../../../docs/MODELS_SETUP.md) — install-free (the key
  already exists), the safety + privacy notes, the think-box behaviour.

**Dependencies:** **LUMI-152** (the base client); composes with LUMI-153 (the loop carries the thinking
config per round); pairs with v0.38 (the inner-voice / think-box seam).

**Expected result:**
On Gemini, a reply's reasoning **surfaces in the think-box** (and the `lumi.think` log); `LUMI_EFFORT` tunes
the thinking budget; the status bar shows `thinking: on`.

**Acceptance criteria:**
- [ ] `thinkingConfig.includeThoughts` is sent; `thought` parts → `last_thinking`; `_thinking` reflects it (status bar).
- [ ] `LUMI_EFFORT` maps to a thinking budget (omitted when unset).
- [ ] `MODELS_SETUP.md` documents the Gemini engine (safety + privacy + think-box).
- [ ] **No paid calls** (mock `_transport`).

## Notes

- **Sizes:** LUMI-151 **S** (the gate probe), LUMI-152 **L** (the base client + structured output + wiring),
  LUMI-153 **L** (the tool-loop port — the bulk), LUMI-154 **M** (thinking + docs).
- **No contract change** — `{reply, emotion, intensity}` + memory record shapes are untouched; the v0.3 gate
  covers JSON-mode output. Off → byte-identical; the **Anthropic path is unchanged** (a regression test pins
  it).
- **Two risks designed in:** the **safety filters** (LUMI-151 is the gate; LUMI-152 degrades gracefully) and
  the **schema-vs-tools conflict** (LUMI-153's split). See [GEMINI_ENGINE.md](../../../docs/GEMINI_ENGINE.md).
- **Model id:** `gemini-3.1-pro` unverified (the repo is on `gemini-2.5-flash`); the seam is id-agnostic — set
  `LUMI_MODEL` to whatever the API exposes.
- **Build order:** **LUMI-151 first** (go/no-go), then **LUMI-152** (the foundation), then **LUMI-153** +
  **LUMI-154** (independent extensions).
