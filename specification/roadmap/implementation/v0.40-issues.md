# Version v0.40 — Issue Breakdown

**Phase:** v0.40 — Model routing: per-operation tiers (cost control, off by default)

Route each **operation** to the cheapest Claude tier that can carry it: the visible **`reply` stays on
Opus 4.8** (the voice — untouched), **thoughts + mood go to Sonnet 4.6**, and the **`session-*` /
`compaction` housekeeping to Haiku 4.5** — via a `_model_for(kind)` resolver and **one line** in
`_housekeeping_reply` (the seam is already there: `model` is a per-call argument, and the bounded
tool-loop follows the call's model, so routing an op routes its whole loop for free). Off → one model,
byte-identical. Target **~−26% of model spend with no change to her voice**. The reply-tier dial
**already shipped with v0.37** (`/model opus ⇄ sonnet ⇄ haiku`) — this phase adds **no new command**,
only the test pinning that the dial **composes** with routing. One **post-v0.39 rule**: the tier vars
name **Claude** ids, so routing applies **only while the active provider is `anthropic`** (a Claude id
never reaches a foreign API). Full measured analysis + design:
[LLM_OPERATIONS_COST.md](../../../docs/LLM_OPERATIONS_COST.md) +
[MODEL_ROUTING_IMPLEMENTATION.md](../../../docs/MODEL_ROUTING_IMPLEMENTATION.md).

### LUMI-155 — Layer 1: per-operation model routing (`_model_for(kind)` + tier config + provider guard)

**Description:**
Today every LLM call — the reply, thoughts, mood, summaries/facts/compaction, and every tool-loop step —
runs on the one `LUMI_MODEL` (`claude-opus-4-8`). The seam is already in place: `model` is a per-call
argument to `LLMClient.reply`/`reply_structured`, one Anthropic client serves all Claude tiers, and
`_housekeeping_reply` passes `model: self._model` in exactly one spot (`core/agent.py`) through which
`think` / `mood` / `session-start` / `session-close` / `compaction` all flow — **and their tool-loops
follow the call's model**. Add the resolver + config; no client change.

**What needs to be done:**
- A **`_model_for(kind)`** resolver on the core: `think` → `LUMI_MODEL_THINK`, `mood` → `LUMI_MODEL_MOOD`,
  `session-start` / `session-close` / `compaction` (+ the bare `housekeeping` default) →
  `LUMI_MODEL_HOUSEKEEPING`; each var **defaults to unset → `self._model`** (byte-identical guard).
- **One line** in `_housekeeping_reply`: `"model": self._model` → `"model": self._model_for(kind)` —
  the `reply` path stays on `self._model` (Opus), untouched.
- **Provider guard (post-v0.39):** the tier vars name **Claude** ids, and the active engine can be
  GPT-5.5 or Gemini (`/model`, v0.37/v0.39) — `_model_for` returns `self._model` unless
  `self._active_provider == "anthropic"` (routing is a no-op on a foreign engine; no Claude id ever
  reaches another provider's API).
- Config: `LUMI_MODEL_THINK` / `LUMI_MODEL_MOOD` / `LUMI_MODEL_HOUSEKEEPING` in `core/config.py`
  (strings, default `""` → unrouted); document beside `LUMI_MODEL`.

**Dependencies:** none — reuses v0.1 (the `LLMClient` seam), v0.15 (caching; the reply's Opus cache is
preserved), v0.37/v0.39 (`switch_model` + `_active_provider`).

**Expected result:**
With the tier vars set, `think`+`mood` run on Sonnet (their tool continuations too), the `session-*` /
`compaction` housekeeping on Haiku, and the visible reply stays on Opus + thinking; after
`/model gemini`, housekeeping calls use `self._model` (guard); unset vars → every call uses
`self._model`, byte-identical.

**Acceptance criteria:**
- [ ] A `MockLLMClient` records the `model` per call: a turn + a `think` + a session close use the configured tiers (`reply`→opus, `think`→sonnet, `session-start`/`session-close`/`compaction`→haiku).
- [ ] Contract: the routed `think`'s **tool continuations** are also Sonnet (the tool-loop follows the op); the emotion contract validates regardless of tier.
- [ ] **Provider guard:** after `/model gemini` (or `gpt-5.5`), housekeeping calls use `self._model` — no Claude tier id reaches a foreign client.
- [ ] **Unset → byte-identical** (every call on `self._model`); the cache-log tags the right model per `kind`.
- [ ] **No paid calls** (model mocked).

### LUMI-156 — Reply-tier dial: pin `/model <tier>` composition + MODELS_SETUP.md

**Description:**
The three-state reply dial **already shipped with v0.37**: `DEFAULT_MODEL_ALIASES` contains
`opus`/`sonnet`/`haiku` → Anthropic tier ids, so `/model sonnet` re-points the reply model mid-session
via `Core.switch_model`, status bar included — and an alias names its **provider**, so a tier swap from
a foreign engine explicitly returns to Anthropic. **No new command** (a `/mode` twin one letter from
`/model` would be a duplicate). What's missing is the **composition pin**: with tier vars set,
`/model sonnet` must move the **reply** (and the unset-tier fallback) while routed ops keep their
configured tiers — plus the operator docs.

**What needs to be done:**
- A **composition test**: tier vars set (`think`→sonnet, housekeeping→haiku) + `/model haiku` → the
  reply model re-points to haiku, `think` stays sonnet, housekeeping stays haiku (the mock records both);
  an unknown alias is still rejected with the non-fatal message.
- A **fallback test**: tier vars **unset** + `/model sonnet` → everything follows `self._model` (the
  v0.37 behaviour, unchanged by routing).
- **Docs:** a **three-tier dial** section in [MODELS_SETUP.md](../../../docs/MODELS_SETUP.md) —
  `/model opus` (quality, default) / `/model sonnet` (balanced) / `/model haiku` (cheapest — *speak on
  Haiku* to save most); persists nothing (next start uses `LUMI_MODEL`); how it composes with the
  Layer-1 tier vars.

**Dependencies:** **LUMI-155** (the routing the composition test exercises); reuses v0.37 (`switch_model`
+ the aliases — shipped, untouched).

**Expected result:**
`/model opus ⇄ sonnet ⇄ haiku` swaps the reply tier mid-session (no restart) and **composes** with
per-op routing — routed ops keep their tiers; the dial is documented as the explicit, reversible way to
*speak* cheaper (**no automatic reply downgrade** anywhere).

**Acceptance criteria:**
- [ ] The composition test: `/model <tier>` moves the reply model while routed ops keep their configured tiers (mock records both); unknown alias rejected.
- [ ] The fallback test: with tier vars unset, `/model <tier>` behaves exactly as shipped in v0.37.
- [ ] `MODELS_SETUP.md` documents the three-tier dial + its composition with the tier vars.
- [ ] **No paid calls** (model mocked).

### LUMI-157 — `read_file`/`read_around` char cap (the un-shipped result-size delta)

**Description:**
Tool-step cost is driven by **result size**, and the line caps already shipped: `LUMI_FILE_READ_LINES`
clamps one `read_file` call (200 lines), `LUMI_FILE_READ_MAX_TOTAL` caps a turn's total reads (2000
lines, shared with `read_around`), and `search_files` is already char-capped
(`LUMI_FILE_SEARCH_MAX_CHARS`, 4000). The gap: **no character cap** on `read_file`/`read_around`
results — 200 long lines can still be enormous and bloat every subsequent loop step. Add the char cap;
**no model change, no coherence risk**.

**What needs to be done:**
- A **`LUMI_FILE_READ_MAX_CHARS`** config (a sane default, e.g. the search cap's order of magnitude):
  truncate a `read_file` / `read_around` result at the cap with an explicit marker (the `search_files`
  pattern — "… (capped at N chars; read a narrower range)") so she knows to re-read narrower, never a
  silent cut.
- Apply it in `core/files.py` on both tools' output (after the existing line clamps; the per-turn line
  budget stays).
- Keep `list_files`/`find_in_file`/`search_files` unchanged (already bounded).

**Dependencies:** none — extends the shipped v0.19/v0.32 file executor; runs in parallel with LUMI-155.

**Expected result:**
A `read_file` of a huge-lined file returns at most `LUMI_FILE_READ_MAX_CHARS` characters with a clear
truncation marker; the tool bill per step is bounded by chars, not just lines; off/default keeps a
generous, non-breaking bound.

**Acceptance criteria:**
- [ ] `read_file` and `read_around` results are clamped to `LUMI_FILE_READ_MAX_CHARS` with an explicit truncation marker (never silent).
- [ ] The existing line caps + per-turn budget still hold (tests unchanged/extended, not weakened).
- [ ] Unset → the default cap applies; behaviour on ordinary files is unchanged (no marker).
- [ ] **No paid calls**.

### LUMI-158 — Layer 2: per-step routing inside the Anthropic tool-loop (optional, gated, off)

**Description:**
Layer 1 routes whole operations; the reply's own tool-loop still runs **every** step on Opus. Layer 2
lets the *intermediate* read/search/navigation steps run on a cheaper tier while the **final visible
step stays on the reply model** (the two-pass "dig on Sonnet, speak on Opus" pattern). **Anthropic-only**
— extend **`AnthropicClient._tool_loop`** alone; the v0.37/v0.39 OpenAI/Gemini loops keep their single
model. Mixing tiers in one message history carries a **coherence risk** → ships behind
`LUMI_TOOL_STEP_ROUTING`, **off by default**, A/B before ever defaulting on.

**What needs to be done:**
- Extend `AnthropicClient._tool_loop` (+ the `_text_tool_loop` twin): when `LUMI_TOOL_STEP_ROUTING` is
  on and the loop runs ≥1 tool round, intermediate rounds call the configured step tier
  (`LUMI_MODEL_TOOL_STEP`, default unset → off) and the **forced/terminal round returns to the call's
  model** (the visible voice).
- The per-round cache/stats log (`last_round_log`) tags each round's actual model, so the cache monitor
  attributes cost correctly.
- Config: `LUMI_TOOL_STEP_ROUTING` (off) + `LUMI_MODEL_TOOL_STEP`; both unset → byte-identical loop.
- A short **A/B note** in [MODEL_ROUTING_IMPLEMENTATION.md](../../../docs/MODEL_ROUTING_IMPLEMENTATION.md)
  (how to compare coherence before enabling).

**Dependencies:** **LUMI-155** (the tier config pattern + resolver it reuses); Anthropic loop only.

**Expected result:**
With the flag on, a tool-heavy reply digs its intermediate steps on the cheap tier and speaks its final
step on the reply model; off (default) → the loop is byte-identical; OpenAI/Gemini loops untouched.

**Acceptance criteria:**
- [ ] With the flag on, a scripted 3-round loop runs rounds 1–2 on `LUMI_MODEL_TOOL_STEP` and the terminal round on the call's model (mock records per-round models).
- [ ] The round log tags each round's actual model (cache monitor attribution).
- [ ] **Off (default) → byte-identical**; the OpenAI/Gemini loops are unchanged (regression pin).
- [ ] **No paid calls** (mock transport).

### LUMI-159 — Cost observability: verify the routing landed + re-measure the saving

**Description:**
`.lumi/cache-log.jsonl` already tags `kind` + `model` per call, and the cost report aggregates per-kind
spend. Close the phase the way it was opened — with **measured numbers**: verify each kind runs on its
configured tier in real usage and re-measure the actual saving (the ~−26% target and the ~29% tool-bill
estimate both predate the shipped line caps + these changes).

**What needs to be done:**
- Re-run the cost report over a stretch of real usage **with the tier vars on** (operator-run, like the
  v0.39 probe pattern); confirm per-kind model attribution matches the config.
- **Record the measured deltas** in [LLM_OPERATIONS_COST.md](../../../docs/LLM_OPERATIONS_COST.md) (a
  short "v0.40 measured" note: per-kind spend before/after, the read-cap effect, the realized %).
- If a kind shows the wrong model (a missed call site), fix the tagging/routing and note it.

**Dependencies:** **LUMI-155** + **LUMI-157** (the changes it measures; LUMI-158 optional — measure with
its flag off unless the A/B is running).

**Expected result:**
A short measured section in the cost doc: the routing verified per kind against real cache-log data and
the real saving quantified — the phase's target checked against reality, not left as an estimate.

**Acceptance criteria:**
- [ ] The cost report over real usage shows each `kind` on its configured tier.
- [ ] `LLM_OPERATIONS_COST.md` gains the "v0.40 measured" note (per-kind before/after + the realized saving).
- [ ] Any mis-attributed call site found is fixed (or explicitly noted).
- [ ] CI untouched (operator-run measurement; **no paid calls in tests**).
