# v0.9 — GitHub Issues

Issues for phase **v0.9 — Richer short memory (recent detail + days at a glance)** (version
**v0 — TUI**), derived from the per-phase Tasks in [ROADMAP.md](../../ROADMAP.md) (§v0.9) and
ARCHITECTURE §Memory. IDs continue from v0.8 (LUMI-033 → **LUMI-034…036**).

Today short memory keeps the **last 5 conversation summaries** (one detailed summary each) and
injects them. v0.9 makes each session's summary **two-tiered** and widens the recall window: at
session close, **one** call returns both a **detailed** summary and a one-line **gist**; the
prompt then injects the **last N=5 conversations in detail** plus **all conversations in the last
D=5 local days as gists** (the recent N not repeated, dated, capped). It's a memory-record shape
change (`ShortSummary` gains `gist`) with a **migration** for old records — so it updates the
`ShortSummary` contract test + ARCHITECTURE §Memory. **Long-term facts (`LongTermFact`) are
untouched.** Per-user isolation holds throughout. The "last D days" window uses the **injected
clock**. No emotion contract change.

## v0.9 — Richer short memory

### LUMI-034 — Two-tier `ShortSummary` — detailed + gist (one call) + migration

**Description:**
Each finished session's summary becomes **two-tiered**: the existing **detailed** `summary` plus a
new one-line **gist**, produced in **one** end-of-session model call. `ShortSummary` gains a
`gist` field; **existing records load fine** (no `gist` → empty). Long-term facts are untouched.
(`/core` + `/state`.)

**What needs to be done:**
- `ShortSummary{user_id, session_id, summary, gist, ts}` — add `gist: str` (default `""` for
  back-compat). Update the JSON store read/write: persist `gist`; an old record without it loads
  with `gist=""` (the **migration**).
- `summary_request` (one call, extended thinking off, as today) returns **both** — a detailed
  summary **and** a short gist — in labeled sections; a parser extracts both. If the model omits
  the gist, fall back to a **truncation of the detailed** summary so a gist always exists.
- `_write_summary` (or `end_session`) persists both onto the `ShortSummary`.
- The end-of-session **facts** path (`LongTermFact`) is unchanged.

**Dependencies:** None (builds on the v0.2 memory layers).

**Expected result:**
After a session, its `ShortSummary` carries a detailed `summary` + a one-line `gist`; old stored
summaries still load (empty gist).

**Acceptance criteria:**
- [ ] `ShortSummary` has a `gist` field; an old JSON record (no `gist`) loads with `gist=""`.
- [ ] One summarization call returns detailed + gist (mock model); the parser extracts both; a missing gist falls back to a truncated detailed.
- [ ] Contract test: the updated `ShortSummary` shape (incl. `user_id`, `gist`).
- [ ] `LongTermFact` (long-term memory) is untouched; per-user isolation holds.
- [ ] Unit tests against a mock model + fake store. No paid calls.

### LUMI-035 — Repository: summaries within the last D local days

**Description:**
A small, user-scoped Repository query that returns a user's summaries **on/after a given date** —
the source for the gist tier (the recent-N stays the recency query). Local JSON impl + the
contract. (`/state` + `/core` interface.)

**What needs to be done:**
- Add `summaries_since(user_id, since_date) -> list[ShortSummary]` to the `Repository` interface
  and the JSON store: that user's summaries with `ts` date ≥ `since_date`, **user-scoped**, in a
  deterministic order (oldest→newest or newest→oldest, documented).
- Keep `recent_summaries(user_id, limit)` for the detailed tier (recency).
- Per-user isolation: only this `user_id`'s records are returned.

**Dependencies:** LUMI-034 (the `gist` field exists on `ShortSummary`).

**Expected result:**
`summaries_since(user, today - D days)` yields that user's recent-days summaries; empty when none.

**Acceptance criteria:**
- [ ] `summaries_since(user_id, date)` returns that user's summaries on/after the date, isolated; empty list when none; deterministic order.
- [ ] Contract test: the query shape + **per-user isolation** (B's summaries never returned for A).
- [ ] Unit tests against a fake/JSON store. No paid calls.

### LUMI-036 — Two-tier short-memory injection (last N detailed + D-day gists)

**Description:**
The short-memory block in the system prompt becomes **two tiers**: the **last N conversations in
detail** + **all conversations in the last D local days as gists** — the recent N **not repeated**,
dated, and **capped**. `N`/`D`/cap are config; the D-day window uses the **injected clock**. (`/core`.)

**What needs to be done:**
- In `Core._system_prompt`, build:
  - **detailed** — `recent_summaries(user, N)` → `"[date] <summary>"`;
  - **gists** — `summaries_since(user, today − D days)` minus the recent-N `session_id`s →
    `"[date] <gist>"`, skipping empty gists (old records); **deduped** by `session_id`; **capped**
    by a max count (and/or a token budget).
- `today` comes from the **injected clock**; the window is the last **D local days**.
- `build_system_prompt`: the summaries block carries the two labeled tiers (recent detail + days
  at a glance), or extend the existing summaries section cleanly.
- Config: `RECENT_SUMMARIES` (`N`, default 5), `GIST_DAYS` (`D`, default 5), and a **max-count cap**
  for the gist tier.

**Dependencies:** LUMI-034 (the gist field + two-tier summary), LUMI-035 (the date query).

**Expected result:**
The prompt shows the last 5 conversations in detail and the last 5 days' conversations as gists —
no duplication, dated, capped; long-term facts unchanged.

**Acceptance criteria:**
- [ ] The prompt injects the **last N detailed** summaries + **last D-day gists**, **deduped** (recent N excluded from the gist tier), dated, **capped**.
- [ ] Old records (empty gist) are omitted from the gist tier; never crash.
- [ ] `N`/`D`/cap config honored; the D-day window uses the injected clock (fixed in tests).
- [ ] Long-term facts unchanged; per-user isolation holds.
- [ ] Unit tests: two-tier assembly (dedup, cap) against a fake store + fixed clock; the local-day window selection. No paid calls.
