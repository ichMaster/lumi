# v0.3 — GitHub Issues

Issues for phase **v0.3 — Emotion field** (version **v0 — TUI**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v0.3) and the emotion contract in [EMOTION.md](../../features/EMOTION.md) (§3 contract, §4 enum, §5 renderer interface, §8 validation/fallback) and [ARCHITECTURE.md](../../ARCHITECTURE.md) (§Emotion channel, §Data model, §Observability). This file is scoped to a single phase; IDs continue from v0.2 (LUMI-012 → **LUMI-013…018**).

v0.3 **locks the emotion channel** — the contract every later render tier (emoji v0.4, local face v0.6, web portrait v2.1, animation v3) reuses without change. The model stops returning a bare string and starts emitting **structured output** `{reply, emotion, intensity}`; the core **validates** it against the fixed 9-value enum and the 0–1 range, **repairs/falls back** on invalid output, **logs** the field per turn, and exposes it through the `IEmotionRenderer` interface with a `LogRenderer` (the "logged" tier). The TUI shows the current state in a small status line. *Лілі returns her state, even if we barely show it yet.*

This phase turns the v0.1 contract `Core.reply(user_text, session) -> str` into `-> EmotionState` — the one breaking change of the phase, made once and then frozen. It extends the existing seams without reshaping them: `build_system_prompt(canon, …)` gains the **emotion-output instruction** (it was authored as an extension seam); the `LLMClient` seam grows a **structured** path (Anthropic tool/structured output for Claude Haiku; the mock returns canned — and deliberately malformed — structured replies); `Message` gains optional `emotion`/`intensity` so assistant turns carry the field. **The contract, enum, and `IEmotionRenderer` interface locked here never change.**

## v0.3 — Emotion field

### LUMI-013 — Emotion contract — `EmotionState` + the fixed 9-value enum

**Description:**
The locked data contract for the whole emotion channel. A fixed 9-value `Emotion` enum and an immutable `EmotionState{reply, emotion, intensity}` that every render tier (v0.3→v3) reuses unchanged. This is the **gate** — pinned by a contract test now, at the schema level, so any later drift in the contract must change the test. (`/core` shapes; the test in `/tests/contract`.)

**What needs to be done:**
- Add an `Emotion` enum with **exactly** the 9 values from EMOTION.md §4: `joy, calm, playful, tender, thoughtful, serious, surprise, doubt, sad`. `calm` is the neutral / fallback.
- Add a frozen `EmotionState` dataclass: `reply: str`, `emotion: Emotion`, `intensity: float` (0.0–1.0) — the model's output portion of the contract (EMOTION.md §3). Reserve `ttl_ms`/`speaking` only in comments (renderer-side; not in the model's output) so the shape is stable when animation/voice arrive.
- Place these in `/core` (e.g. `core/emotion.py`); the core depends on the shape, not on any renderer.

**Dependencies:** None (the v0.3 gate).

**Expected result:**
A single, importable `EmotionState` + `Emotion` enum the rest of the phase builds on; a contract test fails if the enum membership or the field schema changes.

**Acceptance criteria:**
- [ ] `Emotion` has exactly the 9 EMOTION.md §4 names — no more, no fewer.
- [ ] `EmotionState` carries `{reply: str, emotion: Emotion, intensity: float}` and is immutable.
- [ ] **Contract test** pins the enum membership (the 9 names) and the `EmotionState` field set; changing either fails the test.
- [ ] No renderer/interface concerns leak into the contract module.

### LUMI-014 — Structured emotion output + the emotion-output instruction

**Description:**
Make the model **emit** `{reply, emotion, intensity}` as structured output instead of a bare string, constrained so invalid values are rare by construction (EMOTION.md §8: schema enforcement first). The `LLMClient` seam grows a structured path; `AnthropicClient` uses **Anthropic tool / structured output** (Claude Haiku); `build_system_prompt` gains the **emotion-output instruction** (its documented v0.3 extension). The `MockLLMClient` returns canned structured replies — **including deliberately malformed ones** — so the validation gate (LUMI-015) and CI never need a paid call. (`/core`.)

**What needs to be done:**
- Extend the `LLMClient` seam with a structured-reply capability that returns the raw `{reply, emotion, intensity}` object (alongside the existing text path, which the housekeeping calls keep using).
- `AnthropicClient`: request the emotion object via **Anthropic tool/structured output**, constraining `emotion` to the enum and `intensity` to a 0–1 number (EMOTION.md §8). Keep the `LLMError` + bounded-retry behavior. (Per-provider structured output for OpenAI/DeepSeek/MiniMax is deferred to v0.9; the v0.3 gate is the safety net regardless.)
- Add the **emotion-output instruction** to `build_system_prompt` (the v0.3 extension noted in its docstring) — instruct Лілі to return her state in the fixed enum, alongside `reply`. Compose it around the canon without breaking the v0.1/v0.2 assembly.
- `MockLLMClient`: return canned structured replies and a way to script **malformed** ones (unknown emotion, out-of-range/missing intensity, missing reply) for the gate's tests — never a paid API.
- Keep the `<think>` reasoning split working: reasoning stays out of the `reply` field (EMOTION.md keeps `reply` as Лілі's text only).

**Dependencies:** LUMI-013.

**Expected result:**
A turn yields a raw structured emotion object from the model (real or mock); malformed mock outputs are scriptable for the gate.

**Acceptance criteria:**
- [ ] `AnthropicClient` requests `{reply, emotion, intensity}` via Anthropic tool/structured output, enum- and range-constrained.
- [ ] `build_system_prompt` carries the emotion-output instruction; canon/memory/style assembly is unchanged otherwise.
- [ ] `MockLLMClient` returns canned structured replies **and** scriptable malformed ones; tests call no paid API.
- [ ] Bounded retry + `LLMError` behavior preserved; `core` never imports the SDK directly.

### LUMI-015 — Validation & fallback gate (+ repair logging)

**Description:**
The single place emotion can go wrong, so it is the most-tested. Parse the raw model output into a **valid** `EmotionState`, repairing per EMOTION.md §8: unknown/missing `emotion` → `calm`; `intensity` clamped to `[0,1]`, missing → `0.5`; a missing `reply` is an **error surfaced to the interface** (never a silent empty turn). **Every repair is logged** keyed by `session_id`/turn so model drift is visible (ARCHITECTURE §Observability). (`/core`.)

**What needs to be done:**
- A pure `validate(raw, …) -> EmotionState` (e.g. in `core/emotion.py`): unknown/missing emotion → `calm`; clamp `intensity` to `[0,1]`, missing → `0.5`; missing/empty `reply` → a raised error the interface renders (not a swallowed empty reply).
- Log every repair (what was wrong, what it became) keyed by `session_id`/turn — structured, so drift is countable.
- Never trust raw output: the gate runs even though the structured path (LUMI-014) makes invalid values rare.

**Dependencies:** LUMI-013.

**Expected result:**
Any raw model output (valid, malformed, or partial) becomes either a valid `EmotionState` or a clear surfaced error — with repairs logged.

**Acceptance criteria:**
- [ ] Unknown/missing `emotion` → `calm`; `intensity` clamped to `[0,1]`; missing `intensity` → `0.5`.
- [ ] Missing/empty `reply` raises a clear error (surfaced, not silent).
- [ ] Each repair is logged keyed by `session_id`/turn.
- [ ] **Unit tests** drive the rules with deliberately malformed mock replies (unknown emotion, out-of-range/missing intensity, missing reply).

**Dependencies:** LUMI-014, LUMI-015.

**Expected result:**

**Acceptance criteria:**

### LUMI-017 — `IEmotionRenderer` interface + `LogRenderer`

**Description:**
The interface the core codes against and the v0.3 "logged" tier. `IEmotionRenderer` (EMOTION.md §5) is the one interface each later tier implements; `LogRenderer` writes the validated field to the log (the per-turn channel log), with `tick`/`set_speaking` as no-ops. **Only the renderer changes between versions; the interface is locked here.** (`/core`.)

**What needs to be done:**
- Define `IEmotionRenderer` (Protocol, EMOTION.md §5): `render(state: EmotionState) -> None`, `set_speaking(speaking: bool) -> None`, `tick(dt_ms: int) -> None`.
- Implement `LogRenderer`: `render` writes the validated `{emotion, intensity}` (and a reply marker) to a structured log keyed by `session_id`/turn; `tick`/`set_speaking` are no-ops.
- Keep it interface-independent (no TUI/web concerns); the TUI/web supply their own renderers later (emoji v0.4, image v2.1, animation v3.1).

**Dependencies:** LUMI-013.

**Expected result:**
A locked renderer interface plus a working `LogRenderer` — the channel is observable in the logs.

**Acceptance criteria:**
- [ ] `IEmotionRenderer` matches EMOTION.md §5 (`render`/`set_speaking`/`tick`).
- [ ] `LogRenderer.render` logs the validated field keyed by `session_id`/turn; `tick`/`set_speaking` are no-ops.
- [ ] No interface/renderer concern leaks into the contract; the core depends only on `IEmotionRenderer`.
- [ ] Unit test: `LogRenderer` emits the expected structured log line for a given `EmotionState`.
