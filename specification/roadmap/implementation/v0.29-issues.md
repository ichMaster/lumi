# v0.29 — GitHub Issues

Issues for phase **v0.29 — Local file tool III: metadata + create-folder + copy (non-destructive)**
(version **v0 — TUI**), derived from [ROADMAP.md](../../ROADMAP.md) (§v0.29) and
[FILE_TOOL.md](../../features/FILE_TOOL.md). IDs continue from v0.28 (LUMI-112 → **LUMI-113…115**).

A small extension of the shipped v0.19 (read) + v0.20 (write) file tool: Лілі (or you, through her) can
**see a file's created/modified dates**, **make a folder**, and **copy a file** in her sandbox. It is
**additive to `core/files.py`** — it reuses the v0.19 `safe_path` sandbox guard + the bounded tool-loop,
adds **no new seam** and **no contract change**, and keeps the **create-only / non-destructive** guarantee
(no overwrite, no delete, no move).

**Three things, all on the existing executor:**
- **Dates (read).** `list_files` reports each entry's **created + modified** date alongside its size, and a
  new **`stat_file(path)`** returns size + dates for a single file without listing the directory. Both read
  `os.stat` once — modified is `st_mtime`; **created** is `st_birthtime` where the OS provides it (macOS /
  BSD), falling back to `st_ctime` (the metadata-change time) elsewhere, **labelled honestly**. Read-only,
  so the non-destructive invariant is untouched.
- **`create_folder(path)` (write).** Creates a **new** directory under the sandbox — **create-only**
  (refuse if it exists, like `create_file`); parents are created under the root via the same `safe_path`
  guard. No overwrite, no delete.
- **`copy_file(src, dest)` (write).** Copies an existing sandbox file to a **new** destination — both paths
  go through `safe_path`; the source must exist and be a file; the **destination must not exist** (a clash
  is refused — no overwrite). `shutil.copy2` preserves metadata; bounded by a new **`LUMI_FILE_COPY_MAX`**
  source-size cap, separate from the model-content `LUMI_FILE_WRITE_MAX` (a copy moves *existing* bytes,
  not model-supplied content) — an oversize source is refused.

**Hard rules** (FILE_TOOL.md §safety): **sandboxed + per-user** (the shared v0.19 `safe_path` guard under
`.lumi/files/<user_id>/`; a `..` / absolute / symlink escape on **either** path is rejected before any I/O);
**non-destructive** — there is **no overwrite / delete / move** path (`create_folder` and `copy_file`'s dest
are both create-only); **never raises** (every path returns an error string on failure); **off by default**
(`LUMI_FILE_TOOL` — the same flag as v0.19/v0.20, no new toggle). **No emotion-contract change** —
`set_state` stays terminal and the validated `{reply, emotion, intensity}` is unchanged.

Depends on **v0.19** (the bounded tool-loop + `safe_path` + `list_files`/`read_file`) and **v0.20** (the
non-destructive `create_file` / `append_file` write tools) — both shipped. The model is **mocked** in every
test — no paid calls. **A standalone enhancement** of the v0.19/v0.20 file tool (its position is immaterial
— it could equally ship as a `0.20.x`).

## v0.29 — Local file tool III: metadata + create-folder + copy (non-destructive)

### LUMI-113 — `core/files.py`: dates + `stat_file` + `create_folder` + `copy_file`

**Description:**
Extend `core/files.py` with the v0.29 operations, all on the existing `FileTools` executor and the shared
`safe_path` guard — **no new seam, no contract change, still create-only**. Add **dates** to the
`list_files` output and a new read tool **`stat_file(path)`**; add two non-destructive write tools
**`create_folder(path)`** and **`copy_file(src, dest)`**. The new schemas join the existing `READ_TOOLS`
(`list_files` / `find_in_file` / `read_file`) and `WRITE_TOOLS` (`create_file` / `append_file`) lists, and
the new branches are handled in `FileTools.execute`. Fully unit-testable on `FileTools` alone over a temp
sandbox — no model, no network. (`/core`.)

**What needs to be done:**
- **Dates on `list_files`:** alongside each entry's name + size, report its **created** and **modified**
  date from a single `os.stat` — modified = `st_mtime`; **created** = `st_birthtime` where present (macOS /
  BSD), else `st_ctime` as a labelled fallback. A shared helper (e.g. `_stat_dates(path)` → `(created,
  modified)`) so `list_files` and `stat_file` format dates identically.
- **`stat_file(path)`** (read; add to `READ_TOOLS`): returns one file's **size + created + modified** date
  without listing the directory; a missing path / a directory → an error string.
- **`create_folder(path)`** (write; add to `WRITE_TOOLS`): `safe_path`-resolve, **refuse if the path
  already exists** (create-only), then `mkdir(parents=True)` under the root. No overwrite, no delete.
- **`copy_file(src, dest)`** (write; add to `WRITE_TOOLS`): `safe_path`-resolve **both** paths; the source
  must exist and be a file; the **dest must not exist** (refuse a clash — no overwrite); refuse a source
  larger than `LUMI_FILE_COPY_MAX`; copy with `shutil.copy2` (preserve metadata), creating dest parents
  under the root.
- **`FileTools.execute`** dispatches the three new names; **always returns a string** — any failure
  (missing/oversize source, existing dest/folder, traversal/escape on either path, I/O error) degrades to
  an **error string**, never raises.
- **Config** (`core/config.py`): add `file_copy_max` (`LUMI_FILE_COPY_MAX`, e.g. 5 MB) — separate from
  `file_write_max` (`LUMI_FILE_WRITE_MAX`); thread it into `FileTools` (a `copy_max` ctor arg).

**Dependencies:** None (pure handler; reuses `core/files.py` `safe_path` + the v0.20 create-only rule; no
model involved).

**Expected result:**
On a `FileTools` bound to a temp sandbox: `execute("list_files", {...})` and `execute("stat_file",
{"path": "a.txt"})` report created + modified dates (created falls back to `st_ctime` where `st_birthtime`
is absent); `execute("create_folder", {"path": "notes"})` makes the dir and **refuses** a second call;
`execute("copy_file", {"src": "a.txt", "dest": "b.txt"})` copies, and **refuses** an existing dest, an
oversize source, a missing source, and any `..`/absolute/symlink escape on either path. No network.

**Acceptance criteria:**
- [ ] `list_files` reports **created + modified** dates (+ size) per entry; `stat_file(path)` returns one file's size + dates; both use the same `st_mtime` / `st_birthtime`→`st_ctime` fallback (labelled honestly).
- [ ] `create_folder(path)` is **create-only** (refuses an existing path); `copy_file(src, dest)` requires an existing file source, a **non-existing dest** (no overwrite), passes through `safe_path` on **both** paths, uses `shutil.copy2`, and refuses a source over `LUMI_FILE_COPY_MAX`.
- [ ] **No overwrite / delete / move** path exists anywhere in `FileTools`.
- [ ] `FileTools.execute` **always returns a string** — bad input / I/O / traversal degrade to an error string, **never raises**.
- [ ] `file_copy_max` (`LUMI_FILE_COPY_MAX`) added to config and threaded into `FileTools`; unit tests on `FileTools` over a temp sandbox cover dates/stat, create-only folder, copy create-only + size cap + traversal. **No model, no network.**

### LUMI-115 — Contract + per-user isolation + behaviour tests

**Description:**
Pin the whole v0.29 surface against the mock model (no paid calls): dates / `stat_file` (with the
`st_birthtime`→`st_ctime` fallback), the create-only folder + copy-dest rule, the source-size cap, traversal
refusal on **either** path, **per-user isolation**, the tools **absent when off**, and the emotion contract
intact. Extends the shipped `tests/unit/test_files.py`, `tests/unit/test_file_bounds.py`, and
`tests/contract/test_file_tool_contract.py`. (`/tests`.)

**What needs to be done:**
- **Dates / stat test:** `list_files` + `stat_file` report created + modified dates; assert the **created**
  value falls back to `st_ctime` where `st_birthtime` is unavailable (monkeypatch / branch on the stat
  result so the test is platform-independent).
- **Create-only test:** `create_folder` makes a new dir and **refuses** an existing one; `copy_file`
  refuses an **existing dest** (no overwrite), a **missing/non-file source**, and a source over
  `LUMI_FILE_COPY_MAX`; the copied file's bytes match the source.
- **No-destructive-path test:** there is no overwrite / delete / move tool name; the existing
  `create_file` / `append_file` rules are unchanged.
- **Traversal test:** a `..` / absolute / symlink escape on **either** `src` or `dest` (and on
  `stat_file` / `create_folder` `path`) is refused before any I/O → an error string.
- **Per-user isolation test:** user A creates / copies a file; user B (a separate core / sandbox) cannot
  `stat_file` / `copy_file` it — every path resolves under `files_dir/<user_id>/`.
- **Off test:** `LUMI_FILE_TOOL` off → none of the new tools are offered (and the turn is unchanged).
- **Emotion-channel contract test** still validates `{reply, emotion, intensity}` verbatim with the new
  tools active.

**Dependencies:** LUMI-113, LUMI-114.

**Expected result:**
`pytest` proves dates/stat (with the documented created-date fallback), create-only folder + copy-dest,
the source-size cap, traversal refusal on either path, per-user isolation, off-by-default absence, and the
intact emotion contract — all on the mock model, no paid calls.

**Acceptance criteria:**
- [ ] `list_files` / `stat_file` report created + modified dates; the created date falls back to `st_ctime` where `st_birthtime` is absent (platform-independent test).
- [ ] `create_folder` is create-only (refuses an existing dir); `copy_file` refuses an existing dest, a missing/non-file source, and an oversize source; the copy's bytes match the source; **no overwrite/delete/move** path.
- [ ] A `..`/absolute/symlink escape on **either** path (`src`, `dest`, `stat_file`/`create_folder` `path`) is refused → an error string, never raises.
- [ ] **Per-user isolation:** A's file is not `stat`/`copy`-able in B's turn (paths resolve under `files_dir/<user_id>/`).
- [ ] Off (`LUMI_FILE_TOOL`) → the new tools are absent; the `{reply, emotion, intensity}` contract passes with them active; the suite uses the mock model — **no paid calls.**
