# v0.2 — GitHub Issues

Issues for phase **v0.2 — Three-layer memory** (version **v0 — TUI**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v0.2) and the memory/isolation contracts in [ARCHITECTURE.md](../../ARCHITECTURE.md) (§Memory, §Identity/users/memory scopes, §Data model, §Sessions and history). This file is scoped to a single phase; IDs continue from v0.1 (LUMI-006 → **LUMI-007…012**).

v0.2 turns the v0.1 stateless turn into a **remembering** one, and — crucially — makes the data model **user-scoped** so the v1 multi-user server is additive, not a rewrite. Three per-user memory layers land: **session history** trimmed to a rolling window in context; **short memory** (`ShortSummary`, summarized at session end, injected at startup); **long-term memory** (`LongTermFact`, accumulated, injected at startup). Every record carries a `user_id` (single default `owner`), and the per-user **isolation invariant** is pinned by a contract test at the data level. Memory is viewable and clearable from the TUI.

This phase extends the v0.1 seams without breaking them: `Session`/`Message` gain `user_id`; `build_system_prompt(canon)` gains `summaries`/`facts` (it was authored to); `Core.reply(user_text, session)` keeps its signature but trims history and writes memory; the `Repository` grows summary/fact methods behind the same interface.

## v0.2 — Three-layer memory

### LUMI-007 — User scoping — `User` + `user_id` on every record

**Description:**
Make the core and the `Repository` **user-scoped**, running with a single default `owner` — the one change that makes the v1 multi-user server additive instead of a rewrite. Every per-user record carries a `user_id`, and the **isolation invariant** (a record written under user A is never readable in user B's context) is pinned by a contract test now, at the data level. (`/core` shapes + interface, `/state` impl.)

**What needs to be done:**
- Add a minimal `User` shape and a single default `owner` id (ARCHITECTURE §Data model; full `User{id, role, pass_hash, …}` arrives in v1.3 — v0.2 needs only `id` and the default `owner`).
- Add `user_id` to `Session` and `Message` (ARCHITECTURE §Data model): `Session{id, user_id, started_at, ended_at?}`, `Message{session_id, user_id, role, text, ts}`. Update `make_message`/`JsonRepository` accordingly.
- Key the `Repository` by `user_id`: `create_session(user_id)`, `load_messages(session_id)` scoped to its owner, and per-user lookups; reads/writes always resolve through a `user_id`.
- `Core` carries the active `user_id` (default `owner`); `reply(...)` keeps its signature but operates within the user's scope.
- The **isolation invariant** is enforced at the data level (a query for user B never returns user A's records).

**Dependencies:** None (the v0.2 gate).

**Expected result:**
Every session/message is stored under a `user_id` (default `owner`); a read in one user's scope never returns another user's records.

**Acceptance criteria:**
- [ ] `User` + default `owner`; `Session`/`Message` carry `user_id` (shapes match ARCHITECTURE §Data model)
- [ ] `Repository` is keyed by `user_id` (create/append/load resolve through it)
- [ ] `Core` runs scoped to the active `user_id` (default `owner`); `reply(...)` signature unchanged
- [ ] **Contract test:** the per-user isolation invariant — a record written under user A is never retrievable in user B's context
- [ ] Existing v0.1 tests updated for the new shapes; suite stays green

### LUMI-008 — Rolling-window session history

**Description:**
Trim the in-context session history to a **rolling window** (last N turns / a token budget) before each model call, so long conversations stay within budget while keeping recent context. Replaces v0.1's "send the full session every turn". (`/core`.)

**What needs to be done:**
- Before each `LLMClient` call, trim the loaded history to the last N turns (use `config.memory_window`, the placeholder wired in v0.1) — older turns are dropped from context (folded into the short summary by LUMI-009, not sent verbatim).
- Keep persistence complete (the full history is still stored); only the **in-context** window is trimmed.
- Make the window size config-driven (no hardcoding) and the trimming a pure, unit-testable function.

**Dependencies:** LUMI-007

**Expected result:**
A long session sends only the last N turns to the model; persistence keeps everything.

**Acceptance criteria:**
- [ ] History is trimmed to `config.memory_window` turns before each model call
- [ ] Full history still persisted; only the context window is trimmed
- [ ] Window size is config-driven (no hardcoded N)
- [ ] Unit test: windowing/trimming keeps the last N turns and is exact at the boundary

### LUMI-009 — Short memory — `ShortSummary` + end-of-session summarization

**Description:**
At session end, the model compresses the conversation into a `ShortSummary`; the last few for that user are kept and injected at startup, so Лілі remembers the **thread** of past conversations. (`/core` logic, `/state` storage.)

**What needs to be done:**
- Add the `ShortSummary{user_id, session_id, summary, ts}` record (ARCHITECTURE §Data model) and `Repository` methods to add one and list a user's recent summaries.
- End-of-session summarization: when a session ends, build a summarization prompt from its history, call the `LLMClient`, and persist a `ShortSummary` for the user (mock model in tests — no paid call).
- Surface a session-end hook the TUI can call on exit (the actual injection at startup is wired in LUMI-011).

**Dependencies:** LUMI-007

**Expected result:**
Ending a session writes a `ShortSummary` for the user; recent summaries are retrievable.

**Acceptance criteria:**
- [ ] `ShortSummary{user_id, session_id, summary, ts}` record + `Repository` add/recent methods
- [ ] End-of-session summarization calls the `LLMClient` and persists a `ShortSummary` (per-user)
- [ ] **Contract test:** the `ShortSummary` shape (incl. `user_id`)
- [ ] Unit test: summarization produces and stores a summary against a **mock model** (no paid call)

### LUMI-010 — Long-term memory — `LongTermFact` accumulation

**Description:**
Durable facts about *that user* accumulate across sessions and persist, injected at startup — what makes Лілі remember things about you, not just the last thread. (`/core` logic, `/state` storage.)

**What needs to be done:**
- Add the `LongTermFact{user_id, fact, meta, confidence, ts}` record (ARCHITECTURE §Data model) and `Repository` methods to add a fact and list a user's facts.
- Fact accumulation: extract durable facts (model-driven at session end, alongside summarization, or a simple heuristic to start) and persist them per-user; accumulate across sessions.
- Keep extraction conservative and the storage append-only for now (dedup/decay can come later).

**Dependencies:** LUMI-007

**Expected result:**
Durable facts about the user accumulate and persist across sessions.

**Acceptance criteria:**
- [ ] `LongTermFact{user_id, fact, meta, confidence, ts}` record + `Repository` add/list methods
- [ ] Facts accumulate per-user across sessions and persist
- [ ] **Contract test:** the `LongTermFact` shape (incl. `user_id`)
- [ ] Unit test: fact accumulation/injection against a **mock model** + fake store (no paid call)

### LUMI-011 — Startup rehydration — assemble the user's context

**Description:**
Assemble the user's full context at startup and inject it into the system prompt — `canon` + shared experience (trivially the owner's until v2) + the user's recent `ShortSummary`s + the user's `LongTermFact`s. Extends the v0.1 `build_system_prompt` seam, which was authored to gain these parameters. (`/core`.)

**What needs to be done:**
- Extend `build_system_prompt(canon, summaries=…, facts=…)` to compose the memory blocks **around** the canon (canon always at the base; EMOTION/ARCHITECTURE assembly order in §Data model).
- At startup, the core rehydrates **for the active user**: load recent summaries + long-term facts and build the system prompt from them; the turn then runs with that context.
- Keep the assembly a pure, unit-testable function; never let another user's records enter (reuse the LUMI-007 isolation invariant).

**Dependencies:** LUMI-009, LUMI-010

**Expected result:**
On startup Лілі's prompt carries the user's canon + recent summaries + facts; a restart recalls prior context.

**Acceptance criteria:**
- [ ] `build_system_prompt` composes canon + summaries + facts (canon at the base; isolation preserved)
- [ ] Startup rehydration loads the user's recent summaries + facts into the prompt
- [ ] Unit test: prompt assembly places summaries + facts around the canon
- [ ] **Integration test:** restart rehydrates the user's context — Лілі recalls a prior session's summary + a stored fact (mock model + fake store, no paid call)

### LUMI-012 — TUI memory commands — view & clear

**Description:**
Expose the core memory commands (`memory.view`, `memory.clear`, ARCHITECTURE §Contracts) and surface them in the TUI: view the user's summaries + facts, and clear (wipe the user's short + long-term memory). (`/core` commands, `/tui` surface.)

**What needs to be done:**
- Core memory commands: `view_memory(user_id)` (return the user's recent summaries + facts) and `clear_memory(user_id)` (wipe the user's short + long-term; `Repository` clear method).
- TUI commands/keys to view memory (render summaries + facts) and clear it, with a confirmation; clearing affects only the active user's relationship memory (not the canon, not other users).
- Keep the TUI thin — it calls the core commands, no memory logic of its own.

**Dependencies:** LUMI-011

**Expected result:**
You can view and clear Лілі's memory of you from the terminal; `clear` wipes short + long-term for the active user only.

**Acceptance criteria:**
- [ ] Core `view_memory`/`clear_memory` commands (per-user); `Repository` clear method
- [ ] TUI surfaces view + clear (clear is confirmed); only the active user's memory is affected
- [ ] TUI stays thin (calls core commands; no memory logic)
- [ ] Unit test: `clear` wipes short + long-term for the user and leaves canon/other scopes intact; TUI view/clear drives the core commands (mock model)

## v0.2 scope notes

**Total effort:** ~1.5–2 weeks for a single developer (LUMI-007 gates and is the riskiest — it reshapes the data model and pins isolation; LUMI-008/009/010 then run largely in parallel; LUMI-011 integrates; LUMI-012 closes the phase).

**Critical path:** LUMI-007 → (LUMI-008 + LUMI-009 + LUMI-010) → LUMI-011 → LUMI-012.

**Phase DoD (ROADMAP §v0.2):** after a restart Лілі recalls what was discussed in previous sessions and durable facts about you.

**Contracts pinned this phase:** `user_id` on every per-user record; the `ShortSummary` and `LongTermFact` record shapes; the **per-user isolation invariant** at the data level (auth-boundary enforcement comes in v1.3). Changing any of these must change its contract test.

**Model note:** summarization and fact extraction call the model behind the v0.1 `LLMClient` seam; **no paid APIs in CI** — `MockLLMClient` returns canned summaries/facts for unit + integration tests.
