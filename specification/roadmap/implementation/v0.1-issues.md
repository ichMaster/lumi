The core exposes one `reply(...)` contract; no interface logic leaks into it. LUMI IDs start here (LUMI-001…).

## v0.1 — Skeleton and canon

### LUMI-001 — Project skeleton & tooling

**Description:**
Scaffold the repository and the interface-independent `core` package plus the test harness — the foundation every later issue builds on. (`/core`, `/tui`, `/tests`, root config.)

**What needs to be done:**
- Create the package layout per ARCHITECTURE §Stack and repository layout: `core/`, `tui/`, `tests/{unit,integration}` (and a `core/canon/` folder for the canon file).
- `pyproject.toml`: **ruff** (line-length 100; `select = ["E","F","I","W","UP","B"]`, ignore `E501`) and **pytest** (`testpaths=["tests"]`, `pythonpath=["."]`).
- `.env` loading for **`ANTHROPIC_API_KEY`** (Claude Haiku) via `python-dotenv` or equivalent — keys read from the environment, never committed.
- A minimal **config** module exposing the model id (default a current Claude Haiku id), the canon path, and a memory-window placeholder (filled in v0.2).
- Add the project deps (the `anthropic` SDK, `textual`, `python-dotenv`) to `pyproject.toml`.

**Dependencies:** None (first issue; the gate).

**Expected result:**
A fresh checkout installs, `ruff check` and `pytest` run green on a stub suite, and config + `.env` load.

**Acceptance criteria:**
- [ ] Package layout (`core/`, `tui/`, `tests/`, `core/canon/`) created
- [ ] `pyproject.toml` configures ruff (line-length 100) + pytest (`testpaths`, `pythonpath`)
- [ ] `.env` loading wired; `ANTHROPIC_API_KEY` read from the environment (not committed)
- [ ] `config` exposes the model id (default Claude Haiku), canon path, memory-window placeholder
- [ ] `ruff check` and `pytest` both run green on a stub suite

### LUMI-002 — `LLMClient` seam + Anthropic Claude Haiku

**Description:**
The thin **`LLMClient`** seam the core depends on (never a concrete SDK), its **Anthropic Claude Haiku** backend, and a **mock** for tests. Claude Haiku is the only model to start; v0.9 adds more behind this same seam. (`/core`.)

**What needs to be done:**
- Define `LLMClient` (a `Protocol`/ABC): `reply(system: str, messages: list, model: str) -> str` (v0.1 returns text; v0.3 returns an `EmotionState`).
- Implement `AnthropicClient` via the official **`anthropic`** SDK — model id from config (default Claude Haiku), `ANTHROPIC_API_KEY` from the environment.
- Implement `MockLLMClient` returning canned replies (used by unit/integration tests — **no paid API call**, ARCHITECTURE §Testing and CI).
- Minimal error handling: bounded retry then a clear surfaced error on API failure (ARCHITECTURE §Error handling) — never hang.
- The Anthropic SDK is imported **only inside `AnthropicClient`**; the rest of `core` depends on the `LLMClient` seam.

**Dependencies:** LUMI-001

**Expected result:**
`core` obtains a reply from Claude Haiku through `LLMClient`; tests run against `MockLLMClient`.

**Acceptance criteria:**
- [ ] `LLMClient` interface defined; `core` depends on it, not the SDK
- [ ] `AnthropicClient` calls Claude Haiku via the official SDK; model id from config; key from env
- [ ] `MockLLMClient` returns canned (and deliberately malformed, for later v0.3) replies
- [ ] API failure → bounded retry then a clear error (no hang)
- [ ] Unit test: config-driven model id selection; the mock returns a canned reply with no network call

### LUMI-003 — Canon loading & system-prompt assembly

**Description:**
Author/generate Лілі's **canon** and assemble it into the system prompt — the base of the prompt every later version extends *around* (memory v0.2, the emotion-output instruction v0.3, the mood block v0.5). (`/core`; canon at `core/canon/lili.md`.)

**What needs to be done:**
- A first-draft canon `core/canon/lili.md` authored or generated per [docs/CANON_SPEC.md](../../../docs/CANON_SPEC.md) (biography, values, voice, the 9-emotion palette, motifs). Refined later; a solid first draft is enough for v0.1.
- `load_canon(path)` reads the canon file; the path comes from config.

- The core never hardcodes character content — it all lives in `lili.md`.

**Dependencies:** LUMI-001

**Expected result:**
Лілі's character rides in the system prompt; she sounds like one specific person in a session.

**Acceptance criteria:**
- [ ] `core/canon/lili.md` authored per CANON_SPEC (biography, values, voice, emotional palette, motifs)
- [ ] `load_canon(path)` loads the canon from the config-referenced path
- [ ] `build_system_prompt(canon)` places the canon into the system prompt (the extension seam)
- [ ] Unit test: system-prompt assembly puts the canon content into the `system` field

### LUMI-004 — `Repository` interface + local store

**Description:**
The `Repository` seam and a local store that **persists session messages**; the core depends on the interface, never a concrete store, so the v1 server DB is a swap. (`/core` interface, `/state` impl.)

**What needs to be done:**
- Define `Repository` (a `Protocol`/ABC): create a session, append messages, load a session's messages.
- A local **JSON or SQLite** implementation behind it (per ARCHITECTURE §Storage).
- Data shapes (ARCHITECTURE §Data model): `Session{ id, started_at, ended_at? }`, `Message{ session_id, role: "user"|"lili", text, ts }`. (**`user_id` is added in v0.2** — v0.1 is single-session, no user concept yet.)
- Persist each turn's user + Лілі messages; reload a session's history.

**Dependencies:** LUMI-001

**Expected result:**
Session messages persist to the local store and reload across a restart.

**Acceptance criteria:**
- [ ] `Repository` interface defined; `core` depends on it, not the concrete store
- [ ] Local JSON/SQLite implementation persists `Session` + `Message` records
- [ ] A session's messages reload after a restart
- [ ] Unit test: persist → load round-trip against the local store
- [ ] No `user_id` yet (noted as a v0.2 addition); shapes match ARCHITECTURE §Data model

### LUMI-005 — Core turn — `reply(...)` orchestration

**Description:**
The one `reply(user_text, session)` contract that ties **canon + `LLMClient` + `Repository`** into a turn — the seam every interface calls (the TUI now, the server in v1). No interface logic in `core`. (`/core`.)

**What needs to be done:**
- `core.reply(user_text, session) -> str`: build the system prompt (canon, LUMI-003) + the session's messages, call the `LLMClient` (LUMI-002), persist the user and Лілі messages via the `Repository` (LUMI-004), and return the reply.
- In-session history in context — the full session for now (rolling-window trimming arrives in v0.2).
- Keep `reply(...)` the single public contract; the TUI/CLI/web later call exactly this.

**Dependencies:** LUMI-002, LUMI-003, LUMI-004

**Expected result:**
A full turn runs end to end against the mock model, with messages persisted.

**Acceptance criteria:**
- [ ] `reply(user_text, session)` assembles canon + session history + the user line, calls `LLMClient`, persists both messages, returns the reply
- [ ] In-session history is included in the model call
- [ ] `core` touches the model only via `LLMClient` (no direct SDK use)
- [ ] Integration test: `user_text → reply` against `MockLLMClient` (no paid call); both messages persisted to the store

### LUMI-006 — TUI loop (Textual)

**Description:**
The Textual terminal client — input, scrollable history, exit — calling `core.reply(...)` **in-process** (v0). Delivers the v0.1 DoD: hold a dialogue with Лілі. (`/tui`.)

**What needs to be done:**
- A Textual app: an input widget, a scrollable conversation view, and exit handling.
- On submit: call `core.reply(user_text, session)` **off the UI thread** (a worker / `asyncio.to_thread`) so the UI never freezes during the model call; append `You: …` and `Лілі: …` lines.
- A graceful, readable error line if the model call fails (e.g. "Лілі is unavailable") — never crash the loop.
- The TUI imports only `core` (and renders) — no model/SDK or storage logic.

**Dependencies:** LUMI-005

**Expected result:**
You can hold a dialogue with Лілі (Claude Haiku) in the terminal; she keeps her character within a session.

**Acceptance criteria:**
- [ ] Textual loop: text input, scrollable history, clean exit
- [ ] Submitting a line calls `core.reply` without freezing the UI; the reply renders
- [ ] A model-call failure surfaces as a readable line, not a crash
- [ ] (Manual) a real Claude Haiku turn works with `ANTHROPIC_API_KEY` set
- [ ] TUI smoke test (Textual pilot) drives a turn against `MockLLMClient`
