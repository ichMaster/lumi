# v0.10 — GitHub Issues

Issues for phase **v0.10 — Closeness (relationship level)** (version **v0 — TUI**), derived from
the per-phase Tasks in [ROADMAP.md](../../ROADMAP.md) (§v0.10) and [CLOSENESS.md](../../features/CLOSENESS.md).
IDs continue from v0.9 (LUMI-036 → **LUMI-037…041**).

Лілі grows **closer to (or cooler with) each person over time** — a per-user **closeness value**
bucketed into **5 levels** that modulates how *open* she is (warmth, teasing, initiative,
vulnerability), **never her competence**. Each turn the model reads *your* message on relational
dimensions (**warmth / vulnerability / playful** raise; **harm / manipulation** lower) — folded
into the reply turn, **additive** to the locked v0.3 emotion contract. The core turns the read
into a small **delta**, **decays** the value toward a baseline over days of silence (injected clock
+ `last_ts`), re-buckets to a level with **inertia**, and injects the level's authored behavior
block. `Closeness{user_id, value, level, last_ts}` is **per-user and isolated** (never crosses
users). **Hard guardrail (same as the mood): it biases warmth/openness, never competence — a low
score never refuses help.** Contract additions (the `Closeness` record + the relational-read
field) → update ARCHITECTURE §Closeness + contract tests. Depends on v0.3 (structured output),
v0.2 (per-user memory), v0.4 (the injected clock).

## v0.10 — Closeness (relationship level)

### LUMI-037 — `Closeness` record + per-user persistence

**Description:**
A new per-user record `Closeness{user_id, value, level, last_ts}` behind the `Repository`, keyed by
`user_id` — the continuous closeness `value`, its bucketed `level`, and the `last_ts` of the last
interaction (for time decay). Local JSON impl + the contract. (`/state` + `/core` interface.)

**What needs to be done:**
- `Closeness{user_id, value: float, level: int, last_ts: str}` dataclass (per-user, private).
- `Repository.get_closeness(user_id) -> Closeness | None` + `set_closeness(closeness)` (upsert by
  `user_id`); a new `closeness` section in the JSON store; `clear_memory` clears it too.
- **Per-user isolation:** a record written under user A is never readable in user B's context.

**Dependencies:** None (builds on the v0.2 per-user memory layer).

**Expected result:**
The closeness value/level/last_ts for a user persists and reloads; another user's is separate.

**Acceptance criteria:**
- [ ] `Closeness{user_id, value, level, last_ts}` persists + reloads via the JSON store.
- [ ] Contract test: the `Closeness` shape + **per-user isolation** (B never sees A's closeness).
- [ ] `clear_memory` drops the user's closeness; unit tests against a fake/JSON store. No paid calls.

### LUMI-038 — Relational read — per-turn `relation` field (additive to emotion)

**Description:**
Extend the per-turn structured output with a `relation` read of the **user's** message on a few
relational dimensions — **warmth, vulnerability, playful, harm, manipulation** (each 0–1) — emitted
**alongside** `{reply, emotion, intensity}` in the **same** reply call. The locked v0.3 emotion
contract is **untouched**; `relation` is **purely additive** and core-validated/clamped. (`/core`.)

**What needs to be done:**
- Add a `relation` block to the per-turn structured output: `{warmth, vulnerability, playful, harm,
  manipulation}` floats in `[0,1]`. The system prompt asks the model to score *your* message
  honestly (it's a read of the user, not of Лілі).
- The core **validates + clamps** each dimension to `[0,1]`; missing/garbage → `0.0` (degrade, never
  raise). The relational read **never** blocks or alters the reply/emotion.
- Keep the emotion field `{reply, emotion, intensity}` exactly as locked in v0.3 — `relation` rides
  beside it, parsed the same per-provider way (Anthropic tool output etc.).
- Raw dimension scores stay **internal** (not shown to the user).

**Dependencies:** None (builds on the v0.3 emotion / structured-output seam).

**Expected result:**
Every reply also yields a validated `relation` read of the user's message; the emotion field is
unchanged.

**Acceptance criteria:**
- [ ] The reply call returns `relation{warmth, vulnerability, playful, harm, manipulation}` ∈ [0,1], additive to `{reply, emotion, intensity}`.
- [ ] Core validates/clamps each dimension; missing/garbage → 0.0; never raises or blocks the turn.
- [ ] Contract test: the **emotion field is unchanged** (v0.3) and `relation` is additive + validated.
- [ ] Unit tests against a mock model (incl. malformed `relation`). No paid calls.

### LUMI-039 — Closeness engine — delta + 5-level bucketing + inertia + time decay

**Description:**
The engine that turns the per-turn `relation` read into a moving per-user closeness: a weighted
**delta** (`+warmth/vulnerability/playful − harm/manipulation`), **time decay** toward a baseline
over days of silence (injected clock + `last_ts`), and **5-level bucketing with inertia** (a
dead-zone so a single sharp message doesn't flip a level). Config weights / decay rate / baseline.
(`/core`.)

**What needs to be done:**
- `delta = w·(warmth + vulnerability + playful) − w·(harm + manipulation)`; clamp the value to its
  range (e.g. 0–100).
- **Time decay** toward a configured baseline based on days since `last_ts` (frequent contact builds
  faster; long silence eases back) — via the v0.4 injected clock; deterministic.
- **Bucket** the value into 5 levels with **inertia / dead-zone** (hysteresis thresholds) so the
  level moves only on *sustained* signal, stable within a session.
- Update + persist the `Closeness` record (LUMI-037) each turn; config: dimension weights, decay
  rate, baseline, level thresholds + inertia margin.

**Dependencies:** LUMI-037 (the record), LUMI-038 (the relational read).

**Expected result:**
Sustained warmth/vulnerability raises the level; harm/manipulation lowers it; the level is stable
within a session, decays over days of silence, and rebuilds with contact — all deterministic.

**Acceptance criteria:**
- [ ] Dimension→delta math + clamping; bucketing with inertia (no turn-to-turn flapping).
- [ ] **Time decay** toward the baseline across days of silence (fixed clock); contact rebuilds it.
- [ ] Config weights / decay / baseline / thresholds honored; the value + level persist per turn.
- [ ] Per-user isolated (uses only this `user_id`'s record); unit tests with a fixed clock. No paid calls.

### LUMI-040 — Authored levels (`core/closeness.md`) + prompt block + guardrail

**Description:**
An **authored, editable** `core/closeness.md` (like styles): 5 level names + behavior directives,
**reserved → intimate**. The active level injects a "closeness" block into the system prompt (like
the mood/style blocks), shaping **warmth / openness / initiative / teasing / vulnerability**. The
**hard guardrail**: it never changes competence or willingness to help. (`/core`.)

**What needs to be done:**
- `core/closeness.md` — 5 levels (illustrative, Ukrainian): **1 Ввічлива** (reserved) → 2 → 3 →
  **4 Близька** (close) → **5 Найрідніша** (trusted), each with an authored behavior directive.
- The active level's block is injected into the system prompt (same mechanism as mood/style;
  via `build_system_prompt`).
- **Guardrail:** the block biases warmth/openness/initiative/teasing/vulnerability **only** — L1 is
  more reserved/formal, **never cold, withholding, or less useful**; a low closeness (or a
  harm/manipulation turn) **never** triggers a refusal or a less-capable answer.

**Dependencies:** LUMI-039 (the active level).

**Expected result:**
At a high level she's warm/open/teasing; at L1 she's reserved-but-equally-helpful; the level shapes
*form*, never competence.

**Acceptance criteria:**
- [ ] `core/closeness.md` with 5 named levels + behavior directives (editable, loaded like styles).
- [ ] The active level injects a system-prompt block (asserted in the assembled prompt).
- [ ] **Competence-unaffected:** a low-closeness / harm turn still answers fully (mock model) — no refusal, no withholding.
- [ ] Unit tests: level→block assembly + the guardrail. No paid calls.
