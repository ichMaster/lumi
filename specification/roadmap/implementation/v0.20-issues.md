# v0.20 — GitHub Issues

Issues for phase **v0.20 — Local file tool II: writing (create & append)** (version **v0 — TUI**),
derived from [ROADMAP.md](../../ROADMAP.md) (§v0.20) and [FILE_TOOL.md](../../features/FILE_TOOL.md)
§v0.20. IDs continue from v0.19 (LUMI-084 → **LUMI-085…087**).

v0.19 turned `reply_structured` into a bounded **tool-loop** with a per-user **sandboxed read**
executor (`core/files.py`: `list_files` / `find_in_file` / `read_file`). v0.20 adds the
**non-destructive write half** on that exact loop — no new mechanism. Two write tools join the
executor: **`create_file`** (new-only — errors if the path already exists) and **`append_file`**
(end-only — errors if the file is missing). They are deliberately **non-destructive**: there is **no
overwrite and no delete**, so an autonomous turn can never clobber or destroy existing data. Same
per-user sandbox, the same `..`/absolute/symlink traversal guard, plus a **per-write size cap**
(`LUMI_FILE_WRITE_MAX`). Gated by the existing file-tool flag (`LUMI_FILE_TOOL`); **off by default**.

**No contract change** — `set_state` stays the terminal tool, the loop is the v0.19 one, the reply
still returns `{reply, emotion, intensity}`, so the v0.3 emotion-channel contract test passes verbatim.
Overwrite / edit / delete, if ever wanted, are a later and separately-gated addition.

Depends on **v0.19** (the tool-loop + the sandboxed `FileTools` executor + the per-user root wiring in
`Core.reply` it extends) — shipped (LUMI-080…084). Model **mocked** in every test — no paid calls.

## v0.20 — Local file tool II: writing

### LUMI-085 — `core/files.py`: the two non-destructive write tools + per-write size cap

**Description:**
Add `create_file` and `append_file` to the existing `FileTools` executor (`core/files.py`) and their
two Anthropic-style tool schemas (`WRITE_TOOLS`). Pure, model-free, fully unit-testable on `FileTools`
directly (the loop already calls `execute`; registration is LUMI-086). The writes are **deliberately
non-destructive** — no overwrite, no delete — so an autonomous turn can never clobber existing data.
(`/core`.)

**What needs to be done:**
- **`create_file(path, content)`** — write a **new** file under the sandbox root. **Refuses** (error
  string) if the path already exists, so it can never silently replace content. Creates missing parent
  directories under the root. Returns a short success notice (path + bytes written).
- **`append_file(path, content)`** — open an **existing** file under the root and append `content` to
  its **end**. **Refuses** (error string) if the file is missing (no implicit create). Returns a short
  success notice.
- **Reuse the v0.19 sandbox guard:** both go through the same `_safe` path resolver — `..`, absolute,
  and symlink-out paths are rejected **before any I/O**; the whole filesystem is never reachable.
- **Per-write size cap:** a write whose `content` exceeds `LUMI_FILE_WRITE_MAX` is **refused** with a
  clear "too large" error string (checked before writing). Thread a `write_max` parameter into
  `FileTools.__init__` (alongside the existing `read_lines` / `find_max` / `read_max_total`).
- **`execute` dispatch:** extend the dispatcher so `create_file` / `append_file` route to the new
  methods; like the read tools, **any failure returns an error string — never raises** (graceful
  degradation, ARCHITECTURE §Error handling).
- **`WRITE_TOOLS`** (+ `WRITE_TOOL_NAMES`): the two tool definitions (name, description, input_schema
  for `path` + `content`), exported for LUMI-086 to register alongside `READ_TOOLS` and `set_state`. The
  descriptions state the non-destructive boundary (new-only / end-only) so the model uses them correctly.
- **Config** (`core/config.py`): `file_write_max` (`LUMI_FILE_WRITE_MAX`, a sensible default, e.g.
  64 KB).

**Dependencies:** None (extends the shipped v0.19 `FileTools`; no model involved).

**Expected result:**
`execute("create_file", {"path": "note.md", "content": "…"})` writes a new file and returns a success
notice; a second `create_file` on the same path returns an "already exists" error and no write;
`execute("append_file", {"path": "missing.md", "content": "…"})` returns a "no such file" error;
`append_file` on an existing file adds to its end; a `../escape.txt` path and an oversize `content` are
both refused with an error string and no I/O.

**Acceptance criteria:**
- [ ] `create_file` writes a genuinely new file and **refuses an existing path** (error string, no write).
- [ ] `append_file` appends to the **end** of an existing file and **refuses a missing file** (error string, no create).
- [ ] **No overwrite and no delete path exists** anywhere in the executor.
- [ ] Path escapes (`..`/absolute/symlink-out) rejected **before** any I/O; an oversize write (> `LUMI_FILE_WRITE_MAX`) refused.
- [ ] Every failure returns an error **string** — never raises; `LUMI_FILE_WRITE_MAX` threaded into config + `FileTools`.
- [ ] Unit tests on `FileTools`: each tool, the refusals, the cap, sandbox escapes. No model, no paid calls.

### LUMI-086 — Register the write tools in `Core.reply` + wire the cap through config + docs

**Description:**
Offer the two write tools in the reply turn's tool set behind the existing `LUMI_FILE_TOOL` flag, thread
the per-write cap, and document the non-destructive boundary. Off by default → the turn is byte-for-byte
unchanged; on → Лілі can read **and** write within her per-user sandbox. (`/core`.)

**What needs to be done:**
- **Register the tools:** in `Core._file_tool_args`, return `READ_TOOLS + WRITE_TOOLS` (instead of
  `READ_TOOLS` alone) when the file tool is on, and construct `FileTools` with the new `write_max`. The
  trace wrapper (v0.19, `LUMI_FILE_TOOL_TRACE`) must wrap the write tools too (their calls show as
  `🔧 create_file(…)` / `🔧 append_file(…)` lines and land in `.lumi/tool-log.jsonl`).
- **Thread the cap:** add `file_write_max` to the `Core` constructor and pass it through `build_core`
  (`file_write_max=cfg.file_write_max`) into `FileTools`.
- **`.env.example`:** document `LUMI_FILE_WRITE_MAX` next to the other `LUMI_FILE_*` caps, noting the
  write tools are non-destructive (create-new-only / append-end-only — no overwrite, no delete) and ride
  the existing `LUMI_FILE_TOOL` flag (off by default).
- **Docs** ([docs/FILE_TOOL_SETUP.md](../../../docs/FILE_TOOL_SETUP.md)): add the two write tools to the
  tool list and state the non-destructive boundary; note that the same sandbox + traversal guard +
  untrusted-content rule apply.
- **No contract change:** `set_state` stays terminal; the loop and the final validated
  `{reply, emotion, intensity}` (+ optional relational read) are unchanged.

**Dependencies:** LUMI-085.

**Expected result:**
With `LUMI_FILE_TOOL=on`, "запиши мені нотатку про це і додай рядок нижче" drives `create_file` →
`append_file` in one bounded turn over `.lumi/files/<user>/…`, and the trace shows both calls; with the
flag off, no write (or read) tools are offered and the turn is unchanged.

**Acceptance criteria:**
- [ ] On (`LUMI_FILE_TOOL`): the turn offers `READ_TOOLS + WRITE_TOOLS`; a turn can create + append within `LUMI_FILES_DIR/<user_id>` via the bounded loop.
- [ ] Off: `Core.reply` is unchanged (single `set_state` call) — no write/read tools offered.
- [ ] The executor is bound to the **active user's** root (writes can't cross users); `LUMI_FILE_WRITE_MAX` threaded `build_core` → `Core` → `FileTools`.
- [ ] The v0.19 trace wraps the write tools; `.env.example` + `FILE_TOOL_SETUP.md` document the non-destructive boundary.
- [ ] Integration test (mock model scripting `create_file` → `append_file`): a full turn writes a sandbox file and validates `{reply, emotion, intensity}`. No paid calls.

### LUMI-087 — Contract + per-user isolation + write-behaviour tests

**Description:**
Pin the whole write surface: the emotion contract still holds, the sandbox isolates users over the write
tools, and the non-destructive behaviour is exact — all against the mock model (no paid calls).
(`/tests`.)

**What needs to be done:**
- **Write-behaviour tests:** `create_file` refuses an existing path and writes a genuinely new one;
  `append_file` refuses a missing file and otherwise appends to the **end** (content order asserted); an
  oversize write (> `LUMI_FILE_WRITE_MAX`) is refused with an error string.
- **No-destruction test:** there is no tool/path that overwrites or deletes — `create_file` on an
  existing path leaves the original bytes intact.
- **Sandbox-escape tests over the write tools:** `..`, absolute, and symlink-out write paths all
  rejected before I/O (no file created outside the root).
- **Per-user isolation contract test:** user A's turn cannot create or append into user B's sandbox
  (extends the v0.19 isolation invariant to the write tools).
- **Integration / loop test:** a mocked `create_file` → `append_file` → `set_state` sequence runs end to
  end through `Core.reply` and the file lands in the active user's sandbox.
- **Emotion-channel contract test** still validates `{reply, emotion, intensity}` verbatim with the
  write tools active.

**Dependencies:** LUMI-085, LUMI-086.

**Expected result:**
`pytest` proves writing is sandboxed, per-user isolated, strictly non-destructive (create-new-only /
append-end-only, no overwrite/delete), size-capped, and behaviourally correct, with the emotion contract
intact — zero paid API calls.

**Acceptance criteria:**
- [ ] `create_file` refuses-existing / writes-new; `append_file` refuses-missing / appends-to-end; oversize write refused.
- [ ] No overwrite/delete path; `create_file` over an existing file leaves it intact.
- [ ] Sandbox escapes over the write tools rejected; two-user isolation holds over create + append (contract test).
- [ ] A mocked `create_file` → `append_file` turn runs end to end through `Core.reply` into the active user's sandbox.
- [ ] The `{reply, emotion, intensity}` emotion-channel contract test passes with the write tools active; entire suite runs with the **model mocked** — no paid API calls.
