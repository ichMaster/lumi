# Version v1.4 — Issue Breakdown

**Phase:** v1.4 — Streaming replies (TUI) + sentence-chunked voicer

Лілі's reply appears **as it generates** instead of after the whole turn — perceived TTFT drops from
~10 s to **~2–3 s** — via streaming behind the [`LLMClient`](../../../core/llm.py) seam
([LATENCY.md](../../features/LATENCY.md) **S3, Phase A**), and the local voicer speaks it
**sentence-by-sentence** as it arrives (the **voicer half of S6, Phase B**). The **prose** streams;
because the reply is **structured** (`{reply, emotion, intensity}`), `emotion`/`intensity` is resolved
and **validated at completion** — the v0.3 gate and the locked emotion contract are **untouched**, the
face just updates when the stream ends. **Telegram is unaffected** (a complete-message channel — it
keeps receiving the one finished record off the file bus). This is the prerequisite for the later
live-voice loop. Off (`LUMI_STREAM`, default) → today's blocking path, **byte-identical to v1.3**.
Full design: [ROADMAP §v1.4](../../ROADMAP.md); background: [features/LATENCY.md](../../features/LATENCY.md).

### LUMI-188 — `LLMClient` streaming seam

**Description:**
The engine of the phase. Today [`LLMClient`](../../../core/llm.py) returns the whole reply in one shot.
Add an **optional streaming path** so the reply's **prose** arrives incrementally — an `on_delta`
callback / iterator (Anthropic / Gemini / OpenAI all support SSE; **mock streams in tests**, no paid
CI). Because the reply is **structured** (`{reply, emotion, intensity}` via Gemini `responseSchema` /
Anthropic tool), stream the **prose** and resolve `emotion`/`intensity` **at completion** — the v0.3
gate still validates (unknown → `calm`, clamp intensity). A **streaming `<think>` tag filter** routes
the reasoning to the think channel and **never leaks a half-tag** into the shown reply. In a
**tool-loop**, only the **terminal answer round** streams (intermediate rounds are tool calls,
non-streaming). A **malformed / aborted / short stream degrades to today's blocking path** — the turn
never dies. Gated on `LUMI_STREAM` (default off).

**What needs to be done:**
- Extend the `LLMClient` seam with an optional streaming call (an `on_delta` callback or a generator)
  **beside** the existing blocking call; `MockLLMClient` gains a **scripted-deltas** mode for tests.
- Stream the **prose**; buffer and resolve `{emotion, intensity}` at stream completion via the existing
  structured path; `validate()` runs on the assembled result (unknown → `calm`, clamp).
- A **partial-tag-safe** streaming `<think>` filter: never emit a half `<think>`/`</think>` or a tag
  fragment into the shown reply; the reasoning routes to the think sink.
- **Tool-loop:** only the terminal answer round streams; intermediate tool rounds stay blocking.
- **Fallback:** any malformed / aborted / truncated stream → fall back to the blocking call; the turn
  completes with the same `{reply, emotion, intensity}` contract.
- **Config:** `LUMI_STREAM` (off by default) in `Config` + `load_config`.

**Dependencies:** v0.3 (the emotion validation gate) — shipped.

**Expected result:**
With `LUMI_STREAM` on, the core can drive a reply as a delta stream against the mock client; the
assembled reply **equals** the blocking reply and the emotion is validated at completion; a broken
stream silently falls back to blocking.

**Acceptance criteria:**
- [ ] A mock streaming client emits deltas → the assembled reply **==** the non-streamed reply; `{emotion, intensity}` parsed/validated at completion (unit).
- [ ] The `<think>` filter **never leaks a partial tag** into the shown text (unit — adversarial partial-tag chunk boundaries).
- [ ] A **tool turn streams only its terminal round** (unit).
- [ ] A **malformed/aborted stream degrades to the blocking path**; the turn succeeds (contract).
- [ ] **Off (`LUMI_STREAM`) → the blocking path, byte-identical output/records to v1.3** (pinned).
- [ ] **No paid calls** (all streams mocked).

### LUMI-189 — TUI incremental render

**Description:**
The visible half. Render the streamed reply **incrementally** in the TUI: append deltas into the reply
widget on the **v1.2 non-blocking worker** (off the UI thread), and fire the **emotion/face render on
stream completion** (so the face updates when `{emotion, intensity}` resolves). The think-box receives
the routed reasoning as it streams. The core stays interface-independent — this is the TUI consuming
LUMI-188's `on_delta`.

**What needs to be done:**
- Wire LUMI-188's `on_delta` into the TUI: a streaming turn appends text deltas to the reply widget via
  the v1.2 worker (never blocking the UI thread).
- The emotion/face renderer fires **once, on completion** (the contract resolves at the end).
- Route the streamed `<think>` reasoning to the existing think-box (the LUMI-188 filter keeps it clean —
  no partial-tag leakage).
- With `LUMI_STREAM` off, the TUI uses today's one-shot render (unchanged).

**Dependencies:** **LUMI-188** (the streaming seam), v1.2 (the non-blocking worker).

**Expected result:**
With the flag on, the reply text appears within **~2–3 s** and grows incrementally; the face updates at
the end; the UI stays responsive throughout.

**Acceptance criteria:**
- [ ] Deltas append to the reply widget **on the non-blocking worker** (integration with a mock stream; the UI thread is not blocked).
- [ ] The emotion/face renders **exactly once, on completion**.
- [ ] Streamed reasoning routes to the think box with **no partial-tag leakage**.
- [ ] **Off → today's one-shot render** (unchanged).
- [ ] **No paid calls** (mock stream).

### LUMI-190 — Sentence-chunked voicer (Phase B)

**Description:**
The voice half (the voicer slice of S6). The local voicer speaks the reply **sentence-by-sentence** as
text becomes available, instead of waiting for the whole reply. A **pure sentence chunker** (ordered,
never mid-word, flushes the tail) feeds the TTS incrementally. The **outbox record stays one complete
reply** (single writer, FIFO) — Telegram and the outbox contract are **untouched**; only the voicer's
consumption is incremental.

**What needs to be done:**
- A **pure sentence chunker**: split a growing text stream into ordered **whole** sentences, never
  mid-word, flush the trailing partial at end. Unit-tested in isolation (no I/O).
- The voicer consumes the stream per sentence and calls the (**mock in CI**) TTS adapter **per sentence,
  in order**.
- The **outbox record remains one complete reply** — the TUI still writes a single finished record; the
  streaming is voicer-internal. **Telegram byte-identical.**

**Dependencies:** **LUMI-188** (the stream source), v0.14 (the voicer / outbox bus) — shipped.

**Expected result:**
The voicer begins speaking the first sentence while later sentences are still arriving; the outbox
record is unchanged (one reply); Telegram sees no difference.

**Acceptance criteria:**
- [ ] The **sentence chunker** splits into ordered whole sentences and flushes the tail (unit; never mid-word).
- [ ] The voicer speaks **sentence-by-sentence, in order** against a mock TTS (unit/integration).
- [ ] The **outbox record is one complete reply** regardless of streaming; **Telegram byte-identical** (contract).
- [ ] **Off → the voicer speaks the whole reply as today** (unchanged).
- [ ] **No paid calls** (mock TTS).
