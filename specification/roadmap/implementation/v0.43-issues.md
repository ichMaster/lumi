# Version v0.43 — Issue Breakdown

**Phase:** v0.43 — Model roles: register-routed replies (talking / thinking / emotional)

The reply model stops being one fixed choice: each turn answers in one of three **registers** —
**`talking`** (everyday voice, fast/cheap), **`thinking`** (hard analytical tasks), **`emotional`**
(loaded moments) — picked automatically by a staged classifier on a fourth **`classifier`** role (a
tiny model), with **stickiness** so a register holds through its moment. A **lexical stage** (code, no
LLM) routes the clear cases; only *unsure* turns consult the classifier (**hybrid firing**: clear →
one request; unsure → talking ∥ classifier in parallel, escalate on the verdict, discard the draft).
Routing reads the **user's message, never her mood — never competence**. Roles live in the v0.41
profiles; the v0.40 op-tiers and `/model-set` compose unchanged; `/model` pins + suspends routing.
Off (`LUMI_MODEL_ROLES`, default) → today's single reply model, **byte-identical**. Depends on v0.41
(profiles + `switch_profile`), v0.40 (`_model_for`), v0.3 (the emotion gate). Full design:
[MODEL_ROLES.md](../../features/MODEL_ROLES.md).

### LUMI-170 — Roles in the profiles: `ModelProfile` + `core/models.toml` + config

The data foundation — the four role models per provider, and the feature's config. Extends the v0.41
`ModelProfile` (`{provider, reply, think, mood, housekeeping}`) with **`talking` / `thinking` /
`emotional` / `classifier`**, each **additive and defaulted from `reply`** (a profile without them behaves
exactly as today). Ships the authored anthropic set in `core/models.toml`. No routing yet — just the
data + flags.

**What needs to be done:**
- Extend `ModelProfile` with `talking` / `thinking` / `emotional` / `classifier` (each defaults to the
  profile's `reply` when the TOML/env omits it — off-safe).
- `_load_models_file` (v0.41): read the four role fields from each `[profiles.*]` block; missing → default

  (e.g. talking `claude-sonnet-5`, thinking `claude-opus-4-8` (thinking on), emotional `claude-opus-4-8`,

  `LUMI_ROLES_FILE` (`core/roles.md`) — wired into `Config` + `load_config`.

**Dependencies:** none — extends v0.41 profiles + the models.toml loader.

**Expected result:**
`Config().model_profiles["anthropic"]` carries the four role ids (defaulting to `reply` when omitted); the
role config vars load with their defaults; unset → `LUMI_MODEL_ROLES=off`.

**Acceptance criteria:**
- [ ] `ModelProfile` has the four role fields; a profile that omits them defaults each to `reply` (unit-tested).
- [ ] `core/models.toml` ships the anthropic role set; the loader reads it; `LUMI_MODEL_PROFILES` env still merges.
- [ ] The role config vars load with documented defaults; unset → roles off.
- [ ] **No paid calls.**

### LUMI-171 — Lexical stage: `core/roles.py` + `core/roles.md`

**Description:**
The pure, no-LLM classifier *floor* — a table-driven function that routes the clear cases instantly
(~0 ms) so the classifier only fires when genuinely unsure. Reads an authored `core/roles.md` (the
closers/greetings, distress markers, thinking markers, override phrases) and applies length / code-block /
multi-question heuristics + the active hold → one of `talk | think | emotional | unsure`.

**What needs to be done:**
- A `core/roles.py` with a pure `classify_lexical(message, *, hold=None) -> str` returning
  `talk | think | emotional | unsure`:
  - **talk** — short message (≤ ~40 chars) or a closer/greeting from the authored list («дякую»,
    «добраніч», «привіт»); short messages **always** land here (they bypass the hold).
  - **emotional** — distress markers («мені важко», «я не можу більше», crisis vocabulary — authored).
  - **think** — code blocks / stack traces, multi-part questions, «поясни/спроєктуй/порахуй» verbs, length ≫.
  - **override** — `/role think|emotional|talk` or authored phrases («подумай як слід») → as asked.
  - **hold** — an active held register + a *substantive* message → the held register (short-circuit).
  - else → **unsure**.
- An authored **`core/roles.md`** (path = `LUMI_ROLES_FILE`): sections for closers/greetings, distress,
  thinking markers, override phrases — Ukrainian-first, editable; a loader (missing → sensible built-ins).
- The heuristics (length threshold, code-fence/`Traceback` detection, `?`-count) as small pure helpers.

**Dependencies:** **LUMI-170** (the `LUMI_ROLES_FILE` config).

**Expected result:**
`classify_lexical("дякую")` → `talk`; `"поясни, чому кеш інвалідовується"` → `think`; `"мені сьогодні
важко"` → `emotional`; an ambiguous mid-length statement → `unsure`; a substantive message while `hold`
is set → the held register.

**Acceptance criteria:**
- [ ] Table-driven cases: closers/greetings + short → talk; distress → emotional; code/multi-question/verbs → think; override → as asked; unsure otherwise (unit-tested, no LLM).
- [ ] Short messages classify **fresh** and never return the held register (hold bypass).
- [ ] `core/roles.md` loads its lists; a missing file → built-in defaults; never raises.
- [ ] **No paid calls.**

### LUMI-172 — Classifier call: one structured `{label, confidence}`

**Description:**
When the lexical stage says *unsure*, one structured request to the **`classifier`** role decides the
register. **Minimal by design**: the prompt carries only the user's message + the last few register labels
— **no memory, no facts, no persona state** (pinned by a contract test, the wiki/news de-id family). Below
`LUMI_ROLE_CONFIDENCE` → talk; any malformed/failed response → talk (never blocks a turn).

**What needs to be done:**
- A `Core` method (e.g. `_classify_role(message, recent_labels) -> tuple[str, float]`): build the minimal
  classifier prompt (message + recent labels only), call `LLMClient.reply_structured` on the profile's
  `classifier` model with a `{label: talk|think|emotional, confidence: 0..1}` schema, parse + validate
  (unknown label → talk; clamp confidence).
- **Floor + degrade:** confidence < `LUMI_ROLE_CONFIDENCE` → talk; a parse error / LLM error → talk (logged).
- **No personal data:** the prompt builder must not pull memory/facts/closeness/mood — a contract test
  asserts the classifier system+user text contains none of the per-user fields.

**Dependencies:** **LUMI-170** (the `classifier` role in the profile); reuses the `LLMClient` structured
path + the v0.3 validation pattern.

**Expected result:**
`_classify_role("розкажи як влаштований кеш", [])` → `("think", 0.9)`; a low-confidence answer → treated as
`talk`; a classifier LLM error → `talk`; the classifier prompt contains no memory/personal fields.

**Acceptance criteria:**
- [ ] Returns a validated `(label, confidence)`; unknown label → talk; confidence clamped.
- [ ] Below the confidence floor → talk; a malformed/failed classifier call → talk (never raises).
- [ ] **Contract:** the classifier prompt carries no memory/facts/closeness/mood (pinned test).
- [ ] **No paid calls** (classifier mocked).

### LUMI-173 — Hybrid firing + stickiness (the reply-path router)

**Description:**
The headline — wire the lexical stage + classifier + profile roles into `Core.reply` with the hybrid
firing and the register hold. A clear lexical verdict fires **one** request on that register's model; an
*unsure* turn fires **talking + the classifier in parallel** and, on an escalating verdict, re-sends the
message to the chosen register (discarding the talking draft — the accepted rare double-pay). The chosen
register's model is the reply call's `model`, so **the tool-loop follows it** (v0.40-style) and the
`{reply, emotion, intensity}` contract is untouched. **Stickiness:** an escalation holds the register for
`LUMI_ROLE_COOLDOWN` substantive turns; short messages bypass the hold (answered by talking) but the
countdown keeps running; a new escalation re-targets + resets; then revert to talking.

**What needs to be done:**
- The **register resolver** in `reply`: `talk|think|emotional` → the active profile's matching role model
  (via a `_model_for_role` helper mirroring `_model_for`); the reply `model` becomes that (was `self._model`).
- **Hybrid firing:** lexical-clear → one `reply_structured` on the register model; *unsure* → fire the
  talking reply and the classifier **in parallel**, then escalate on the verdict (re-send on the register
  model, discard the talking draft) or keep the talking reply (no escalation → zero added latency).
- **Register hold** (per-conversation, in-memory — the v0.10 relation-read pattern): a `{register, turns_left}`
  advanced each turn; a substantive message during the hold routes to the held register (lexical
  short-circuit); a short message bypasses but decrements; a new escalation re-targets + resets;
  expiry → talking. Not persisted; never crosses users.
- **Routing reads only the user's message + recent labels** — never mood/needs (never competence); the
  emotion gate is unchanged (every register answers through `{reply, emotion, intensity}`).
- **Off / provider guard:** `LUMI_MODEL_ROLES` off → the reply uses `self._model` (byte-identical); a
  manual `/model` (v0.37) suspends routing for the session (an explicit choice wins).

**Dependencies:** **LUMI-170** (role models), **LUMI-171** (lexical stage), **LUMI-172** (classifier).

**Expected result:**
A hard task answers on `thinking`, a loaded message on `emotional`, small talk on `talking`; the lexical
stage alone routes clear cases (no classifier call); an escalated register **holds** N turns; a "дякую"
mid-hold is answered by talking while the hold keeps counting; off → byte-identical.

**Acceptance criteria:**
- [ ] The firing matrix (mock LLM records the model per turn): clear-talk → one talking call; clear-think/emotional → one call on that register; unsure → talking ∥ classifier, escalate on the verdict (draft discarded).
- [ ] The routed register's tool-loop follows its model (continuations on the register tier).
- [ ] The hold: countdown, short-message bypass (talking, countdown still runs), re-target/reset on a new escalation, revert to talking at expiry (fixed-clock/scripted, no sleeps).
- [ ] Routing never reads mood/needs; the emotion contract validates regardless of register; per-user isolation untouched.
- [ ] **Off → byte-identical** (reply on `self._model`); a manual `/model` suspends routing.
- [ ] **No paid calls** (LLM mocked); an integration test walks talk → think → hold → bypass → revert.

### LUMI-174 — Surfacing: status-bar register + `/roles` + `/model` suspend

**Description:**
Make the register visible and controllable. The status bar shows the answering register when it isn't
`talking` (`✦ emotional`, the v0.33 quiet style); a `/roles` command reads the state (the three registers
+ their models, the active hold + turns left, whether routing is suspended); a manual `/model` swap
**suspends** role-routing for the session (already the behavior from LUMI-173 — surfaced here + documented).

**What needs to be done:**
- **Status bar:** show `✦ <register>` when the last answering register wasn't `talking` (Core exposes the
  active register + hold, like `mood`/`profile`).
- **`/roles` command** (`tui/app.py`): list the three registers with their models (from the active profile),
  the active hold + turns left, and whether routing is suspended (a `/model` pin). Off → a note that roles
  are off.
- **`/model` suspend surfaced:** after a manual `/model`, the status/`/roles` reflect "routing suspended";
  `/model-set <profile>` re-enables it (the new profile brings its roles). Update MODELS_SETUP.md +
  the `/`-commands help line.

**Dependencies:** **LUMI-173** (the router + hold state it surfaces).

**Expected result:**
`/roles` shows talking/thinking/emotional/classifier with their models + the active hold; the status bar
marks a non-talking register; `/model sonnet` suspends routing and `/roles` says so; `/model-set` restores it.

**Acceptance criteria:**
- [ ] `/roles` lists the registers + models, the active hold (register + turns left), and the suspended state.
- [ ] The status bar shows `✦ <register>` for a non-talking answer; plain otherwise.
- [ ] A manual `/model` suspends routing (surfaced in `/roles` + status); `/model-set` re-enables it.
- [ ] MODELS_SETUP.md documents roles + `/roles`; **no paid calls** (mock core).

## Notes

- **Sizes:** LUMI-170 **S** (data + config), LUMI-171 **M** (lexical table + authored file), LUMI-172 **M**
  (one classifier call + the no-personal-data contract), LUMI-173 **L** (the reply-path router + hold — the
  bulk), LUMI-174 **S** (status bar + `/roles` + docs).
- **No contract change** — `{reply, emotion, intensity}`, the v0.3 gate, and per-user isolation are
  untouched; **off (`LUMI_MODEL_ROLES`) → byte-identical** (pinned per issue).
- **Never competence** — the register changes *which engine* answers, never *whether* she helps; her own
  mood/needs are **not** routing inputs (only the user's message + recent labels).
- **No personal data in the classifier prompt** — pinned by a contract test (LUMI-172).
- **Never blocks a turn** — lexical fail → unsure; classifier fail/floor → talk; escalation fail → the
  talking draft.
- **Provider-homogeneous** — roles are profile fields (the v0.41 rule); no cross-engine flapping mid-turn.
- **Build order:** LUMI-170 → (LUMI-171 ‖ LUMI-172) → LUMI-173 → LUMI-174.
