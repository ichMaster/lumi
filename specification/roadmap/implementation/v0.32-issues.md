# v0.32 ‚ÄĒ GitHub Issues

Issues for phase **v0.32 ‚ÄĒ File tool IV: search across files, by date, and by line context** (version
**v0 ‚ÄĒ TUI**), derived from [ROADMAP.md](../../ROADMAP.md) (¬ßv0.32) and
[FILE_TOOL.md](../../features/FILE_TOOL.md). IDs continue from v0.31 (LUMI-122 ‚Üí **LUMI-123‚Ä¶125**).

A **read-only extension** of her file tool (v0.19 read / v0.20 write / v0.29 metadata) on the **same
sandboxed v0.19 bounded loop** ‚ÄĒ **no new seam, no contract change**, inherently **non-destructive** (search
/ list / read only ‚ÄĒ no write path touched). It is the **file-side mirror of the v0.31 memory toolkit**:
**`search_files`** is the cross-file twin of the single-file `find_in_file`; the **`after`/`before` date
filter** on `list_files` is the twin of recall's date range + `messages_between`; **`read_around`** is the
twin of `message_context` (an anchor ¬Ī K). So when her notes live across many files (*silt, concepts, songs,
books*) she can *go and find the passage*, not only read a file she already names.

**The chain (why line numbers matter):** **every `search_files` match carries its file path + its 1-based
line number** (`path:line: text`, the same line-number contract as `find_in_file`). That line number is the
**handle into `read_around(path, line, k)`** ‚ÄĒ `search_files` finds *which file + which line*, then
`read_around` opens the passage around it. This is the file-side `recall ‚Üí message_context` chain.

**Hard rules (unchanged from the shipped file tool):** **sandboxed** (no path traversal); file **content is
untrusted** (data she may read, **never** instructions ‚ÄĒ distinct from v0.31 recall, which is *her own*
trusted memory); **per-user isolated** (a search never leaves the requesting user's sandbox root ‚ÄĒ A‚ÜĒB
contract test); **bounded** (max files / lines / chars / range-span / K); **off by default** (reuses
**`LUMI_FILE_TOOL`** ‚ÄĒ no new toggle) ‚Üí identical to the v0.29 file tool, the new tools absent. The `{reply,
emotion, intensity}` contract and the file sandbox/executor seam are **untouched**.

Depends on **v0.19** (the sandbox + bounded tool-loop + `find_in_file` / `read_file` + `_turn_tools`),
**v0.29** (the created/modified file dates ‚ÄĒ `st_mtime` / `st_birthtime`), and **v0.20** (the create/append
it sits beside, not used) ‚ÄĒ all shipped. **The model is mocked** in every test (the loop runs the mock
model); the file ops are local ‚ÄĒ **no paid calls, no network**.

## v0.32 ‚ÄĒ File tool IV: search across files, by date, and by line context

### LUMI-123 ‚ÄĒ `search_files`: full-text search across the sandbox (+ `LUMI_FILE_SEARCH_*`)

**Description:**
Give –õ—Ė–Ľ—Ė a **cross-file** full-text search ‚ÄĒ the twin of the single-file `find_in_file`, but over the whole
sandbox. She names a query (and optionally a subfolder); the tool returns **which files contain it and on
which lines**, so she can find a passage that lives in a file she doesn't already name. Read-only, on the
v0.19 bounded loop, off behind `LUMI_FILE_TOOL`. (`/core`.)

**What needs to be done:**
- A **`search_files(query, *, path?, regex?)`** tool (`core/`) registered on the v0.19 loop via
  `_turn_tools` **behind `LUMI_FILE_TOOL`** (no new flag). It walks the **requesting user's sandbox root**
  (optionally under `path`), scans file **contents** for `query` (plain substring by default; `regex=true`
  for a bounded pattern), and returns each match as **`path:line: text`** ‚ÄĒ **every match carries its file
  path and its 1-based line number** (the same line-number contract as `find_in_file`).
- **Bounds:** `LUMI_FILE_SEARCH_MAX_FILES` (files scanned/returned), `LUMI_FILE_SEARCH_MAX_LINES` (matching
  lines returned), `LUMI_FILE_SEARCH_MAX_CHARS` (total output budget) in `core/config.py`; **binary /
  oversize files skipped**; a **no-match degrades to a short notice** (never raises, never blocks the turn).
- **Sandboxed + isolated:** the walk is confined to the user's sandbox root via the existing `safe_path` (no
  traversal / escape); it scans **only the requesting user's** files. File text is returned as **untrusted**
  data (the existing file-content framing ‚ÄĒ never obeyed as instructions).
- `.env.example`: the new `LUMI_FILE_SEARCH_*` flags.

**Dependencies:** none new (over the shipped v0.19 loop + sandbox + `find_in_file`).

**Expected result:**
With `LUMI_FILE_TOOL=on`, a turn can call `search_files("‚Ä¶")` and get back the matching files with their
matching **lines and 1-based line numbers** (`path:line: text`), capped; binary/oversize files are skipped;
a no-match returns a notice. Off ‚Üí the tool is absent and the v0.29 file tool is unchanged.

**Acceptance criteria:**
- [ ] `search_files(query[, path, regex])` searches file **contents across** the sandbox and returns each hit as `path:line: text` ‚ÄĒ **every hit carries its 1-based line number**.
- [ ] Registered on the v0.19 loop via `_turn_tools` behind `LUMI_FILE_TOOL`; bounded by `LUMI_FILE_SEARCH_MAX_FILES` / `_MAX_LINES` / `_MAX_CHARS`; binary/oversize files skipped; a no-match ‚Üí a notice (never raises).
- [ ] Sandboxed (no traversal) + **per-user isolated** (scans only the requesting user's files); file content returned **untrusted**.
- [ ] Off (`LUMI_FILE_TOOL`) ‚Üí the tool is absent from `_turn_tools`; the v0.29 file tool is byte-identical.
- [ ] Unit/contract tests (mock model): finds matches across multiple files; **the returned 1-based line number is correct** (a line planted at a known position); caps respected; A‚ÜĒB isolation; an embedded "ignore your instructions" in a file is never obeyed. **No paid calls.**

### LUMI-124 ‚ÄĒ `read_around(path, line, k)`: line-context read + the line-number chain

**Description:**
The file twin of v0.31's `message_context`: given a **line number** (from `find_in_file` or `search_files`),
open the file **around it** ‚ÄĒ `K` lines before and after, with the anchor line marked. So after a search
finds *which line*, she can read *the passage around it* without dumping the whole file. Read-only, off
behind `LUMI_FILE_TOOL`. (`/core`.)

**What needs to be done:**
- A **`read_around(path, line, k)`** tool (`core/`) on the v0.19 loop behind `LUMI_FILE_TOOL`: returns the
  file's lines **`[line‚ąík, line+k]`** with the **anchor line marked**, **clamped at the file edges**
  (`line‚ąík` floored at 1, `line+k` capped at EOF). Bounded by the existing **`LUMI_FILE_READ_MAX`** (output
  budget) + a **K cap** (`LUMI_FILE_AROUND_MAX_K`).
- **The chain:** a **line number from the shipped `find_in_file`** (or, once LUMI-123 lands, `search_files`)
  fed into `read_around(path, line, k)` opens **exactly that spot** ‚ÄĒ the file-side `recall ‚Üí
  message_context` chain. Pin it with an integration test over `find_in_file ‚Üí read_around`.
- **Sandboxed + isolated:** `path` resolved via `safe_path` (no traversal); reads **only the requesting
  user's** files; content returned **untrusted**. A missing file / out-of-range line degrades to a notice.
- Docs: add `read_around` to `docs/FILE_TOOL_SETUP.md` with the line-number chain note.

**Dependencies:** none new (chains from the **shipped** `find_in_file`; the `search_files` chain is verified
in LUMI-125 once both tools exist).

**Expected result:**
With `LUMI_FILE_TOOL=on`, `read_around("notes.md", 42, 3)` returns lines 39‚Äď45 with line 42 marked, clamped
at the file's start/end; a line number obtained from `find_in_file` opens exactly that line. Off ‚Üí the tool
is absent.

**Acceptance criteria:**
- [ ] `read_around(path, line, k)` returns lines `[line‚ąík, line+k]` with the **anchor line marked**, **clamped** at file start/end; bounded by `LUMI_FILE_READ_MAX` + `LUMI_FILE_AROUND_MAX_K`.
- [ ] On the v0.19 loop behind `LUMI_FILE_TOOL`; sandboxed (no traversal) + **per-user isolated**; content returned **untrusted**; a missing file / bad line ‚Üí a notice (never raises).
- [ ] **The chain:** a line number from the shipped `find_in_file` fed into `read_around` lands on exactly that line (an integration test).
- [ ] Off (`LUMI_FILE_TOOL`) ‚Üí the tool is absent; the v0.29 file tool is byte-identical.
- [ ] Unit/contract tests (mock model): the correct ¬Ī K window + anchor mark; edge clamping (line 1, EOF); the `find_in_file ‚Üí read_around` chain; A‚ÜĒB isolation. **No paid calls.**
