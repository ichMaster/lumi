# v0.34 — GitHub Issues

Issues for phase **v0.34 — Lean memory: tool-pull, not push (the prompt slims down)** (version **v0 — TUI**),
derived from [ROADMAP.md](../../ROADMAP.md) (§v0.34) and
[PROMPT_OPTIMIZATION_II.md](../../../docs/PROMPT_OPTIMIZATION_II.md). IDs continue from v0.33
(LUMI-135 → **LUMI-136…138**).

The system prompt is **~73% injected memory** (detailed conversations + day/week digests + facts + thoughts),
re-sent and re-cached every turn. Now that the on-demand retrieval toolkit has shipped (v0.17 auto-RAG +
v0.31 `recall` / `messages_on` / `messages_between` / `message_context`), the verbose tiers move from
**always-injected** to **pulled on demand** — each tier collapses to a **compact dated index** with the body
fetched by a tool when a turn needs it. This phase is the **low-risk trims** of that line: the **day/week
digests** (currently multi-line paragraphs) become a **dated one-line index**, and the **style palette** is
compressed. (The riskier/structural tiers are their own phases: conversations **v0.35**, facts **v0.36**,
thoughts **v0.37**.) Target **~−18–22%** of the system prompt here.

**Hard rules (unchanged):** never drop a tier to **zero** — drop it to **its index** (a dated gist that keeps
ambient awareness + the exact `messages_on(date)` key); **rules and identity never move** — the canon,
boundaries, and identity-core facts stay always-injected; each change is behind a flag, **reversible**, and
**off → byte-identical** to today; per-user isolation + the `{reply, emotion, intensity}` contract are
untouched. **No paid calls** in tests (the model is mocked; day/week summaries are derived from kept session
summaries).

Depends on **v0.9** (the two-tier day/week summaries it gists), **v0.15** (caching — the steadier-prefix
win), **v0.17/v0.31** (the pull tools `messages_on` / `recall` the index falls back to).

## v0.34 — Lean memory: tool-pull, not push

### LUMI-136 — Day/week digests → a dated one-line index

**Description:**
The `## Останні дні` / `## Останні тижні` blocks are ~25% of the system prompt — each day/week is a
**multi-line prose paragraph** (the generation prompt asks for *"one coherent short paragraph"*, ~250 words).
Collapse each to a **dated one-line gist** so the tier becomes a compact index she can `messages_on(date)`
into for the verbatim detail. (`/core`.)

**What needs to be done:**
- Rewrite **`DAY_SUMMARY_SYSTEM`** and **`WEEK_SUMMARY_SYSTEM`** (`core/memory.py`) to produce a **single
  tight gist line** (≤ ~20 words — the one thing that mattered that day/week), not a paragraph. **Note:** the
  `MAX_DAY_ROWS` / `MAX_WEEK_ROWS` row cap is the **wrong lever** — it clamps *lines*, and these digests are a
  single paragraph (one line), so the length must come from the **prompt**, not the cap.
- Gate the new format behind **`LUMI_MEMORY_INDEX`** (off → the existing paragraph prompts, unchanged; on →
  the gist prompts) for reversibility.
- The prompt builder (`core/prompt.py`) already renders each as `- [date] {body}` and joins the body to one
  line — **verify** it becomes a clean dated index once the body is a gist (no builder change expected).
- *(Optional)* expose **`LUMI_DAY_DAYS`** / **`LUMI_WEEK_DAYS`** to trim how many days/weeks are injected
  (today hardcoded constants).

**Dependencies:** none new (over the shipped v0.9 day/week summaries).

**Expected result:**
With `LUMI_MEMORY_INDEX=on`, the day/week tier renders as a **dated one-line gist index** (the `## Останні
дні` / `## Останні тижні` blocks shrink ~−18% of the prompt) and a date she references is fetched verbatim via
`messages_on(date)`. Off → today's paragraph digests, byte-identical.

**Acceptance criteria:**
- [ ] `DAY_SUMMARY_SYSTEM` / `WEEK_SUMMARY_SYSTEM` produce a **one-line gist** per day/week (a unit test asserts a single ≤ ~20-word line).
- [ ] `LUMI_MEMORY_INDEX` (off by default) selects the format; **off → the paragraph prompts are byte-identical** to today.
- [ ] The injected block renders as a **dated index** (`- [date] gist`), per-user (a contract/isolation check: A's day-gist never appears for B).
- [ ] **Reconstruction:** with the index on, a planted reference to a past day makes the model call `messages_on(date)` (mock tool-script) and the answer is correct.
- [ ] Integration: a representative prompt dump's day/week tier shrinks **~−18%** on; **byte-identical** off. **No paid calls** (model mocked).

### LUMI-137 — Regenerate the stored day/week digests

**Description:**
The lazy `ensure_day_summaries` / `ensure_week_summaries` rebuild a day/week **only when its session count
changes**, so existing "finished" days keep their old **paragraph** digest — LUMI-136's new gist prompt is
**not retroactive**. Add a one-off rebuild so the current window picks up the index format. (`/core`, `/state`.)

**What needs to be done:**
- A **regenerate path** — a `regenerate_summaries(window)` helper (and/or a small CLI flag) that **clears**
  the stored `DaySummary` / `WeekSummary` rows in the recall window (or re-runs their generation) so the next
  prompt builds rebuilds them with the LUMI-136 gist prompt.
- **Safe + lossless:** the day/week digests derive from the kept **session summaries** (the source of truth is
  untouched); a failed regenerate degrades to the existing stored digest, never blocks a turn.
- Document the one-liner in the operator notes (stop the app / back up the store first, per the store-free
  discipline used by the recall reindex skills).

**Dependencies:** LUMI-136 (the gist prompts must exist, or a regenerate reproduces the old paragraphs).

**Expected result:**
After running the regenerate once (with `LUMI_MEMORY_INDEX=on`), **all** day/week digests in the window render
as the dated one-line index — not just days that happened to gain a session. Re-running is idempotent.

**Acceptance criteria:**
- [ ] A `regenerate_summaries` path clears/rebuilds the stored day/week digests for the window; rebuilt entries use the LUMI-136 gist format.
- [ ] **Lossless:** the underlying session summaries are unchanged; a regenerate of a window with no session summaries is a no-op (no crash).
- [ ] Per-user scoped (regenerating user A's digests never touches B's); idempotent (re-running yields the same result).
- [ ] Unit test: a stored paragraph digest, after regenerate, is the one-line gist; the session-summary source is intact. **No paid calls** (model mocked).

### LUMI-138 — Compress the style palette

**Description:**
The `# Стиль відповіді` block is ~5–6% of the system prompt — authored style guidance with a lot of redundant
phrasing. This is **static, not pullable**: a one-time authoring pass that keeps the load-bearing directives
and cuts the rest. (`/core` — the authored style source.)

**What needs to be done:**
- An authoring pass on the **style block source** (the `# Стиль відповіді` palette): keep the **voice
  anchors** + the **budget-of-expressiveness** rule + the load-bearing style directives; cut the redundant /
  repeated phrasing.
- Target roughly **halving** the block's length without losing the directives that shape her voice.
- Where reasonable, fold near-duplicate guidance into single lines; do **not** drop any boundary/voice
  invariant (those belong to the canon, untouched).

**Dependencies:** none.

**Expected result:**
The `# Стиль відповіді` block is ~half its size with the same load-bearing guidance; her voice/style is
unchanged in a before/after spot-check on a few representative replies.

**Acceptance criteria:**
- [ ] The style block source is compressed ~50% (measured on the rendered `# Стиль відповіді` block) with the voice anchors + expressiveness budget retained.
- [ ] No boundary/identity/voice invariant is dropped (those live in the canon, untouched).
- [ ] A before/after spot-check (a few mock-model turns) shows no style drift; the prompt-builder test for the style block still passes. **No paid calls.**

## Notes

- **Sequencing.** LUMI-136 + LUMI-138 in parallel; LUMI-137 after 136. The whole phase is low-risk
  (authoring + a small migration) — no new seam, no contract change.
- **The other lean-memory phases.** Conversations (gist tier) = **v0.35**; facts core + `recall(scope=facts)`
  = **v0.36**; thoughts cap + thought-RAG + `recall(scope=thoughts)` = **v0.37**. Together they reach the
  ~−55–65% target; this phase banks the cheapest ~−18–22% first.
- **Measurement.** Verify each change by diffing a fresh `.lumi/prompt-*.md` dump before/after, and confirm
  the gisted day/week tier reduces the prompt-cache **write** rate (the regrowth the v0.15 caching refinement
  chased), not just raw size.
