The **send-it half** of the image tool: a **`send_image`** tool on the **v0.19 bounded tool-loop** that
lets –õ—ñ–ª—ñ **choose** to send a picture from her per-user sandbox (one she generated in v0.23, or one you
dropped in) to **your Telegram** ‚Äî the explicit, in-character complement to the v0.23 auto-display
(`LUMI_IMAGE_SHOW=telegram`). It rides the **shipped v0.13 Telegram bridge** (the outbound daemon already
does `send_photo` for the emotion face).

The one architectural care is the **single-writer outbox**: the core **never** touches Telegram or
`outbox.jsonl`. `send_image` calls an **injected `telegram_sink`** (a callable Core param, default
`None`); the **TUI** ‚Äî already the sole outbox writer (`mirror_reply`) ‚Äî supplies it and appends a
**`photo` record**. So **no core ‚Üî bridge coupling and no second writer**. The outbound daemon sends a
record's `photo` via the v0.13 `send_photo` path, **always** (not probability-gated like the face) and
**on its own** (not N-batched).

**Hard rules** (IMAGE_TOOL.md ¬ßsafety): **sandboxed + per-user** (the shared `safe_path` guard +
image-type check); **off by default** (`LUMI_IMAGE` **and** the bridge connected ‚Üí else the tool returns
"Telegram not connected"); the recipient is **the owner** (the single Telegram user); `send_image`
returns a **string** notice/error and **never raises**. **No emotion-contract change** ‚Äî `set_state`
stays terminal. **No core ‚Üî Telegram coupling** (the sink is an injected callable; the core never imports
the bridge).

Depends on **v0.22** (the image-tool surface + the `_turn_tools` wiring + the sandbox), **v0.23** (the
generated images to send) and **v0.13** (the Telegram bridge ‚Äî the outbox + `send_photo`) ‚Äî all shipped.
**No real Telegram in tests** ‚Äî the sink and the bot are faked (no network, no paid calls).

## v0.24 ‚Äî Local image tool III: send to Telegram (`send_image`)

### LUMI-098 ‚Äî `send_image` tool in `core` + the injected `telegram_sink` seam

**Description:**
Add the **`send_image(path)`** tool to the reply turn's bounded tool-loop and the **injected
`telegram_sink`** Core seam it calls. Pure and bridge-free ‚Äî the core never touches Telegram or the
outbox; it calls a callable supplied from outside, fully unit-testable with a fake sink. (`/core`.)

**What needs to be done:**
- **`send_image(path)` tool:** resolve `path` against the **active user's** sandbox via the v0.19
  `safe_path` guard, confirm it's an **image** (extension/type check, like `view_image`), then call the
  **injected `telegram_sink(abs_path, caption)`** (the caption is the turn's reply text, caption-capped
  downstream). Returns a **string** notice (`sent <name> to Telegram`); any failure (non-image /
  traversal / missing file / **no sink**) ‚Üí an **error string** (`Telegram not connected` when the sink
  is `None`), **never raises** (graceful degradation).
- **The `telegram_sink` seam:** a `telegram_sink: Callable[[str, str], None] | None` Core param,
  **default `None`**. The core holds it and passes it to the tool; it **never imports the bridge** and
  **never writes the outbox** (the single-writer invariant). `None` ‚Üí the tool replies "Telegram not
  connected."
- **Register on the loop:** add `send_image` (via a `_sendimage_tool_args` / the existing
  `_turn_tools`) to the name-routing executor when `LUMI_IMAGE` is on, **alongside** the file / wiki /
  vision / generation tools. The v0.19 trace covers it. Off (`LUMI_IMAGE` off) ‚Üí no `send_image`
  offered.
- **Thread the param:** `telegram_sink` through `build_core` ‚Üí `Core` (default `None`; the TUI supplies
  the real one in LUMI-099).
- **No contract change:** `set_state` stays terminal; `send_image` is a string-returning tool; the
  validated `{reply, emotion, intensity}` is unchanged.

**Dependencies:** None (pure handler; reuses the v0.19 `safe_path` + the v0.22 image-type check; the sink
is an injected callable).

**Expected result:**
`send_image({"path": "art/cat.png"})` with a **fake sink** resolves the sandbox path, calls
`sink(abs_path, caption)`, and returns `sent cat.png to Telegram`; a non-image / traversal / missing path
returns an error string and **does not** call the sink; with the sink `None` it returns "Telegram not
connected"; with `LUMI_IMAGE` off the tool is absent.

**Acceptance criteria:**
- [ ] `send_image(path)` resolves against the **active user's** sandbox (`safe_path`) + an image-type check, then calls the **injected `telegram_sink(abs_path, caption)`**.
- [ ] Returns a success notice; non-image / traversal / missing / **no sink** ‚Üí an **error string**, **never raises** (sink not called on failure).
- [ ] `telegram_sink` Core param (default `None`); the core **never** imports the bridge or writes the outbox.
- [ ] Registered on `_turn_tools` behind `LUMI_IMAGE` (merged with file/wiki/vision/generation); off ‚Üí absent; the v0.19 trace covers it.
- [ ] Unit tests with a **fake sink**: send-ok, non-image, traversal, missing, no-sink, off. **No real Telegram.**

### LUMI-099 ‚Äî TUI supplies the sink (`photo` record) + outbound daemon sends the `photo` + docs

**Description:**
Wire the two ends that already exist: the **TUI** (the single `outbox.jsonl` writer) supplies the
`telegram_sink` when the bridge is on ‚Äî appending a **`photo` record** ‚Äî and the **outbound daemon**
learns to send that record's `photo` via the shipped `send_photo`. No second writer, no core ‚Üî bridge
coupling. (`/tui`, `/telegram`.)

**What needs to be done:**
- **TUI provides the sink:** when the Telegram bridge is on (`LUMI_BRIDGE` + an outbox path) and
  `LUMI_IMAGE` is on, build a `telegram_sink(abs_path, caption)` that **appends a `photo` record** to the
  outbox (`kind="lili"` + a `photo=<abs_path>` field + the caption as text) via the existing
  single-writer append. Pass it into `build_core` / `Core`. Bridge off ‚Üí sink stays `None` (the tool
  reports "not connected").
- **Outbound daemon sends the `photo`:** extend the v0.13 face-photo path in `telegram/outbound.py` ‚Äî a
  record carrying a **`photo`** field is sent via `send_photo(chat, FSInputFile(photo), caption=‚Ä¶)`
  **always** (not gated by `LUMI_TELEGRAM_PHOTO`, which stays the *face* probability) and **on its own**
  (a chosen image is **not** N-batched with other replies); reuse the `CAPTION_LIMIT` cap (over-long
  caption ‚Üí photo then the text chunked, as today). A missing/unreadable photo file degrades to sending
  the text, never wedges the loop.
- **Supersede the auto-push:** this explicit `send_image` path **supersedes** the `telegram` target of
  `LUMI_IMAGE_SHOW` (the v0.23 auto-display); note the change in the docs (auto-push of every generated
  PNG ‚Üí she *chooses* via `send_image`).
- **Docs:** `.env.example` (the `LUMI_IMAGE` + bridge requirement note) + `docs/IMAGE_SETUP.md` (a
  "send to Telegram" section: the tool, that it needs the bridge connected, off-by-default, owner-only) +
  a cross-note in `docs/TELEGRAM_SETUP.md` (the daemon now sends chosen `photo` records).

**Dependencies:** LUMI-098.

**Expected result:**
With `LUMI_IMAGE=on` **and** the Telegram bridge connected, "–Ω–∞–¥—ñ—à–ª–∏ –º–µ–Ω—ñ —Ü–µ–π –º–∞–ª—é–Ω–æ–∫" drives
`send_image` ‚Üí the TUI appends a `photo` record to the outbox ‚Üí the outbound daemon sends the picture to
the **owner's** Telegram as a **photo** with the reply as caption; with the bridge off, `send_image`
returns "Telegram not connected" and the turn completes.

**Acceptance criteria:**
- [ ] The TUI supplies a `telegram_sink` only when the bridge **and** `LUMI_IMAGE` are on; it appends a **`photo` record** to the outbox (it stays the **single writer**); bridge off ‚Üí sink `None`.
- [ ] The outbound daemon sends a record's **`photo`** via `send_photo` **always** (not `LUMI_TELEGRAM_PHOTO`-gated) and **on its own** (not N-batched); caption-cap reused; a bad photo file degrades to text.
- [ ] The explicit path **supersedes** `LUMI_IMAGE_SHOW=telegram`; documented.
- [ ] `.env.example` + `docs/IMAGE_SETUP.md` + `docs/TELEGRAM_SETUP.md` updated (tool, bridge requirement, off-by-default, owner-only).
- [ ] Integration test (mock model scripting `send_image`; **fake sink**): a full turn appends a `photo` record and validates `{reply, emotion, intensity}`. **No real Telegram.**

### LUMI-100 ‚Äî Fake-sink + daemon-photo + degrade + isolation + emotion-contract tests

**Description:**
Pin the whole `send_image` surface: the core tool against a **fake `telegram_sink`** and the outbound
daemon against a **mocked bot** ‚Äî send-ok, the degrade paths, per-user isolation, off-by-default, and the
emotion contract. No real Telegram, no network, no paid calls. (`/tests`.)

**What needs to be done:**
- **Sink-call test:** `send_image` of a sandbox image calls the **fake sink** with the **resolved
  absolute** sandbox path + the caption, and returns the success notice.
- **Degrade tests:** non-image / traversal / missing path ‚Üí an error string and the **sink is not
  called**; sink `None` ‚Üí "Telegram not connected"; the turn completes in every case.
- **Isolation test:** `send_image` only ever resolves the **active user's** sandbox ‚Äî user A can't send
  user B's file (the path resolves under A or is refused).
- **Daemon photo test:** the outbound daemon, given an outbox **`photo` record**, calls the **mocked
  bot's `send_photo`** with the path + caption (caption-capped), **always** (independent of
  `LUMI_TELEGRAM_PHOTO`) and **on its own** (not merged into a text batch); a `kind="user"` record is
  still skipped.
- **Off test:** `LUMI_IMAGE` off ‚Üí no `send_image` tool offered.
- **Emotion-channel contract test** still validates `{reply, emotion, intensity}` verbatim with
  `send_image` active.

**Dependencies:** LUMI-098, LUMI-099.

**Expected result:**
`pytest` proves `send_image` calls the sink with the right sandbox path, degrades on non-image / traversal
/ missing / no-sink, is per-user isolated, is absent when off, and that the daemon sends a `photo` record
as a photo via a mocked bot ‚Äî with the emotion contract intact and **zero real Telegram calls**.

**Acceptance criteria:**
- [ ] Send-ok calls the fake sink with the resolved sandbox path + caption; success notice returned.
- [ ] Non-image / traversal / missing / no-sink degrade to an error string (sink **not** called); the turn completes.
- [ ] `send_image` resolves only the active user's sandbox (A cannot send B's file).
- [ ] The outbound daemon sends a `photo` record via a **mocked** `send_photo` (path + capped caption), always + on its own; `kind="user"` skipped.
- [ ] Off (`LUMI_IMAGE`) ‚Üí no `send_image` tool; the `{reply, emotion, intensity}` contract passes with `send_image` active. **No real Telegram, no paid calls.**
