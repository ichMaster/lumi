# Version v0.42 — Issue Breakdown

**Phase:** v0.42 — Thought scheduler (proactive thoughts on a clock — an in-TUI module)

Лілі's directives fire on a **clock she can keep** — *every 10 min*, *idle 15 min*, *at 08:00*,
*between 07:00–09:00 every 20 min*, *Mondays only* — replacing the single in-TUI idle timer with a small
**in-TUI scheduler** (no separate process, no file bus). The TUI is already the **only brain**, so a pure
`due(now, last_fired, spec)` predicate per `core/schedule.toml` entry drives `run_directive` **directly**
(the `%`-router, not the reply path), with a `schedule.state` (last-fired per entry) for a **startup
catch-up**. A **fast tick service** runs **ephemeral code handlers** (`%update_state` — callbacks, *not*
model directives). The v0.4 nudge + the v0.12 `%think` idle timer **fold in** as `idle:` entries; the
in-app timers retire. Off (`LUMI_SCHEDULER`, default) → today's idle timer unchanged, byte-identical.
Depends on: v0.12 (`run_directive` + `resolve` + `tick_think`), v0.4 (clock + quiet hours + `should_nudge`);
enriched by v0.33 (the directives it schedules). Full design: [THOUGHT_SCHEDULER.md](../../features/THOUGHT_SCHEDULER.md).

### LUMI-166 — Trigger model: `due()` + `schedule.toml` parser + `schedule.state` + config

**Description:**
The pure, testable foundation — no TUI, no timers. A **schedule entry** binds one **trigger** to one
**directive (+ a raw seed)**; each trigger reduces to a pure `due(now, last_fired, spec) -> bool`
(clock-driven, **no sleeps**, unit-tested with a fixed clock, exactly like `should_nudge`). Five trigger
types + the authored `core/schedule.toml` parser + the small `schedule.state` (last-fired per entry).

**What needs to be done:**
- A `core/schedule.py` module: a `ScheduleEntry` dataclass `{directive, trigger, spec, topic?, enabled}`
  and a pure **`due(now, last_fired, spec)`** covering the five triggers:
  - **`every: <dur>`** — a wall-clock interval since `last_fired`;
  - **`idle: <dur>`** — idle since the last real input (the caller passes `last_input`; the migrated
    v0.4/v0.12 rule — reuse `should_nudge`/`proactive_due` semantics);
  - **`at: <HH:MM> [days]`** — a fixed daily/weekly minute, **once** at its minute (guarded by `last_fired`);
  - **`between: <HH:MM-HH:MM>, every: <dur>`** — a windowed periodic;
  - **`cron: <expr>`** — a raw 5-field cron (the power form; a tiny in-repo evaluator, no new dep).
- The **`core/schedule.toml`** parser (`tomllib`, already available): a list of `[[schedule]]` entries;
  the `{placeholder}` topic is kept **raw** (resolved at fire time by `run_directive`→`resolve`, not here);
  malformed entries skipped (never fatal); `enabled` defaults true.
- **`schedule.state`** — last-fired per entry (a small JSON keyed by a stable entry id): read/write helpers
  (a temp file in tests), used by `at:`/`between` idempotence and the LUMI-167 catch-up.
- **Config:** `LUMI_SCHEDULER` (off), `LUMI_SCHEDULE_PATH` (`core/schedule.toml`), `LUMI_SCHED_TICK_MS`
  (30000), `LUMI_SCHED_TICK_FAST_MS` (60000), `LUMI_SCHED_CATCHUP_H` (6), `LUMI_SCHED_DAY_CAP` (24) — wired
  into `Config` + `load_config`. An authored `core/schedule.toml` shipping the doc's example entries
  (idle `%think`, a daytime `%catchup`, a morning `%brief`, a nightly `%learn`, a `%prompt`), all `enabled=false`.

**Dependencies:** none — pure core; reuses v0.4 (`should_nudge`/quiet-hours semantics) + v0.12 conceptually.

**Expected result:**
`due()` returns the right boolean per trigger against a fixed clock; the parser turns `schedule.toml` into
entries with raw placeholder topics; `schedule.state` round-trips last-fired stamps; the config vars load
(unset → scheduler off).

**Acceptance criteria:**
- [ ] `due(now, last_fired, spec)` unit-tested per trigger type (every/idle/at/between/cron) with a **fixed clock, no sleeps**; `at:` fires once at its minute; `between+every` never double-fires on a tick boundary.
- [ ] The `schedule.toml` parser yields entries with **raw** `{placeholder}` topics; malformed entries skipped; `enabled` respected.
- [ ] `schedule.state` read/write round-trips last-fired stamps (temp file in tests).
- [ ] Config vars load with the documented defaults; unset → `LUMI_SCHEDULER=off`.
- [ ] **No paid calls, no real sleeps.**

### LUMI-167 — In-TUI scheduler loop: tick → `due()` → `run_directive`, catch-up, caps + operator guide

**Description:**
The headline — an **in-TUI module** (no daemon, no bus). A `set_interval(LUMI_SCHED_TICK_MS)` timer
evaluates `due()` for each enabled entry and runs due directives **directly via `Core.run_directive`** (the
`%`-router — silent → a `Thought`; graduated/outward → the outbox), then stamps `last_fired` (memory +
`schedule.state`). `idle:` entries read the TUI's **own in-memory** last-input (`_last_activity`) — **no
`activity.txt`, no `directive-queue.jsonl`**. On startup a **catch-up pass** fires wall-clock entries missed
while closed (within `LUMI_SCHED_CATCHUP_H`, most-recent only). Restraint: quiet-hours veto (except a
deliberate `at:` inside the window) + a per-directive per-day cap + the global `LUMI_SCHED_DAY_CAP`.

**What needs to be done:**
- A scheduler module in `tui/` (e.g. `tui/scheduler.py`) owned by `LumiApp`: load the schedule at startup
  (when `LUMI_SCHEDULER=on`), start the tick, evaluate `due()` per entry each tick, run due entries via
  `run_directive` (placeholders resolve at fire time, in-process), stamp `last_fired`.
- The **startup catch-up pass**: for wall-clock entries (`at`/`between`/`every`) whose last-fired is stale,
  fire the **most-recent** due occurrence within `LUMI_SCHED_CATCHUP_H`; skip older (no backlog flood).
- **Caps + quiet hours:** the global daily cap + a per-directive per-day cap (over `schedule.state` counts);
  the quiet-window veto (reuse `LUMI_THOUGHTS_QUIET_HOURS`), with an explicit `at:` allowed to pierce it.
- **Surfacing:** a silent fire records a `Thought`; a graduated/outward fire rides the existing outbox path
  (v0.13) — reuse, no new sink. The v0.33 status-line/act surfacing shows a scheduled act like any directive.
- **Operator guide:** a `docs/SCHEDULER_SETUP.md` (or a section) — how to author `schedule.toml`, the five
  triggers, quiet hours + caps, the catch-up window; ties to [THOUGHT_SCHEDULER.md](../../features/THOUGHT_SCHEDULER.md).

**Dependencies:** **LUMI-166** (the trigger model + parser + state + config); reuses v0.12 (`run_directive`)
+ v0.13 (the outbox) + v0.33 (act surfacing).

**Expected result:**
With `LUMI_SCHEDULER=on` and an authored `schedule.toml`, entries fire on their clock **in-process** (a
`%brief` fires as a directive, not literal chat); a restart after downtime fires the most-recent missed
wall-clock entries within the cap; quiet-hours + per-day caps suppress; **off → the in-app idle timer is
unchanged** (LUMI-169 hasn't retired it yet).

**Acceptance criteria:**
- [ ] A due entry runs through `run_directive` → records a `Thought` (integration, injected clock); a graduated fire reaches the outbox.
- [ ] The **startup catch-up** skips stale (older than `LUMI_SCHED_CATCHUP_H`) and fires the **most-recent** due; `last_fired` stamped so it doesn't re-fire.
- [ ] Quiet-hours veto holds (a deliberate `at:` inside the window still fires); per-directive + global day caps suppress silently (logged, never queued).
- [ ] `idle:` reads the TUI's in-memory last-input — **no `activity.txt` / `directive-queue.jsonl`** created.
- [ ] **Off → byte-identical** (no scheduler; the in-app timer still fires `%think`); per-user isolation + the `{reply, emotion, intensity}` contract untouched.
- [ ] **No real sleeps, no paid calls** (mock model; the timer glue covered by one integration test with an injected clock).

### LUMI-168 — Tick service: fast timer for ephemeral code handlers (`%update_state`)

**Description:**
A **separate, fast** in-TUI timer (`LUMI_SCHED_TICK_FAST_MS`) for **lightweight, ephemeral acts** —
**registered code handlers, not model directives** (the `%`-name is just a label). A tick handler is silent
bookkeeping: **no `Thought`, no model call**, fire-and-forget, **not** persisted, **a no-op if missed** (it
runs only while the TUI is alive). This is the home of the future **`%update_state`** (v1.1 needs / v1.3
inner-life: a split-invariant advance-to-`now`). This phase ships the **mechanism + the registration seam**
(v1.1 wires the real handler); a backlog collapses to one run, a missed tick is dropped.

**What needs to be done:**
- A tiny **handler registry** (name → zero-arg callback) + a `set_interval(LUMI_SCHED_TICK_FAST_MS)` fast
  timer in `LumiApp` that runs every registered handler each tick; distinct from the scheduler's model-
  directive path (never touches `run_directive`, never records a `Thought`).
- **Fire-and-forget semantics:** a handler exception is logged and swallowed (never breaks the UI); a
  backlog collapses to a single run (no catch-up, no queue); a missed tick is a no-op.
- A **demonstration/no-op handler** (or a test-only handler) proving the seam — the real `%update_state`
  lands with v1.1; document the seam so v1.1 just registers a callback.
- Reuses the `LUMI_SCHED_TICK_FAST_MS` config from LUMI-166; gated by `LUMI_SCHEDULER`.

**Dependencies:** **LUMI-166** (the fast-tick config); independent of LUMI-167 (a separate timer).

**Expected result:**
With the scheduler on, the fast timer runs registered code handlers while the TUI is alive — silently, no
`Thought`, no model call; a missed tick is harmless; a raising handler doesn't crash the app.

**Acceptance criteria:**
- [ ] A registered handler runs on the fast tick (injected clock / direct invocation — **no real sleeps**); it records **no `Thought`** and makes **no model call**.
- [ ] Fire-and-forget: a raising handler is swallowed (logged); a backlog collapses to one run; a missed tick is a no-op.
- [ ] The seam is documented for the v1.1 `%update_state` handler (a callback, not a model directive).
- [ ] **Off → the fast timer doesn't run**; no paid calls.

### LUMI-169 — Migrate idle triggers to `idle:` entries; retire `_maybe_think`/`_maybe_nudge`

**Description:**
Fold the two existing proactive triggers into the scheduler as `idle:` entries, then **retire** the in-app
timers — one clock at the end, not two. The v0.4 **idle nudge** (authored openers after silence) and the
v0.12 **`%think` idle trigger** become `idle:` schedule entries; `tui/app.py`'s
`set_interval(_maybe_think / _maybe_nudge)` is deleted once the queue path is proven. **Phased:** the
scheduler already shipped alongside the in-app timer (LUMI-167); this issue moves the idle rule in and
removes the old timers.

**What needs to be done:**
- Add the two `idle:` entries to the authored `core/schedule.toml` (a `%think` free-muse + the nudge
  opener), preserving today's `LUMI_THOUGHTS_INTERVAL_S` cadence + spoken ratio + per-session→per-day cap.
- Route the nudge opener through the scheduler (it graduates to a spoken turn via the existing outbox/
  self-turn path); ensure the `idle:` trigger reads `_last_activity` exactly as `proactive_due` did.
- **Retire** `_maybe_think` / `_maybe_nudge` + their `set_interval(30, …)` registrations (and the now-unused
  `_last_think_ts`/`_last_nudge_ts` bookkeeping if fully subsumed); keep `_last_activity` (the scheduler
  reads it).
- **Off-path guard:** with `LUMI_SCHEDULER=off`, the in-app idle behavior must still work — so the retirement
  is gated (scheduler-on → scheduler owns idle; scheduler-off → the legacy timer, until a later cleanup).

**Dependencies:** **LUMI-167** (the scheduler loop + `idle:` trigger it migrates onto).

**Expected result:**
With the scheduler on, idle `%think` + the nudge opener fire from `idle:` schedule entries (same cadence as
before), and the in-app `_maybe_think`/`_maybe_nudge` timers are gone; with the scheduler off, the legacy
idle behavior is preserved (byte-identical).

**Acceptance criteria:**
- [ ] Scheduler-on: an `idle:` entry fires `%think` after the idle interval (same cadence/ratio/cap as `tick_think` today); the nudge opener graduates to a spoken turn.
- [ ] `_maybe_think`/`_maybe_nudge` + their `set_interval` are removed on the scheduler-on path (no double-fire with the scheduler).
- [ ] Scheduler-**off** → the legacy idle behavior is unchanged (byte-identical); `_last_activity` still resets on real input.
- [ ] Per-user isolation + the emotion contract untouched; **no paid calls, no real sleeps**.

## Notes

- **Sizes:** LUMI-166 **M** (five triggers incl. a tiny cron eval + parser + state + config), LUMI-167 **L**
  (the in-TUI loop + catch-up + caps + outbox surfacing + operator guide — the bulk), LUMI-168 **S** (a
  second fast timer + a registry seam), LUMI-169 **M** (phased migration + careful retirement, both paths).
- **No core change** — the scheduler is a **TUI module**; the core never learns it exists (it only calls the
  shipped `run_directive`/`resolve`). The `{reply, emotion, intensity}` contract + per-user isolation are
  untouched; **off → byte-identical** (pinned per issue).
- **No bus, no daemon** — no `directive-queue.jsonl`, no `activity.txt`; the only persisted state is
  `schedule.state` (last-fired per entry), read once on startup for the catch-up.
- **Restraint is the invariant** — off by default, each entry opted in, quiet hours + per-day caps, catch-up
  cap; a scheduled outward act is **a gift, never a demand** (THOUGHT_SCHEDULER §Safety).
- **Determinism** — `due()` is pure (fixed clock, no sleeps); `schedule.state` is a temp file in tests; the
  timer loop is the only un-unit-tested glue (one integration test, injected clock). No paid calls.
- **Build order:** LUMI-166 → (LUMI-167 ‖ LUMI-168) → LUMI-169.
