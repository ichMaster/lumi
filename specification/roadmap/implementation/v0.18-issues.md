# v0.18 ‚Äî GitHub Issues

Issues for phase **v0.18 ‚Äî More models (model & provider switching)** (version **v0 ‚Äî TUI**), derived
from [ROADMAP.md](../../ROADMAP.md) (¬ßv0.18) and the
[ARCHITECTURE.md](../../ARCHITECTURE.md) `LLMClient` seam + the
[EMOTION.md](../../features/EMOTION.md) structured-output contract. IDs continue from v0.17
(LUMI-074 ‚Üí **LUMI-075‚Ä¶079**).

v0.1 wired **Claude Haiku** behind the thin **`LLMClient`** seam (`core/llm.py`: the `LLMClient`
Protocol with `reply` / `reply_structured`, the `AnthropicClient`, the `MockLLMClient`). v0.18 adds
**more backends behind that same seam**, selected purely in config ‚Äî other **Anthropic** tiers
(Opus / Sonnet / Haiku), an **OpenAI-compatible** adapter covering **OpenAI**, **DeepSeek** and any
**local** OpenAI-compatible server (Ollama / LM Studio), and **MiniMax**. The core does not change: it
depends only on `LLMClient`, never on an SDK.

**Structured output is per-provider** ‚Äî Anthropic tool output (today), OpenAI/DeepSeek JSON-schema
`response_format`, MiniMax JSON ‚Äî but every backend maps its raw result into the **same**
`{reply, emotion, intensity}` shape and feeds the **v0.3 validation gate** (unknown emotion ‚Üí `calm`,
clamp intensity). So the locked emotion contract is untouched; only a new producer sits behind it.

Pulled ahead right after RAG because it depends only on **v0.1** (the seam) and **v0.3** (the emotion
field / structured output) ‚Äî both shipped ‚Äî and it unlocks cheaper or **local** models from the start
(e.g. a local model for everyday chat). **No paid calls in CI:** every adapter is tested against a
**stubbed transport / mock SDK per provider**.

## v0.18 ‚Äî More models (model & provider switching)

### LUMI-075 ‚Äî `build_llm` factory: provider/model selection from config (+ Anthropic tiers)

**Description:**
Introduce a single `build_llm(cfg) -> LLMClient` factory that selects the backend from
`cfg.provider` + `cfg.model` + the matching key, and wire it into `build_core` (replacing the direct
`AnthropicClient(...)` construction). Prove the seam by making the **Anthropic tiers** (Opus / Sonnet /
Haiku) switchable through it ‚Äî they already work behind `AnthropicClient`, just parametrised by model
id. The adapters for other providers land in LUMI-076/077; this issue is the dispatch + config
foundation. (`/core`.)

**What needs to be done:**
- **Factory** (`core/llm.py`): add `build_llm(cfg) -> LLMClient` that maps `cfg.provider` ‚Üí
  the concrete client (`anthropic` ‚Üí `AnthropicClient`; other providers raise a clear
  `LLMError("unknown provider ‚Ä¶")` until LUMI-076/077 register them). Resolve the active provider's
  **key** (and `base_url` where relevant) from config; only the **active** provider needs its key.
- **Wire `build_core`** (`core/agent.py`): replace the inline `AnthropicClient(cfg.api_key, ‚Ä¶)` with
  `llm = build_llm(cfg)` (keep the `llm` injection param for tests / `MockLLMClient`).
- **Config** (`core/config.py`): `provider` already exists (default `anthropic`); add per-provider
  key + optional `base_url` fields (`openai_api_key` / `deepseek_api_key` / `minimax_api_key` /
  `llm_base_url`), each from `.env` (`OPENAI_API_KEY` / `DEEPSEEK_API_KEY` / `MINIMAX_API_KEY` /
  `LUMI_LLM_BASE_URL`), `repr=False` for secrets. Validate `provider` against a known set with a clear
  error. `DEFAULT_MODEL` stays the Anthropic default.
- **Anthropic tiers:** confirm `AnthropicClient` honours any `cfg.model` (Opus/Sonnet/Haiku) ‚Äî no new
  code beyond passing it through; document the tier ids.
- **Errors:** a missing key / unknown provider raises `LLMError` with an actionable message (which env
  var to set), surfaced at startup like the existing `ANTHROPIC_API_KEY` check.

**Dependencies:** None (builds on v0.1 `LLMClient` + the existing `AnthropicClient`).

**Expected result:**
Setting `LUMI_PROVIDER=anthropic` + `LUMI_MODEL=claude-sonnet-4-6` (or `claude-opus-4-8`) switches the
tier with no code change; an unknown provider or a missing key fails fast with a clear message.

**Acceptance criteria:**
- [ ] `build_llm(cfg)` returns the right `LLMClient` for `cfg.provider`; `build_core` uses it (the `llm` injection param still works for tests).
- [ ] Anthropic tiers (Opus/Sonnet/Haiku) are selectable by `cfg.model` through the factory ‚Äî no core change.
- [ ] Config carries `provider` + per-provider keys/`base_url` (secrets `repr=False`); only the active provider's key is required.
- [ ] Unknown provider / missing key ‚Üí `LLMError` with an actionable message (no traceback at startup).
- [ ] Unit tests: factory selection per `provider`; missing-key + unknown-provider errors. No paid calls.

### LUMI-076 ‚Äî OpenAI-compatible backend (OpenAI / DeepSeek / local server)

**Description:**
A single `OpenAICompatibleClient` implementing `LLMClient`, covering **OpenAI**, **DeepSeek**, and any
**local** OpenAI-compatible server (Ollama / LM Studio) ‚Äî they differ only by `base_url` + key +
model id. Structured output via JSON-schema `response_format`, mapped to `{reply, emotion, intensity}`
and fed through the v0.3 gate. (`/core`.)

**What needs to be done:**
- **Adapter** (`core/llm.py`): `OpenAICompatibleClient` with `reply` + `reply_structured`, talking the
  OpenAI Chat Completions shape (lazy import the `openai` SDK, an optional extra ‚Äî never imported by
  `core`'s import path). Configurable `base_url`, `api_key`, `model`.
- **Structured output:** request `{reply, emotion, intensity}` via `response_format` JSON-schema
  (`strict` where supported); parse the JSON content into the shared emotion-field shape. A model that
  ignores the schema ‚Üí degrade to a best-effort JSON/text parse, then the v0.3 gate (never raises).
- **Providers via config:** `openai` (default base_url) / `deepseek` (`https://api.deepseek.com`) /
  a local server (`LUMI_LLM_BASE_URL`), each registered in the LUMI-075 factory; key per provider.
- **Stats:** populate `ResponseStats` (model, latency, token usage where the API returns it; `None`
  fields are fine ‚Äî the usage report handles missing counts).
- **Tests:** a **stubbed transport / fake OpenAI client** returning canned (and deliberately malformed)
  JSON ‚Üí assert the emotion field is produced and validated; no network, no paid call.

**Dependencies:** LUMI-075 (the factory + config).

**Expected result:**
`LUMI_PROVIDER=deepseek` + `DEEPSEEK_API_KEY` (or `LUMI_PROVIDER=openai`, or a local
`LUMI_LLM_BASE_URL=http://localhost:11434/v1`) runs a full turn that yields a valid `EmotionState`.

**Acceptance criteria:**
- [ ] `OpenAICompatibleClient` satisfies `LLMClient` (`reply` + `reply_structured`); the `openai` SDK is lazily imported (optional extra), never on `core`'s import path.
- [ ] JSON-schema `response_format` produces `{reply, emotion, intensity}`; malformed output degrades to the v0.3 gate, never raises.
- [ ] OpenAI / DeepSeek / a local `base_url` all selectable via config (key per provider).
- [ ] `ResponseStats` populated (missing token counts tolerated).
- [ ] Unit tests against a stubbed transport (valid + malformed JSON). No paid calls.

### LUMI-078 ‚Äî Structured-output parity ‚Üí the v0.3 gate (+ thinking/effort/stats degradation)

**Description:**
Unify how all backends produce the emotion field, and make Anthropic-only request features degrade
cleanly on the others ‚Äî so a turn behaves the same regardless of provider. (`/core`.)

**What needs to be done:**
- **Shared mapping:** a single helper that takes a provider's raw structured result and returns the
  validated `{reply, emotion, intensity}` via the **existing v0.3 gate** (unknown emotion ‚Üí `calm`,
  clamp intensity 0..1). OpenAI/DeepSeek/MiniMax adapters all route through it (Anthropic keeps its
  tool-output path but lands on the **same** gate).
- **Feature degradation:** `thinking` / `effort` / 1h-cache / `task_budget` are Anthropic features ‚Äî
  on other providers they are **ignored, not errors** (log once at debug; the turn still completes).
  Extended-thinking `<think>` parsing falls back to ‚Äúno thinking shown‚Äù when the provider has none.
- **Stats parity:** every backend returns a `ResponseStats` (model id + latency always; token fields
  populated when the provider reports them, `None` otherwise ‚Äî the v0.17 usage report already tolerates
  missing counts).
- **Docs:** `.env.example` documents `LUMI_PROVIDER` + `LUMI_MODEL` + each provider's key/`base_url`,
  noting that thinking/effort are Anthropic-only.

**Dependencies:** LUMI-076, LUMI-077.

**Expected result:**
The same `Core.reply` path yields a valid `EmotionState` on Anthropic, OpenAI, DeepSeek, MiniMax, and a
local server; switching providers never changes the contract, only the producer.

**Acceptance criteria:**
- [ ] All backends route their raw output through one shared mapping ‚Üí the **v0.3 gate** (unknown emotion ‚Üí `calm`, clamp intensity).
- [ ] Anthropic-only features (thinking/effort/cache/task_budget) are ignored ‚Äî not errors ‚Äî on non-Anthropic providers; the turn completes.
- [ ] Every backend returns a `ResponseStats` (model + latency always; token fields optional).
- [ ] `.env.example` documents provider selection + per-provider keys/base_url + the Anthropic-only features.
- [ ] Unit tests: the shared mapping + feature-degradation on a non-Anthropic stub. No paid calls.
