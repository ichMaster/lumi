# v0.23 — GitHub Issues

Issues for phase **v0.23 — Local image tool II: generation (text → PNG)** (version **v0 — TUI**),
derived from [ROADMAP.md](../../ROADMAP.md) (§v0.23) and [IMAGE_TOOL.md](../../features/IMAGE_TOOL.md)
§v0.23. IDs continue from v0.22 (LUMI-094 → **LUMI-095…097**).

The **creates-artifacts half** of the image tool: a **`generate_image`** tool on the **v0.19 bounded
tool-loop** that calls an image model, saves a **new** PNG into her per-user sandbox (**create-only**,
like `create_file`), and **shows** it. The backend is a thin injected **`ImageGen`** seam — default the
**Gemini Nano Banana** caller (`gemini-2.5-flash-image`, the existing `GEMINI_API_KEY`, stdlib `urllib`,
already proven in the `/generate-faces` skill); tests inject a stub returning canned PNG bytes (**no
paid image calls in CI**). Display reuses shipped infra via `LUMI_IMAGE_SHOW`: always the **path**, the
**v0.7 viewer** (a PNG signal), and a **Telegram photo** (v0.13).

**Hard rules** (IMAGE_TOOL.md §safety): **non-destructive** (no overwrite/delete — create-only);
**sandboxed + per-user**; **no personal data** in the prompt sent to the external API (the v0.21 wiki
rule); provider **content-safety** filters; **off by default** (`LUMI_IMAGE`, generation needs
`GEMINI_API_KEY`); **paid + per-turn-capped** (`LUMI_IMAGE_MAX_GEN`). **No emotion-contract change** —
`set_state` stays terminal. **No SDK in `core`** (the generator is a seam). Vision (v0.22) is the
*read* half; this is the *write* half, on the **same image seam + tool-loop** — no new mechanism.

Depends on **v0.22** (the image-block seam + the `_turn_tools` image wiring + the sandbox) and **v0.19**
(the bounded tool-loop + `safe_path`) — both shipped. Model + image generation **mocked** in every test.

## v0.23 — Local image tool II: generation

### LUMI-095 — `core/imagegen.py`: the `ImageGen` seam + `generate_image` tool (create-only)

**Description:**
Add `core/imagegen.py` — a thin injected **`ImageGen`** seam (`generate(prompt) -> png_bytes`) and the
**`generate_image`** tool that calls it and saves a **new** PNG into the per-user sandbox. Pure and
model-free, fully unit-testable with a stub generator. The default backend lifts the **Gemini Nano
Banana** caller already proven in the `/generate-faces` skill. (`/core`.)

**What needs to be done:**
- **`ImageGen` seam:** a callable `generate(prompt: str, *, size: int) -> bytes` (the PNG bytes). The
  default implementation calls Gemini (`gemini-2.5-flash-image`, `GEMINI_API_KEY`, stdlib `urllib` —
  the `generateContent` endpoint + the `inlineData` extract from the faces skill), returning PNG bytes;
  a missing key / provider error / content refusal raises a clear error (caught by the tool). **No SDK**
  — a plain callable, injected in tests.
- **`generate_image(prompt[, filename])` tool:** call the `ImageGen`, then save the bytes as a **new**
  file under the sandbox via the v0.19 `safe_path` guard — **create-only** (error if the path exists; no
  overwrite, no delete), a sensible default filename if none given (e.g. a slug + `.png`). Returns a
  **string** notice (`created art/<name>.png (<n> bytes)`); any failure (refusal, HTTP error, oversize,
  existing path, traversal) → an **error string**, never raises (graceful degradation).
- **Tool schema:** the function-calling schema (Anthropic-safe name `generate_image`, inputs `prompt`
  (required) + optional `filename`), exported (`GENERATE_TOOLS` / `GENERATE_TOOL_NAMES`) for LUMI-096.
- **Config** (`core/config.py`): `image_provider` (`LUMI_IMAGE_PROVIDER`, default `gemini`),
  `image_model` (`LUMI_IMAGE_MODEL`, default `gemini-2.5-flash-image`), `image_size` (`LUMI_IMAGE_SIZE`),
  `image_max_gen` (`LUMI_IMAGE_MAX_GEN`, per-turn generation cap).

**Dependencies:** None (pure handler; reuses the v0.19 `safe_path`; no model involved).

**Expected result:**
`generate_image({"prompt": "кіт в окулярах"})` with a stub `ImageGen` writes a new PNG into the sandbox
and returns `created … .png (… bytes)`; a second call on the same filename returns an "already exists"
error; a generator error returns an error string and no file.

**Acceptance criteria:**
- [ ] `ImageGen` seam (`generate(prompt, size) -> bytes`), default = the Gemini caller; **no SDK**; injectable.
- [ ] `generate_image` saves a **new** PNG under the sandbox (create-only — refuses an existing path; no overwrite/delete) and returns a success notice + path.
- [ ] Any failure (refusal / HTTP error / oversize / existing / traversal) → an **error string**, never raises.
- [ ] `GENERATE_TOOLS` schema exported; config `LUMI_IMAGE_PROVIDER` / `_MODEL` / `_SIZE` / `_MAX_GEN` threaded.
- [ ] Unit tests with a **stub `ImageGen`** (canned PNG): write-new, refuse-existing, error-degrades, sandbox escape. **No paid calls.**

### LUMI-096 — Wire `generate_image` into `Core.reply` + cap + no-personal-data + display + docs

**Description:**
Offer `generate_image` in the reply turn behind `LUMI_IMAGE`, merged with the file/wiki/vision tools,
with a per-turn generation cap, the no-personal-data prompt rule, and the display wiring. Off by default
→ unchanged. (`/core`.)

**What needs to be done:**
- **Wire into `_turn_tools`:** add `generate_image` (via an `_imagegen_tool_args`) to the name-routing
  executor when `LUMI_IMAGE` is on; bound to the **active user's** sandbox; an **injected `image_gen`**
  Core param (default the real Gemini caller; tests pass a stub). The v0.19 trace covers it.
- **Per-turn cap:** a closure counter bounds generations by `LUMI_IMAGE_MAX_GEN` (paid calls) —
  independent of the file/wiki/vision caps; over-cap → a "limit reached" notice.
- **No personal/memory data:** the prompt passed to `ImageGen` is **only what the model supplies** — the
  core never augments it with relationship memory, facts, or secrets (same rule + test as the v0.21 wiki
  query).
- **Display** (`LUMI_IMAGE_SHOW`, e.g. `path,viewer,telegram`): **path** always (the tool result names
  the saved file); **viewer** — write the generated PNG's path to an image-display signal the v0.7
  viewer can show; **telegram** — the generated PNG rides out as a **photo** via the v0.13 outbox.
  Best-effort: a display failure never breaks the turn (the file is still saved + named).
- **Config threading + docs:** thread the LUMI-095 settings + `LUMI_IMAGE_SHOW` through `build_core` →
  `Core`; `.env.example` + `docs/IMAGE_SETUP.md` (add the generation section: `GEMINI_API_KEY`, the
  flags, create-only/no-personal-data/off-by-default, the paid + content-safety notes).
- **No contract change:** `set_state` stays terminal; the validated `{reply, emotion, intensity}` is
  unchanged.

**Dependencies:** LUMI-095.

**Expected result:**
With `LUMI_IMAGE=on` (+ `GEMINI_API_KEY`), "намалюй кота в окулярах" drives `generate_image` → a new PNG
in `.lumi/files/<user>/`, shown per `LUMI_IMAGE_SHOW`, and Лілі replies; the outgoing prompt carries no
personal data; with the flag off, no `generate_image` tool is offered.

**Acceptance criteria:**
- [ ] On (`LUMI_IMAGE`): the turn offers `generate_image` (merged with file/wiki/vision); a turn generates + saves a PNG and shows it per `LUMI_IMAGE_SHOW`.
- [ ] Off: `Core.reply` is unchanged — no `generate_image` offered.
- [ ] Per-turn `LUMI_IMAGE_MAX_GEN` bounds generations; the outgoing prompt has no memory/personal data.
- [ ] Display emits path (always) + viewer signal + Telegram photo per `LUMI_IMAGE_SHOW`; a display failure degrades, never breaks the turn.
- [ ] Config threaded `build_core` → `Core`; `.env.example` + `docs/IMAGE_SETUP.md` document the flags + rules.
- [ ] Integration test (mock model scripting `generate_image`; **stub `ImageGen`**): a full turn writes the PNG, signals display, and validates `{reply, emotion, intensity}`. **No paid calls.**

### LUMI-097 — Contract + create-only + no-personal-data + cap + isolation tests

**Description:**
Pin the whole generation surface: create-only (non-destructive), no personal data in the prompt, the
per-turn cap, per-user isolation, graceful degradation, off-by-default, and the emotion contract — all
against the mock model + a **stub `ImageGen`** (no paid image calls). (`/tests`.)

**What needs to be done:**
- **Create-only test:** `generate_image` writes a new PNG and **refuses an existing path** (no overwrite);
  no delete path exists.
- **No-personal-data test:** the prompt the `ImageGen` stub receives is **exactly** the model's prompt —
  no relationship memory / facts leak in.
- **Cap test:** more than `LUMI_IMAGE_MAX_GEN` generations in a turn → the cap notice.
- **Degradation test:** a generator refusal/error returns an error string and the turn completes; a
  traversal filename is refused before any write.
- **Isolation contract test:** a generated PNG lands only in the active user's sandbox (A's generation
  never in B).
- **Off / display tests:** `LUMI_IMAGE` off → no `generate_image`; with display on, the saved file's path
  is emitted (path / viewer signal / Telegram photo asserted via fakes).
- **Emotion-channel contract test** still validates `{reply, emotion, intensity}` verbatim with
  `generate_image` active.

**Dependencies:** LUMI-095, LUMI-096.

**Expected result:**
`pytest` proves generation is create-only, leaks no personal data, is capped + isolated, degrades on
error, is absent when off, and behaviourally correct, with the emotion contract intact — zero paid image
calls.

**Acceptance criteria:**
- [ ] Create-only (refuses existing, no overwrite/delete); the prompt carries no memory/personal data.
- [ ] Per-turn `LUMI_IMAGE_MAX_GEN` cap holds; a generator error degrades to an error string, never raises.
- [ ] A generated PNG lands only in the active user's sandbox (isolation); traversal filename refused.
- [ ] Off (`LUMI_IMAGE`) → no `generate_image` tool; display emits the path/signal/photo per `LUMI_IMAGE_SHOW`.
- [ ] The `{reply, emotion, intensity}` contract passes with `generate_image` active; suite uses a **stub `ImageGen`** — no paid calls.
