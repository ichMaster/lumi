# v0.30 ‚Äî GitHub Issues

Issues for phase **v0.30 ‚Äî Semantic recall III: chunking long messages** (version **v0 ‚Äî TUI**), derived
from [ROADMAP.md](../../ROADMAP.md) (¬ßv0.30) and [SEMANTIC_RECALL_CHUNKING.md](../../features/SEMANTIC_RECALL_CHUNKING.md).
IDs continue from v0.29 (LUMI-115 ‚Üí **LUMI-116‚Ä¶119**).

A **refinement of the recall line** (v0.16 index + v0.17 auto-RAG / context expansion), **not a new
capability**. v0.16 embeds each message as **one vector** ‚Äî for a long paste that vector is an *average*, so
a query about one part of it matches weakly. Chunking changes the **unit indexed**: a message above a length
threshold is split into ~`chunk_chars` passages (sentence/paragraph boundaries, small overlap), each
embedded as its **own** `VectorRecord`; a short message stays one chunk (v0.16 behaviour = the
`chunk_count == 1` case). Retrieval ranks by **chunk**, then expands at **two granularities**: **chunk-level**
‚Äî the matched chunk **¬± `chunk_w` adjacent chunks of the same message** (the relevant *passage*) ‚Äî inside the
**message-level** ¬±`rag_w` neighbour thread (v0.17), where the long parent renders as its *passage* and short
neighbours render in full. *"Search fine, show coarse"* ‚Äî now at two levels.

**Data model (additive):** `VectorRecord` gains **`parent_msg_id`** (the message a chunk came from) +
**`chunk_index`** (its 0-based ordinal within the message); `msg_id` becomes the **chunk's** content-addressed
id (a hash of `parent|chunk_index|text`), so re-chunking is idempotent. A one-chunk message has
`chunk_index == 0` and `parent_msg_id == the message id` (back-compatible). This is a **contract change** ‚Äî
ARCHITECTURE ¬ßSemantic recall + the memory-records contract test (`tests/contract/test_memory_records.py`)
update **in the same commit** as the field change.

**Hard rules** (SEMANTIC_RECALL_CHUNKING.md ¬ßContract & isolation): **per-user, unchanged** ‚Äî a chunk
inherits its message's `user_id`; search, the chunk window, **and** the message window run **only over the
requesting user's** data (the isolation invariant, extended to assert the **passage and its chunk neighbours**
are single-user); the `{reply, emotion, intensity}` field and the `Embedder` / `VectorStore` seams are
**untouched**; **graceful** ‚Äî a failed chunk-neighbour lookup degrades to the bare matched chunk, a failed
message-neighbour lookup to the passage alone, never blocks a turn; **off by default** (`LUMI_RAG_CHUNK`) ‚Üí
one vector per message, **identical to v0.16/0.17**.

Depends on **v0.16** (the index + the `Embedder`/`VectorStore` seams, `VectorRecord`, `vector_msg_id`,
`backfill_vectors`/`ensure_backfill`, the `embed_model@embed_max_chars` staleness tag) and **v0.17** (the
per-turn RAG `_recall_block` + the `_expand_hits` context expansion + `recall_moments`/`/recall`) ‚Äî both
shipped. The **embedder is mocked** in every test ‚Äî no paid calls.

## v0.30 ‚Äî Semantic recall III: chunking long messages

### LUMI-116 ‚Äî The chunker + the `LUMI_RAG_CHUNK*` config

**Description:**
Add a pure, model-free **chunker** in `core/` that splits a long message into ordered passages, plus the
chunking config (off by default). Given the text + params it returns the passages; everything downstream
(LUMI-117 indexing, LUMI-118 expansion) builds on it. A message **at or below** `chunk_threshold` returns
**one chunk** (the whole text) ‚Äî so the off-path and short-message behaviour is exactly v0.16. Fully
unit-testable on the function alone ‚Äî no embedder, no store. (`/core`.)

**What needs to be done:**
- A **`chunk_text(text, *, chunk_chars, overlap, threshold) -> list[str]`** (e.g. `core/chunking.py`):
  - text length `<= threshold` ‚Üí `[text]` (one chunk; the `chunk_count == 1` case).
  - otherwise split into ~`chunk_chars` passages on **sentence/paragraph boundaries** (prefer paragraph,
    then sentence; never mid-word), with `overlap` chars carried between adjacent chunks so a boundary
    sentence is reachable from either side.
  - deterministic, order-preserving, total coverage (no dropped text); empty/whitespace ‚Üí `[]` or `[text]`
    consistent with how `_index_messages` already skips empty messages.
- **Config** (`core/config.py`): `rag_chunk` (`LUMI_RAG_CHUNK`, **off**), `rag_chunk_chars`
  (`LUMI_RAG_CHUNK_CHARS`, e.g. 800), `rag_chunk_overlap` (`LUMI_RAG_CHUNK_OVERLAP`, e.g. 120),
  `rag_chunk_threshold` (`LUMI_RAG_CHUNK_THRESHOLD`, e.g. 1200), `rag_chunk_w` (`LUMI_RAG_CHUNK_W`, e.g. 1).
- No wiring yet (indexing/expansion are LUMI-117/118) ‚Äî this issue is the function + config + its units.

**Dependencies:** None (pure function).

**Expected result:**
`chunk_text("‚Ä¶1500 chars‚Ä¶", chunk_chars=800, overlap=120, threshold=1200)` returns several ordered
passages (~800 chars, ~120 overlap, split on boundaries, full coverage); `chunk_text("short", ‚Ä¶)` returns
`["short"]`. The `LUMI_RAG_CHUNK*` config loads with safe defaults (off).

**Acceptance criteria:**
- [ ] `chunk_text` returns one chunk at/below `threshold`, several above it (~`chunk_chars`, ~`overlap`), split on sentence/paragraph boundaries, never mid-word, full coverage, order preserved, deterministic.
- [ ] Degenerate input (empty / whitespace / a single huge unbroken token) is handled without raising.
- [ ] `LUMI_RAG_CHUNK` (off) + `LUMI_RAG_CHUNK_CHARS` / `_OVERLAP` / `_THRESHOLD` / `_W` thread through config with defaults.
- [ ] Unit tests cover threshold, size, overlap, boundary preference, short‚Üíone-chunk, degenerate input. **No embedder, no store.**

### LUMI-117 ‚Äî `VectorRecord` chunk fields + chunk-granular indexing + staleness tag + contract test

**Description:**
Make the **chunk** the unit indexed. `VectorRecord` gains `parent_msg_id` + `chunk_index` (additive);
`msg_id` becomes the chunk's content-addressed id; `_index_messages` / `backfill_vectors` emit **one record
per chunk** (via LUMI-116) when `LUMI_RAG_CHUNK` is on, one record per message when off; the vectors
staleness tag gains `@chunk_chars` so a chunking change re-embeds the history. The data-model contract is
updated **in the same commit**. (`/core`, `/state`, `/tests`.)

**What needs to be done:**
- **`VectorRecord`** (`core/repository.py`): add `parent_msg_id: str` + `chunk_index: int` (additive,
  defaulted for back-compat). A one-chunk message ‚Üí `chunk_index = 0`, `parent_msg_id == the message id`.
- **Chunk id:** extend `vector_msg_id` (or add a chunk-id helper) so a chunk's `msg_id` is a hash of
  `parent_msg_id | chunk_index | text` ‚Äî idempotent re-chunking. `make_vector_record` / `_vector_record`
  set `parent_msg_id` (the message id from `vector_msg_id(session, ts, role, full_text)`) + `chunk_index`.
- **Index-on-write + backfill:** `_index_messages` and `backfill_vectors` run each eligible message through
  `chunk_text` (when on), embed **each chunk** (`text[:embed_max_chars]` per chunk), and `add_vectors` one
  record per chunk; `has_vector` keys off the chunk `msg_id` so backfill stays idempotent at chunk
  granularity. Off ‚Üí exactly one record per message (today's path).
- **Staleness tag:** extend `embed_model` (`{embed_model}@{embed_max_chars}`) to append `@{chunk_chars}`
  when chunking is on, so `ensure_backfill` resets + re-indexes when chunking params change (the same way a
  model/cap change re-embeds).
- **Contract (same commit):** update `tests/contract/test_memory_records.py` (the `VectorRecord` shape) +
  ARCHITECTURE ¬ßSemantic recall to document `parent_msg_id` / `chunk_index` and the chunk-id rule.

**Dependencies:** LUMI-116.

**Expected result:**
With `LUMI_RAG_CHUNK=on`, a long message is stored as several `VectorRecord`s sharing one `parent_msg_id`
with `chunk_index = 0,1,2,‚Ä¶` and per-chunk `msg_id`s; a short message ‚Üí one record (`chunk_index = 0`,
`parent_msg_id == msg_id`). Re-indexing is idempotent; changing `chunk_chars` re-embeds the history. Off ‚Üí
one record per message, byte-identical to v0.16.

**Acceptance criteria:**
- [ ] `VectorRecord` has `parent_msg_id` + `chunk_index` (additive); the chunk `msg_id` is content-addressed over `parent|chunk_index|text` (idempotent).
- [ ] On: `_index_messages` + `backfill_vectors` emit **one record per chunk** (via `chunk_text`), each embedded from its own text; `has_vector` dedups at chunk granularity. Off: exactly one record per message (unchanged).
- [ ] The staleness tag includes `@chunk_chars` when on ‚Üí `ensure_backfill` re-embeds on a chunk-param change; a one-chunk message stays back-compatible (`chunk_index 0`, `parent_msg_id == msg_id`).
- [ ] `tests/contract/test_memory_records.py` + ARCHITECTURE ¬ßSemantic recall updated **in the same commit**; embedding failures still degrade gracefully (message stored, retried by backfill).
- [ ] Unit tests (mock embedder): one-record-per-chunk on write + backfill, idempotent re-index, staleness re-embed on chunk-param change, off‚Üíone-per-message.

### LUMI-118 ‚Äî Two-level expansion (chunk-window inside the message-window) + `/recall` reuse

**Description:**
Make the chunk the unit **retrieved** and the passage the unit **injected**. Search ranks by chunk
(unchanged search path, just over chunk vectors); each hit then expands at **two granularities**: the
**chunk-window** (¬± `chunk_w` adjacent chunks of the same `parent_msg_id`, in `chunk_index` order ‚Üí the
*passage*, anchor chunk marked `‚Üê (matched)`) **inside** the **message-window** (¬± `rag_w` neighbour
messages, v0.17), where the long parent renders as its passage and short neighbours render in full. Extends
`_expand_hits` / `_recall_block`; `recall_moments` (`/recall`) reuses it. (`/core`.)

**What needs to be done:**
- **Chunk-window:** for each hit, gather the same-`parent_msg_id` chunks within ¬± `chunk_w` of the hit's
  `chunk_index`, ordered, the matched chunk marked; merge overlapping chunk windows (no chunk twice).
  Reconstruct the passage text (account for `chunk_overlap` so the join isn't duplicated).
- **Inside the message-window:** in `_expand_hits`, resolve `parent_msg_id` (not the chunk id) to its
  session position via `_position_of`; render the ¬± `rag_w` neighbour **messages** with the parent message
  shown as its **passage** (the chunk-window) and the short neighbours **whole**.
- **Merge + dedup + budget:** merge overlapping message ranges (existing logic); dedup the whole snippet
  against the live window (`window_ids`, LUMI-071/072); keep the total under `rag_max_chars`; the long
  parent is **never** re-injected whole.
- **`/recall` reuse:** `recall_moments` builds the same passage-in-thread snippets (anchor + score), so
  explicit search and per-turn RAG read identically.
- **Off / degrade:** `LUMI_RAG_CHUNK` off ‚Üí today's per-message expansion verbatim; a failed chunk-neighbour
  lookup ‚Üí the bare matched chunk; a failed message-neighbour lookup ‚Üí the passage alone; never raises.

**Dependencies:** LUMI-117.

**Expected result:**
A query about one part of a long paste retrieves **that chunk**, and the recall block injects **that passage**
(matched chunk + adjacent chunks of the same message, anchor marked) inside its ¬± `rag_w` message thread ‚Äî
the whole long message is never re-injected. `/recall` shows the same passage-in-thread. Off ‚Üí identical to
v0.17.

**Acceptance criteria:**
- [ ] Search ranks per chunk; each hit expands to a passage (¬± `chunk_w` same-message chunks, anchor marked) **inside** the ¬± `rag_w` message thread (parent ‚Üí passage, neighbours ‚Üí whole).
- [ ] Overlapping chunk windows merge (no chunk twice); the snippet is deduped vs the live window; the block stays under `rag_max_chars`; the long parent is never injected whole.
- [ ] `recall_moments` (`/recall`) reuses the same assembly (anchor + score); off (`LUMI_RAG_CHUNK`) ‚Üí byte-identical to the v0.17 per-message expansion.
- [ ] Graceful: a failed chunk-neighbour lookup ‚Üí the bare matched chunk; a failed message-neighbour lookup ‚Üí the passage alone; never blocks a turn.
- [ ] Unit tests (mock embedder): chunk-window assembly (order, anchor, merge, overlap join), passage-in-thread render, budget/dedup, `/recall` reuse, off-path equivalence.

### LUMI-119 ‚Äî Isolation contract + behaviour/degradation tests

**Description:**
Pin the chunking surface against the mock embedder (no paid calls): the **isolation invariant** holds at
chunk granularity (the matched chunk, its **chunk neighbours**, and its **message neighbours** are all the
requesting user's), the **off-path is identical** to v0.16/0.17, and retrieval **degrades gracefully**.
Extends `tests/contract/test_semantic_recall_isolation.py` + `tests/contract/test_memory_records.py`.
(`/tests`.)

**What needs to be done:**
- **Isolation contract:** user A stores a long (chunked) message; user B's search never returns A's chunk,
  and B's expansion never pulls A's **chunk neighbours** or **message neighbours** (the passage and its
  neighbours are single-user) ‚Äî the A‚ÜîB test extended over chunks.
- **Off ‚Üí identical:** with `LUMI_RAG_CHUNK` off, indexing emits one record per message and the recall block
  is byte-identical to v0.17 (a snapshot/equivalence assertion).
- **Behaviour end-to-end:** a long message indexed as several chunks; a query about one part injects **that
  passage** in its thread (matched chunk marked), never the whole message; a short message still recalls
  whole.
- **Graceful degradation:** a missing chunk-neighbour / message-neighbour / position-map miss degrades (bare
  chunk ‚Üí passage-alone) and the turn completes; an embedder error ‚Üí no block, never raises.
- **Emotion contract:** `{reply, emotion, intensity}` validates with chunked recall active.

**Dependencies:** LUMI-116, LUMI-117, LUMI-118.

**Expected result:**
`pytest` proves chunk/passage/neighbour retrieval is single-user (A‚ÜîB), the off-path equals v0.16/0.17, a
long paste injects only its relevant passage in-thread, degradation paths complete, and the emotion contract
holds ‚Äî all on the mock embedder, no paid calls.

**Acceptance criteria:**
- [ ] Isolation: A's chunk, its chunk neighbours, and its message neighbours never surface in B's search/expansion (A‚ÜîB contract).
- [ ] Off (`LUMI_RAG_CHUNK`) ‚Üí one record per message + a recall block identical to v0.17 (equivalence test).
- [ ] A long message ‚Üí several chunks; a part-query injects that passage in its ¬± `rag_w` thread (anchor marked), never the whole message; short messages recall whole.
- [ ] Graceful degradation (missing chunk/message neighbour, position miss, embedder error) ‚Üí the turn completes, never raises; the `{reply, emotion, intensity}` contract validates.
- [ ] All via the **mock embedder** ‚Äî no paid calls.
