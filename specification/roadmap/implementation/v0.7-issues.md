# v0.7 — GitHub Issues

Issues for phase **v0.7 — Local emotion viewer (image face)** (version **v0 — TUI**), derived from the per-phase Tasks in [ROADMAP.md](../../ROADMAP.md) (§v0.7), the deep-dive [EMOTION_VIEWER.md](../../features/EMOTION_VIEWER.md), and the asset manifest in [EMOTION.md §7](../../features/EMOTION.md). IDs continue from v0.6 (LUMI-027 → **LUMI-028…030**).

v0.7 gives Лілі a **real image face**, locally, **without a server** — the simplest way to *see* her before the web. It's **another renderer of the locked v0.3 emotion channel** (alongside the v0.5 emoji): the core writes her current emotion to a tiny **local signal**, and a **separate desktop window** polls it and swaps `faces/<emotion>.png`, with a **`calm` fallback** so the window never breaks. No generation — just switching pre-made images by emotion. The `faces/` pack is the same one the web `ImageRenderer` reuses in v2.1; the art prompts already live in [viewer/faces/PROMPTS.md](../../../viewer/faces/PROMPTS.md). No contract change.

## v0.7 — Local emotion viewer

### LUMI-028 — Core writes the current-emotion signal each turn

**Description:**
Each turn the core writes Лілі's current emotion (one word, optionally with intensity) to a small **local signal file**, so a separate viewer process can read it **without touching the core**. The value is the validated `EmotionState` — no new inference. Config-pathed, **best-effort** (never blocks or fails a turn). (`/core` + `/state`.)

**What needs to be done:**
- After `Core.reply` validates the `EmotionState`, write the emotion to the signal file — e.g. `"joy 0.80"` (the emotion word + optional intensity; the emotion alone is enough). Atomic-ish small write.
- Config: `LUMI_FACE_SIGNAL` (path), default next to the store (e.g. `<store dir>/face.txt`).
- Best-effort: wrap in try/except — a write failure never affects the reply.
- (Optional) write `calm` at startup so the viewer has something before the first turn.

**Dependencies:** None (builds on the **locked** v0.3 emotion channel).

**Expected result:**
After each turn the signal file holds the current emotion (e.g. `joy 0.8`); the viewer (LUMI-029/030) reads it.

**Acceptance criteria:**
- [ ] Each turn writes the validated emotion (+ optional intensity) to `LUMI_FACE_SIGNAL` (default next to the store).
- [ ] Best-effort — a signal-write failure never blocks or raises in the turn.
- [ ] The written value matches the turn's validated `EmotionState`.
- [ ] Unit test: a turn writes the expected emotion to a temp signal file; a write error degrades silently.

### LUMI-029 — Face resolver + signal reader (emotion→image, calm fallback)

**Description:**
The **pure, testable** core of the viewer (no GUI): a resolver `face_for(emotion, intensity) → faces/<…>.png` that is **total over the 9-value enum** with a **`calm` fallback**, plus a **signal reader** that parses the one-word (+ optional intensity) signal. (`/viewer`.)

**What needs to be done:**
- `face_for(emotion, intensity, faces_dir, *, exists=…) -> Path`: build `faces/<emotion>.png`; when an **intensity variant** exists (`<emotion>_low.png` / `<emotion>_high.png`, by the v0.5 bands) and intensity is given, pick it; an **unknown emotion or a missing file → `faces/calm.png`** (EMOTION.md §7/§8). Total over the enum.
- `read_signal(path) -> (emotion, intensity|None)`: parse the first token as the emotion, an optional second as intensity; missing/empty/garbled → `calm`.
- A small change-detector helper (did the resolved path change since last poll?) so the window only redraws on change.
- The `exists` check is injectable so tests run against a fake faces dir.

**Dependencies:** LUMI-028 (the signal path + format).

**Expected result:**
Given a signal value, the viewer logic yields the right `faces/*.png` (or `calm.png`), deterministically and total over the enum.

**Acceptance criteria:**
- [ ] `face_for` resolves emotion (+intensity) → a `faces` path; **total over the 9 emotions**; unknown emotion / missing file → `calm`.
- [ ] An intensity variant is chosen when its file exists, else the base `<emotion>.png`.
- [ ] `read_signal` parses emotion (+ optional intensity); missing / garbled → `calm`.
- [ ] Unit tests (fake faces dir + fake signal): total-over-enum, fallback, variant selection, signal parsing, change detection.

### LUMI-030 — Desktop viewer window + faces pack + `./viewer` launcher

**Description:**
A **separate desktop window** (Python/Tkinter) that polls the signal (via LUMI-029) and shows `faces/<emotion>.png`, swapping when it changes — with the `calm` fallback so it never breaks. Plus a **`./viewer` launcher** and a **placeholder faces pack** so it runs before real art exists. (`/viewer`.)

**What needs to be done:**
- A viewer process: poll the signal every ~0.5–1 s (or a filesystem watch) → `face_for(read_signal(...))` → show the PNG (Tk `PhotoImage`); redraw only on change; `calm` fallback so the window never crashes.
- The `viewer/faces/` pack: real art is user-provided (prompts in `viewer/faces/PROMPTS.md`). Ship a **placeholder generator** (the 9 emotion PNGs — name/emoji on a colored card) so the window runs immediately; `calm.png` at minimum.
- A **`./viewer`** launcher (mirroring `./lumi`).
- (Optional) use intensity variants when present (via LUMI-029).

**Dependencies:** LUMI-029 (the resolver + reader).

**Expected result:**
Run `./lumi` and `./viewer` side by side: a local window shows Лілі's face and changes with her emotion as the conversation goes; an unknown/missing emotion falls back to `calm`.

**Acceptance criteria:**
- [ ] A local window shows `faces/<emotion>.png` for the current signal and updates as the conversation changes (manual/integration).
- [ ] Unknown/missing emotion or file → `calm`; the window never crashes.
- [ ] A placeholder pack + generator so it runs without real art; `./viewer` launches it.
- [ ] The GUI shell stays thin — the poll/resolve/fallback logic is covered by the LUMI-029 unit tests (no display needed in CI); the placeholder generator is tested (produces the files).
