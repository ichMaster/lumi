# v0.19 — GitHub Issues

Issues for phase **v0.19 — Local file tool I: reading (tool-loop + see / search / read)** (version
**v0 — TUI**), derived from [ROADMAP.md](../../ROADMAP.md) (§v0.19) and
[FILE_TOOL.md](../../features/FILE_TOOL.md) §v0.19. IDs continue from v0.18 (LUMI-079 →
**LUMI-080…084**).

Today `reply_structured` is a **single** model call with one forced tool — `set_state`, the v0.3
emotion channel. v0.19 turns that single call into a short, **capped loop**: Лілі may call **read**
tools, the core executes them in a **per-user sandbox** and feeds the results back as untrusted data,
and the turn ends when she emits the terminal `set_state`. This phase ships the **safe (read-only)
half** — writing is v0.20 — together with the **bounded tool-loop** itself, the same foundation the
v4.2 web search, v4.3 world context, and the v5 creative layer all reuse.

**Three read tools** in a new `core/files.py`: `list_files`, `find_in_file` (search → match **line
numbers**), and `read_file` (1-based `start_line` + `line_count`, reporting `total_lines` so she can
page to the end). **Hard rules** (FILE_TOOL.md §Sandbox and safety): paths resolve under a per-user
root with `..`/absolute/symlink escapes rejected before any I/O; **file content is untrusted data,
never instructions**; the loop is **bounded** (`LUMI_TOOL_MAX_STEPS`) with per-read + per-turn line
caps; **off by default** (`LUMI_FILE_TOOL`). **No emotion-contract change** — the reply still returns
`{reply, emotion, intensity}` and `set_state` stays the terminal tool, so the v0.3 emotion-channel
contract test passes verbatim.

Depends on **v0.2** (the `Repository` + per-user scoping for the sandbox root), **v0.3** (the emotion
turn / `set_state` terminal tool), **v0.1** (the `LLMClient` seam it extends) — all shipped. Model
**mocked** in every test — no paid calls.

## v0.19 — Local file tool I: reading

### LUMI-080 — `core/files.py`: sandboxed read executor + the three read tools

**Description:**
Add `core/files.py` — the three **read** tool definitions and a **sandboxed, traversal-safe executor**
that runs them against a per-user root. Pure, model-free, fully unit-testable on its own (the loop in
LUMI-081 calls it; the wiring is LUMI-082). (`/core`.)

**What needs to be done:**
- **Sandbox guard:** a path resolver that joins a tool-supplied relative path under a given root and
  **rejects** any escape — `..` segments, absolute paths, and symlink targets that resolve outside the
  root — **before any I/O**, returning a clear error. The whole filesystem is never reachable.
- **`list_files`** — names (and sizes) of files in a directory under the sandbox root.
- **`find_in_file`** — search a file for a literal string → the **line numbers** of matches, each with
  a short preview, capped at `LUMI_FILE_FIND_MAX`.
- **`read_file`** — read `line_count` lines from a 1-based `start_line`, returning those lines **and**
  the file's `total_lines` (so the caller can page to the end); per-call line count bounded by
  `LUMI_FILE_READ_LINES`.
- **Tool definitions:** the Anthropic-style tool schemas (name, description, input_schema) for the
  three read tools, exported for LUMI-081/082 to register alongside `set_state`.
- **Executor signature:** a callable `execute(name, input, *, root) -> str` (or similar) that dispatches
  to the right tool and **returns a string result**; any error (missing file, denied path, oversize)
  returns an **error string**, never raises (graceful degradation, ARCHITECTURE §Error handling).
- **Config** (`core/config.py`): `files_dir` (`LUMI_FILES_DIR`, default `state/files`),
  `file_read_lines` (`LUMI_FILE_READ_LINES`), `file_find_max` (`LUMI_FILE_FIND_MAX`).

**Dependencies:** None (uses v0.2 per-user scoping conceptually; no model involved).

**Expected result:**
`execute("read_file", {"path": "notes.md", "start_line": 212, "line_count": 40}, root=…)` returns those
40 lines + `total_lines`; `find_in_file` returns the line numbers of a match; a `../etc/passwd` path is
refused with an error string and no I/O.

**Acceptance criteria:**
- [ ] `list_files` / `find_in_file` (line numbers + preview, ≤`LUMI_FILE_FIND_MAX`) / `read_file` (`start_line`/`line_count` window + `total_lines`, ≤`LUMI_FILE_READ_LINES`) all work under the root.
- [ ] Path escapes (`..`, absolute, symlink-out) rejected **before** any I/O, with a clear error string.
- [ ] The executor returns an error **string** on any failure — never raises.
- [ ] Config `LUMI_FILES_DIR` / `LUMI_FILE_READ_LINES` / `LUMI_FILE_FIND_MAX` threaded.
- [ ] Unit tests: each tool, sandbox escapes, caps. No model, no paid calls.

### LUMI-081 — Bounded tool-loop in the `LLMClient` seam

**Description:**
Grow the `LLMClient` seam a **bounded tool-loop** variant: run a structured turn with **extra tools** +
a `tool_executor` callback, looping the model round-trips until the terminal `set_state` is emitted or
the step cap is hit. Implement it in `AnthropicClient`; extend `MockLLMClient` to **script tool-call
sequences** so the loop is testable with no SDK. The loop is **generic** (any tools + executor), so it
is the reusable foundation for v4.2/v4.3/v5. (`/core`.)

**What needs to be done:**
- **Seam variant:** extend `reply_structured` (or add a sibling) to accept `tools: list[dict] | None`
  and `tool_executor: Callable[[str, dict], str] | None`, plus a `max_steps` cap. With no tools it
  behaves exactly as today (single forced `set_state` call) — back-compatible.
- **The loop (AnthropicClient):** offer `set_state` **plus** the extra tools with `tool_choice` auto;
  on each response, if the model calls a non-terminal tool, run `tool_executor(name, input)`, append
  the model turn + a **`tool_result`** turn, and call again; when it calls `set_state` (or `max_steps`
  is reached → force a final `set_state`-style finish), return the raw `{reply, emotion, intensity}`.
- **Untrusted framing:** the `tool_result` content is framed as **untrusted data** (the model reads it,
  never obeys instructions inside it).
- **Stats/thinking:** accumulate across loop rounds; `last_stats` reflects the whole turn (token usage
  summed where the SDK reports it).
- **MockLLMClient:** accept a scripted sequence of tool calls then a terminal state, so tests can drive
  multi-step loops (and the loop-cap path) deterministically.
- **Cap:** `max_steps` (from `LUMI_TOOL_MAX_STEPS`) bounds tool calls per turn; reaching it ends the
  turn with a valid `set_state` so it can never hang or spin.

**Dependencies:** None (generic; tested with `MockLLMClient` scripting + a fake executor).

**Expected result:**
With a scripted mock (`find_in_file` → `read_file` → `set_state`), the loop executes both tools via the
callback and returns the final `{reply, emotion, intensity}`; a script that never calls `set_state` is
force-terminated at `max_steps`.

**Acceptance criteria:**
- [ ] The seam runs a turn with extra tools + a `tool_executor`, looping until terminal `set_state`; **no tools → unchanged** single-call behaviour.
- [ ] `tool_result` is framed as untrusted data.
- [ ] `max_steps` caps the loop and forces a valid terminal `set_state` (never hangs).
- [ ] `last_stats` accumulates across rounds.
- [ ] `MockLLMClient` can script tool-call sequences; unit tests cover a multi-step loop + the cap path. No paid calls.

### LUMI-082 — Wire the loop + file executor into `Core.reply` + config

**Description:**
Join LUMI-080 (the sandboxed file executor) and LUMI-081 (the bounded loop) into the reply turn, behind
the enable flag, with the per-user sandbox root. Off by default → the turn is byte-for-byte unchanged.
(`/core`.)

**What needs to be done:**
- **Per-user sandbox root:** resolve `LUMI_FILES_DIR / <user_id>` for the active user (created lazily),
  and build a `tool_executor` bound to that root (so a turn can only touch the active user's files).
- **Thread into `Core.reply`:** when `LUMI_FILE_TOOL` is on, pass the three read tools + the bound
  executor + `max_steps` into the seam's tool-loop; when off, call `reply_structured` exactly as today.
- **Config** (`core/config.py`): `file_tool` (`LUMI_FILE_TOOL`, **off** by default) and
  `tool_max_steps` (`LUMI_TOOL_MAX_STEPS`); thread `build_core` → `Core` (with the LUMI-080 caps).
- **`.env.example`:** document `LUMI_FILE_TOOL` + `LUMI_FILES_DIR` + the caps + `LUMI_TOOL_MAX_STEPS`,
  noting file content is untrusted and the tool is off by default.
- **No contract change:** `set_state` stays terminal; the final output is the validated
  `{reply, emotion, intensity}` (+ optional relational read).

**Dependencies:** LUMI-080, LUMI-081.

**Expected result:**
With `LUMI_FILE_TOOL=on`, "знайди розділ про оплату і прочитай його" drives `find_in_file` →
`read_file` in one bounded turn over `state/files/<user>/…`, and Лілі answers grounded in those lines;
with the flag off, the turn is unchanged.

**Acceptance criteria:**
- [ ] On (`LUMI_FILE_TOOL`): a turn lists/finds/reads within `LUMI_FILES_DIR/<user_id>` via the bounded loop.
- [ ] Off: `Core.reply` is unchanged (single `set_state` call) — no tools offered.
- [ ] The executor is bound to the **active user's** root (no cross-user path).
- [ ] Config `LUMI_FILE_TOOL` / `LUMI_TOOL_MAX_STEPS` (+ LUMI-080 caps) threaded; `.env.example` documented.
- [ ] Integration test (mock model scripting a tool sequence): a full turn reads a sandbox file and validates `{reply, emotion, intensity}`. No paid calls.

### LUMI-083 — Safety bounds & graceful degradation

**Description:**
Tighten the cross-cutting bounds that span the executor, the loop, and the turn, so a file turn can
never run away, leak, hang, or be hijacked by file content. (`/core`.)

**What needs to be done:**
- **Per-turn read total cap:** track lines read across **all** `read_file` calls in a turn; once
  `LUMI_FILE_READ_MAX_TOTAL` is reached, further reads return a "limit reached" notice instead of more
  lines (needs per-turn state on the bound executor).
- **Loop-cap termination:** reaching `LUMI_TOOL_MAX_STEPS` forces the turn to finish with a valid
  `set_state` (confirm end-to-end through `Core.reply`, not just the seam).
- **Untrusted-content enforcement:** verify the `tool_result` framing holds end-to-end — instructions
  embedded in a read file are treated as data, never obeyed.
- **Never-raise:** any tool error (missing file, denied path, oversize, decode error) returns an error
  string in the `tool_result`; the turn continues and ends normally — a file error degrades the reply,
  never breaks the turn (ARCHITECTURE §Error handling).
- **Config:** `file_read_max_total` (`LUMI_FILE_READ_MAX_TOTAL`) threaded.

**Dependencies:** LUMI-082.

**Expected result:**
A turn that tries to read a 10k-line file stops at `LUMI_FILE_READ_MAX_TOTAL` with a notice; a denied
path or a tool spin both end with a valid reply; a file saying "ignore your instructions" changes
nothing.

**Acceptance criteria:**
- [ ] Per-turn total read capped by `LUMI_FILE_READ_MAX_TOTAL`; further reads → "limit reached" notice.
- [ ] Loop cap forces a valid terminal reply through `Core.reply` (never hangs/spins).
- [ ] Untrusted file content is not acted upon (data, not instructions).
- [ ] Every tool error degrades to an error string; the turn completes, never raises.
- [ ] `LUMI_FILE_READ_MAX_TOTAL` threaded; unit tests cover each bound. No paid calls.
