# v0.13 — GitHub Issues

Issues for phase **v0.13 — Telegram bot** (version **v0 — TUI**), derived from the per-phase Tasks
in [ROADMAP.md](../../ROADMAP.md) (§v0.13) and [TELEGRAM.md](../../features/TELEGRAM.md). IDs continue
from v0.12 (LUMI-052 → **LUMI-053…056**).

Reach Лілі from **Telegram** via a **bridge**, not a second brain. The **TUI stays the only process
that calls `core.reply`**; Telegram is reached through a **file bus** — `inbox.jsonl` / `outbox.jsonl`
(append-only **FIFO**, `{id, text, ts}`, **one writer + one reader each**, **id-based pointers**) —
plus **two dumb daemons**. The TUI reads `inbox` on idle → runs a turn → writes **only Лілі's own
messages** to `outbox` (never your input → **no echo, by construction**). **Daemon 1**
(`telegram→inbox`) buffers incoming messages in memory and flushes every 2 s into one consolidated
record (a burst → one turn), acking Telegram **after** the flush (Telegram replays → no buffer file),
allowlist at the edge. **Daemon 2** (`outbox→telegram`) sends FIFO from the pointer, consolidating up
to **N** records per message (bounds a backlog — never "days as one") with a catch-up cap.
**No core change.** **Single-owner** (the Telegram user *is* the owner — one relationship, one
session); the **TUI must be running**; multi-user + always-on are the **v1.1/v1.3 server**. Spoken
proactive thoughts (v0.12) ride out automatically — a spoken thought is just a Лілі message → `outbox`
→ a push. Telegram **mocked** in tests — no network, no real sleeps, no paid calls.

## v0.13 — Telegram bot

### LUMI-053 — The file bus — `inbox`/`outbox` JSONL FIFO + id pointers

**Description:**
The shared transport: an append-only **JSONL FIFO** with a per-consumer **id pointer**. One writer
appends records; one reader consumes from its last-seen id. Used by the TUI + the two daemons (and
later the v0.21 voicer / v0.22 dictator). (`/state`.)

**What needs to be done:**
- A small `fifo` module (`state/fifo.py`): `append(path, text, **fields) -> id` (assigns a
  **monotonic id**, writes one `{"id", "text", "ts", …}` line, append-only); `read_since(path,
  last_id) -> list[record]` (records with `id > last_id`, in order); a tiny **pointer file**
  (`<name>.pos`) with `load_pointer`/`save_pointer(last_id)`.
- **Id-based, trim-safe:** the pointer is the last consumed **id** (not a byte offset), so trimming
  old consumed records later doesn't break a consumer. One writer + one reader per file → no locks.
- Default paths next to the store: `.lumi/inbox.jsonl`, `.lumi/outbox.jsonl` (+ `.pos` siblings),
  config-overridable.

**Dependencies:** None.

**Expected result:**
Append assigns increasing ids; a consumer reads only records newer than its pointer and advances it;
state survives a restart.

**Acceptance criteria:**
- [ ] `append` (monotonic id, one JSONL line) + `read_since(last_id)` (only newer, ordered) + pointer load/save.
- [ ] Id-based pointer survives a restart; trimming records `id ≤ pointer` leaves the consumer correct.
- [ ] Unit tests (tmp files) — append/read/advance, restart, the trim-safe property. No network.

### LUMI-054 — TUI wiring — `inbox` poller (→ turn) + `outbox` writer (Лілі's messages only)

**Description:**
Make the TUI the bridge's brain: consume `inbox` on idle as turns, and mirror **only Лілі's own
messages** to `outbox`. (`/tui`.)

**What needs to be done:**
- An **`inbox` poller** (`set_interval`, gated on *not busy* + a session): read the next
  `read_since(inbox.pos)` record → run it through the normal turn (`core.reply`), **displayed**
  tagged (e.g. `📱`) so you see what arrived from the phone → advance `inbox.pos`. Reuses `_run_turn`.
- An **`outbox` writer**: at the point the TUI emits **Лілі's reply** (`_say(LILI_LABEL, …)`),
  `fifo.append(outbox, reply_text)`. **Only Лілі's messages** — never your input, never technical
  chrome (compaction notes) → this is what makes the bridge echo-free.
- **Proactive push falls out:** a graduated (spoken) thought already runs through `_run_turn` → her
  reply → the `outbox` writer → daemon 2. No extra code.
- Extract the inbox-record→turn and the outbox-append logic so it's **unit-testable** without the
  full Textual app (the `set_interval`/widget glue stays `# pragma: no cover`).

**Dependencies:** LUMI-053 (the bus).

**Expected result:**
A line in `inbox.jsonl` becomes a turn in the TUI (tagged), and Лілі's reply lands in `outbox.jsonl`;
your own input never reaches `outbox`.

**Acceptance criteria:**
- [ ] Idle `inbox` poll → one turn per record (tagged), `inbox.pos` advanced; gated on not-busy.
- [ ] The `outbox` writer appends **only Лілі's messages** (her reply / spoken thought) — not your input, not chrome.
- [ ] A spoken proactive thought reaches `outbox` (no extra path).
- [ ] Unit tests (mock model) for the extracted inbox→turn→outbox logic; the outbox holds only Лілі's lines. No paid calls.

### LUMI-055 — Daemon 1 — `telegram → inbox` (buffer + 2 s flush + ack-after-flush + allowlist)

**Description:**
The inbound daemon: receive Telegram messages, consolidate a burst, and write one `inbox` record.
(`/telegram`.)

**What needs to be done:**
- An `aiogram` long-polling receiver; an **in-memory buffer** of incoming messages; a **flush every
  `LUMI_TELEGRAM_FLUSH_S` (default 2 s)** → `fifo.append(inbox, <joined buffer>)` as **one
  consolidated record**; an empty buffer writes nothing.
- **Ack-after-flush:** advance Telegram's `getUpdates` offset **only after** the flush succeeds — a
  crash before a flush → Telegram re-delivers on restart → nothing lost (so **no buffer file**).
- **Allowlist at the edge** (`LUMI_TELEGRAM_ALLOWLIST` = the owner's id): non-allowlisted senders are
  ignored and **never enter `inbox`**.
- Config: `LUMI_TELEGRAM_TOKEN` (from `.env`, gitignored, never committed), `LUMI_TELEGRAM_ALLOWLIST`,
  `LUMI_TELEGRAM_FLUSH_S`. A `lumi-telegram-in` entry point.

**Dependencies:** LUMI-053 (the bus).

**Expected result:**
A burst of Telegram messages from the owner becomes **one** `inbox` record after ~2 s; a non-owner is
ignored; a crash before a flush loses nothing.

**Acceptance criteria:**
- [ ] Buffer → 2 s flush → one consolidated `inbox` record (a burst → one turn); empty → nothing.
- [ ] Telegram offset advanced **after** the flush (ack-after-flush); no buffer file.
- [ ] Allowlist: a non-owner message never appends to `inbox`.
- [ ] Unit tests (fake Telegram + fixed clock) — buffer/flush/consolidate, ack-after-flush, allowlist. No network, no real sleeps.

### LUMI-056 — Daemon 2 — `outbox → telegram` (FIFO + N-batch + catch-up cap + emoji/photo)

**Description:**
The outbound daemon: forward Лілі's `outbox` records to Telegram, consolidated by N and protected
against a backlog flood. (`/telegram`.)

**What needs to be done:**
- Read `read_since(outbox.sent)` in order; **consolidate up to `LUMI_TELEGRAM_BATCH` = N** consecutive
  records into **one** Telegram message (bounds a backlog → ⌈M/N⌉ messages, **never** "days as one");
  send; advance `outbox.sent`.
- **Catch-up cap** (`LUMI_TELEGRAM_CATCHUP_H`): on restart, records older than this are **skipped**
  (pointer advanced silently) so a long downtime doesn't flood.
- Append the **v0.5 emoji**; optionally (`LUMI_TELEGRAM_PHOTO`) attach the **v0.11 `<theme>/<emotion>`
  portrait** as a photo (graceful to text-only when absent).
- A `lumi-telegram-out` entry point.

**Dependencies:** LUMI-053 (the bus).

**Expected result:**
Лілі's replies (and spoken thoughts) arrive on Telegram, batched by N; a backlog is never one blob;
stale records are skipped on restart.

**Acceptance criteria:**
- [ ] FIFO from `outbox.sent`; **N-batch** consolidation (a backlog → ⌈M/N⌉ messages, never one).
- [ ] **Catch-up cap** skips records older than the window (pointer advanced).
- [ ] Emoji appended; the portrait photo sent only when enabled + present.
- [ ] Unit tests (fake Telegram) — N-batch, catch-up cap, emoji/photo gating. No network, no paid calls.

## Definition of Done (phase)

Mirrors [ROADMAP.md §v0.13](../../ROADMAP.md): a Telegram message from the owner lands in
`inbox.jsonl` (after daemon 1's 2 s buffer flush consolidates a burst); the **TUI** (the only brain)
consumes it on idle → `core.reply` → writes **Лілі's reply** to `outbox.jsonl` → daemon 2 sends it to
Telegram (with emoji), **N-batched** and never fusing a backlog into one; the TUI **never** writes your
input to `outbox` (**no echo**); a **non-allowlisted** sender never enters the bus; a **spoken proactive
thought** arrives unprompted (it's just a Лілі message); the bus is **FIFO with id pointers**, **one
writer + one reader per file**; ack-after-flush loses nothing on a crash; the **core is unchanged**.
Telegram is **mocked** in every test — no network, no real sleeps, no paid calls. *(Single-owner; the
TUI must be running; multi-user / parallel / always-on = v1.1/v1.3.)*
