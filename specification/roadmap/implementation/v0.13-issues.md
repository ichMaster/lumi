## v0.13 вЂ” Telegram bot

### LUMI-053 вЂ” Bot skeleton + config (token + allowlist) + long-polling loop

**Description:**
A standalone `/telegram` bot process: the `aiogram` app, the config (bot token + allowlist), and the
**long-polling** run loop. No core logic вЂ” just the scaffolding the adapter (LUMI-054) plugs into.
(`/telegram`.)

**What needs to be done:**
- A `/telegram` package: `telegram/bot.py` (build the `aiogram` `Bot` + `Dispatcher`, register
  handlers, `start_polling`) + `telegram/__main__.py` + a `lumi-telegram` console entry point.
- Config (`core/config.py` or a small telegram config): `LUMI_TELEGRAM_TOKEN` (from `.env`,
  gitignored, never committed) and `LUMI_TELEGRAM_ALLOWLIST` (a comma-separated list of permitted
  Telegram ids в†’ parsed to a set).
- The **long-polling** entry point (`dp.start_polling(bot)`); a clean shutdown. The bot answers only
  while the process runs (document this вЂ” laptop on; the 24/7 server version is v1.1).
- **One shared `Repository`** is constructed here and handed to the per-user Cores (LUMI-054), so
  concurrent users don't clobber the JSON store.

**Dependencies:** None (builds on the existing `core` + `build_core`).

**Expected result:**
`lumi-telegram` starts a bot that connects to Telegram and long-polls; the token + allowlist load
from `.env`.

**Acceptance criteria:**
- [ ] `/telegram` package + `lumi-telegram` entry point; `aiogram` Bot/Dispatcher + `start_polling`.
- [ ] `LUMI_TELEGRAM_TOKEN` + `LUMI_TELEGRAM_ALLOWLIST` config (token never committed; allowlist parsed to a set).
- [ ] A single shared `Repository` is created for the per-user Cores.
- [ ] Unit test the **config parse** (token present; allowlist string в†’ id set). The polling loop is `# pragma: no cover`. No network.

### LUMI-054 вЂ” Message adapter в†’ `core.reply` + allowlist gate + per-user `user_id`/session (isolation)

**Description:**
The testable heart: an incoming message в†’ map the sender to a Lumi `user_id` в†’ (allowlist gate) в†’
`core.reply(text, session)` в†’ send her reply. Each Telegram user gets their **own** `Core` + session
over the **shared** repository, so the v0.2 isolation invariant holds. (`/telegram`, thin `/core`
use.)

**What needs to be done:**
- An `on_message` handler: `user_id = f"tg:{msg.from_user.id}"`; **allowlist gate** вЂ” a non-allowlisted
  id gets a polite refusal and **never calls the core**.
- A lazy **per-user registry** `user_id в†’ (Core, Session)` вЂ” `build_core(user_id=вЂ¦, repository=shared)`
  + `start_session()`, cached; `core.reply(text, session)` run **off-thread** (`asyncio.to_thread`).
- Send `state.reply` back; surface a model error as a readable line (never crash the loop), mirroring
  the TUI's resilience.
- A typed `%directive` / `/command` from a Telegram user routes the same as the TUI (reuse
  `run_directive` for `%вЂ¦`; commands are LUMI-056) вЂ” a plain message is a normal turn.

**Dependencies:** LUMI-053 (the bot + shared repository).

**Expected result:**
A Telegram message gets Р›С–Р»С–'s reply through the **same `core.reply`**; a non-allowlisted sender is
refused; user A's reply never carries user B's memory.

**Acceptance criteria:**
- [ ] Update в†’ `user_id` в†’ `core.reply` в†’ send reply (off-thread); a model error degrades to a readable line.
- [ ] **Allowlist:** a blocked id is refused and **never** constructs/calls a Core.
- [ ] **Isolation:** per-user `(Core, Session)` over a **shared** repository; A's reply never reflects B's memory/closeness (asserted with two users).
- [ ] Unit tests with a **fake bot + the mock model** (adapter, allowlist, Aв‰ B isolation). No network, no paid calls.

### LUMI-055 вЂ” Emotion emoji + face portrait as a photo

**Description:**
Render the emotion channel on Telegram: append the **v0.5 emoji** to her reply, and (optionally) send
the **v0.11 `<theme>/<emotion>` face portrait** as a photo. (`/telegram`.)

**What needs to be done:**
- Append the v0.5 emoji to the reply text (reuse `EmojiRenderer` + the `EmotionState`).
- Resolve the portrait for the reply's `(theme, emotion)` (reuse the viewer's resolver / the
  `faces/<theme>/<emotion>` path) and **send it as a Telegram photo** alongside the reply вЂ” **off when
  no faces exist or the feature is disabled** (config `LUMI_TELEGRAM_PHOTO`, default off).
- Graceful: a missing portrait в†’ text-only reply (never an error).

**Dependencies:** LUMI-054.

**Expected result:**
Her reply carries the emotion emoji; with photos enabled and a portrait present, the matching face is
attached.

**Acceptance criteria:**
- [ ] The v0.5 emoji is appended to the reply.
- [ ] With photos on + a portrait present, the `<theme>/<emotion>` image is sent; absent/off в†’ text-only.
- [ ] Unit tests (fake bot) вЂ” emoji appended; the photo is sent only when enabled + present. No network.

### LUMI-057 вЂ” Proactive push вЂ” the v0.12 think timer в†’ spoken thoughts via `send_message`

**Description:**
Run the v0.12 proactive-think timer inside the bot so a thought that **graduates to spoken** is
**pushed** to the user вЂ” she speaks first, as a notification (the payoff a bot enables and the TUI
can't). (`/telegram`.)

**What needs to be done:**
- A periodic task (per active user / chat): call `core.tick_think(session, last_activity, now, вЂ¦)`
  on the cadence; reuse the v0.12 gating (`INTERVAL_S` / `CAP` / `SPOKEN_RATIO` / quiet hours).
- On a **spoken** thought в†’ `bot.send_message(chat_id, вЂ¦)` (with the emoji / optional photo);
  **silent** thoughts record only (no push). A free-muse (B) / a `%`-seed from `nudges.md` (A) вЂ” the
  same A+B the TUI uses.
- Track `last_activity` per chat (an incoming message resets it), so push respects real idleness.

**Dependencies:** LUMI-054.

**Expected result:**
After you go quiet, a graduated proactive thought arrives as a Telegram message, unprompted; silent
thoughts stay in the diary.

**Acceptance criteria:**
- [ ] A spoken `tick_think` result в†’ `send_message` to that user's chat; a silent one pushes nothing.
- [ ] Reuses the v0.12 cadence (interval / cap / quiet hours / ratio), per chat.
- [ ] Unit tests (fake bot + mock model + fixed clock) вЂ” a spoken thought pushes, a silent one doesn't; the cap/quiet hours hold. No network, no real sleeps, no paid calls.
