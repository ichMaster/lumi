# v0.5 — GitHub Issues

Issues for phase **v0.5 — Emoji rendering** (version **v0 — TUI**), derived from the per-phase Tasks list in [ROADMAP.md](../../ROADMAP.md) (§v0.5) and the renderer ladder + the emoji/intensity table in [EMOTION.md](../../features/EMOTION.md) (§4/§5/§6). This file is scoped to a single phase; IDs continue from v0.4 (LUMI-022 → **LUMI-023…024**).

v0.5 is the **last step that makes v0 a complete TUI**: Лілі's emotion becomes **visible right in the terminal** as an emoji next to her reply (`Лілі 😄✨:`). It is a **renderer swap, not a contract change** — the v0.3 emotion channel (`EmotionState{reply, emotion, intensity}`, the fixed 9-value enum, `validate()`, the `IEmotionRenderer` interface) is locked and untouched. We only add a new `IEmotionRenderer` implementation (`EmojiRenderer`) and a resolver that maps **emotion + intensity → glyph(s)**.

Two design points carry through the acceptance criteria: (1) `intensity` scales **emphasis, not the feeling** — the *same* face, made stronger by **repeating it or adding an accent emoji** across three bands (low/mid/high; EMOTION.md §6 table); (2) the emoji table is an **editable authored file** (`LUMI_EMOJI_PATH`, default `core/emoji.md`) — like the canon, styles, and nudges — so the user can **change the table and add / remove / replace emojis without touching code**, with a **built-in default** keeping the resolved map **total over the enum** (a missing file / blank row / unknown emotion falls back to the default → base glyph → `calm`, never raising).

## v0.5 — Emoji rendering

### LUMI-023 — Editable emoji map + `EmojiRenderer` (emotion + intensity → glyph)

**Description:**
The v0.5 emoji tier over the locked v0.3 channel: an **editable authored emoji-map file** + loader, a pure `emoji_for(state)` resolver (emotion + `intensity` band → glyph(s)), and an `EmojiRenderer` implementing `IEmotionRenderer`. The built-in default is the EMOTION.md §6 table; the resolved map is **total over the enum** with graceful fallback. **No contract change.** (`/core`.)

**What needs to be done:**
- An **editable emoji-map file** `core/emoji.md` shipping the EMOTION.md §6 default — per emotion, its base **face** + the **low / mid / high** emphasis (repeat the face or add an accent). `#` comments; loaded at startup. A simple line format, e.g.:
  ```
  # emotion = low | mid | high   (one value → all bands; three → low/mid/high)
  joy = 😄 | 😄✨ | 😄✨✨
  calm = 🙂
  sad = 😢 | 😢😢 | 😢😢😢
  ```
- A **loader** (like `load_styles`/`load_nudges`) → the map; a missing file / blank or absent row → fall back to the built-in default.
- `emoji_for(state: EmotionState) -> str`: select the emotion's row, pick the **band by intensity** (low `0.00–0.33`, mid `0.34–0.66`, high `0.67–1.00`), return the glyph(s). **Pure, total over the enum**; unknown emotion / missing row → built-in default → base glyph → `calm`.
- `EmojiRenderer` implementing `IEmotionRenderer` over the loaded map (`render` resolves/exposes the glyph; `set_speaking`/`tick` are no-ops, like `LogRenderer`).
- Config: `LUMI_EMOJI_PATH` (default `core/emoji.md`).

**Dependencies:** None (builds on the **locked** v0.3 emotion channel).

**Expected result:**
A resolved emotion × intensity → emoji map, **user-editable** via `core/emoji.md`, always total over the enum and degrading gracefully.

**Acceptance criteria:**
- [ ] `core/emoji.md` ships the EMOTION.md §6 default (base + low/mid/high; repeat/accent); editable, `#` comments, loaded at startup; `LUMI_EMOJI_PATH` overrides.
- [ ] `emoji_for(state)` resolves emotion + intensity band (3 bands) → glyph(s); the resolved map is **total over the enum**.
- [ ] A missing file / absent or blank row / unknown emotion **falls back** (built-in default → base glyph → `calm`); never raises.
- [ ] `EmojiRenderer` implements `IEmotionRenderer` (`set_speaking`/`tick` no-ops); the v0.3 contract, enum, and interface are unchanged.
- [ ] Unit tests: the loader parses the table; band selection (low/mid/high); total-over-enum; the fallback paths; editing a row (add/remove/change) takes effect.

### LUMI-024 — Show the emoji next to Лілі's reply in the TUI

**Description:**
Render the resolved emoji **beside Лілі's reply label** in the chat (`Лілі 😄✨:`), using `emoji_for(state)` from LUMI-023 — so her emotion reads visually inline, per reply. The status line keeps its `· joy 0.8` text readout. **No contract change.** (`/tui`.)

**What needs to be done:**
- When emitting Лілі's reply, prefix her label with the emoji: `Лілі {emoji}:` (from `emoji_for(state)`). **Per reply**, so scrolling the conversation shows her emotion change line by line.
- Keep the existing status-line `· {emotion} {intensity}` text readout (the precise value alongside the inline face).
- Wire the renderer: the TUI uses `EmojiRenderer` / `emoji_for` for the chat glyph (the `LogRenderer` logging may remain in parallel).

**Dependencies:** LUMI-023 (the `emoji_for` resolver + map).

**Expected result:**
Each of Лілі's replies shows her current emotion emoji next to her name; `intensity` adds emphasis (e.g. `joy 0.9` → `Лілі 😄✨✨:`).

**Acceptance criteria:**
- [ ] Each Лілі reply renders with its emoji next to the `Лілі` label (`Лілі 😄✨:`).
- [ ] The emoji reflects the turn's emotion + intensity band (via `emoji_for`).
- [ ] The status line still shows the `emotion intensity` text readout.
- [ ] Integration test: a reply renders with the expected emoji next to the label; a high-intensity reply shows the emphasized variant.
