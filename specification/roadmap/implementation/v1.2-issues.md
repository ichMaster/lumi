# Version v1.2 — Issue Breakdown

**Phase:** v1.2 — Non-blocking input & message buffer (TUI)

You can keep typing and sending while Лілі is still answering. Today the input box is **disabled**
for the whole turn (`_set_busy` locks it; a submit that slips through is dropped). This phase
**decouples the input widget from the busy state**: the box stays editable, a message you send
**appears in the chat immediately** and is **FIFO-queued**, and Лілі works through the queue **one
turn per message, in order — never merged** (distinct from the v0.13 Telegram burst-consolidation).
A pure **interface** change — **no `core` touch, no contract change**. The hard constraint stays:
`_busy` is the **single-turn mutex** the keyboard, nudge, think, scheduler and inbox all share (two
`core.reply` in flight would race the session/store/`self.last_*`), so the model remains
**serialized** — this phase only stops the *widget* from being held hostage by it. Off
(`LUMI_INPUT_BUFFER`, default) → today's locked-input behavior, **byte-identical**. Full design:
[ROADMAP §v1.2](../../ROADMAP.md).

### LUMI-180 — Decouple input from busy + refactor the dispatch into `_handle_input_line`

**Description:**
The foundation, and a no-behavior-change refactor. Two moves: (1) `_set_busy` stops disabling the
input widget — `_busy` stays the **model mutex** (every proactive path still respects it), but the
box is no longer locked; (2) extract the command / directive / chat dispatch currently inline in
`on_chat_input_submitted` into one reusable `async def _handle_input_line(text, *, echoed=False)` so
both the live submit path **and** the future drain (LUMI-182) route a line identically. Gate the new
behavior on **`LUMI_INPUT_BUFFER`** (off by default): off → the widget is still disabled during a
turn exactly as today (byte-identical); the refactor itself is behavior-preserving whether on or off.

**What needs to be done:**
- `_set_busy`: when `LUMI_INPUT_BUFFER` is on, do **not** set `prompt.disabled` (keep it editable +
  focused); off → today's `prompt.disabled = busy`. `_busy` stays the mutex for the proactive workers.
- Extract `_handle_input_line(text, *, echoed=False)` from `on_chat_input_submitted`: the full
  command table (`/memory`, `/style`, `/model`, …), the `%directive` router, and the fall-through to
  `_run_turn`. `on_chat_input_submitted` becomes: strip/clear the box → call `_handle_input_line`.
- The `echoed` flag threads to `_run_turn` (LUMI-181 uses it) — for now always `False`, no change.
- Config: `LUMI_INPUT_BUFFER` (off) wired into `Config` + `load_config`, passed to the TUI.

**Dependencies:** none.

**Expected result:**
With `LUMI_INPUT_BUFFER` on, the input box stays editable during a turn (nothing else changes yet —
a submit-while-busy is still dropped by the existing guard until LUMI-181); off → the box locks as
today. Every existing command/turn still routes correctly through `_handle_input_line`.

**Acceptance criteria:**
- [ ] `_handle_input_line` handles every case `on_chat_input_submitted` did (commands, directives, chat) — the live path is unchanged (existing TUI tests pass verbatim).
- [ ] `LUMI_INPUT_BUFFER` on → the widget is not disabled during a turn; off → it is (the current behavior).
- [ ] `_busy` still gates the proactive workers (no behavior change there).
- [ ] **No `core` change** beyond the config flag; **no paid calls** (mock model).

### LUMI-181 — Enqueue + immediate echo while busy

**Description:**
The queue. With the buffer on, a submit **while `_busy`** no longer drops — it **appends** the raw
line to a `deque` and **echoes the user row into the chat now**, then returns without starting a turn
(the model is still busy). The input clears and keeps focus so you can immediately type the next. A
`pending`/`echoed` marker rides with the queued line so the eventual turn (LUMI-182) does **not**
re-render the user row.

**What needs to be done:**
- `self._input_queue: deque[str]` on the app (in-memory, per session; not persisted).
- `on_chat_input_submitted`: if `LUMI_INPUT_BUFFER` on **and** `_busy` → append the line, call the
  echo helper to show the user row immediately (the same styling `_run_turn` uses), clear + refocus
  the box, return. (Off, or not busy → today's path.)
- The echo helper shows only the **user** row (no turn); the queued line is tagged so LUMI-182 passes
  `echoed=True` to `_handle_input_line`/`_run_turn`, which then suppresses the duplicate user row.
- A queued **command**/`%directive` is stored as its raw text (routed at drain time, not now).

**Dependencies:** **LUMI-180** (the flag + `_handle_input_line` + the un-disabled widget).

**Expected result:**
While Лілі is answering, typing "а ще?" and pressing enter shows "а ще?" in the chat at once, clears
the box, and leaves it focused; the queue holds one entry; no second turn starts yet.

**Acceptance criteria:**
- [ ] Submit-while-busy (buffer on) shows the user row immediately and enqueues the line; **no turn starts** while busy.
- [ ] The box clears + keeps focus; multiple rapid submits enqueue in order (FIFO).
- [ ] The queued line carries the `echoed` marker so its turn won't double-render the row.
- [ ] Off, or not busy → today's behavior (a submit runs the turn directly); **no paid calls**.

### LUMI-182 — Drain: one turn per queued line, in order

**Description:**
The payoff. When the current turn finishes (`_run_turn` completion), if the queue is non-empty and
`_busy` is clear, **pop the oldest line and run it** through `_handle_input_line(text, echoed=True)`
— so a queued `/command`/`%directive` routes exactly as a typed one, and a chat line runs a normal
turn with the user row suppressed (already echoed by LUMI-181). Each queued message gets **its own
reply**; lines are **never merged**. The drain repeats until the queue empties, one turn at a time
(serialized by `_busy`).

**What needs to be done:**
- A `_drain_input_queue()` invoked at the end of `_run_turn` (in the worker, after `_set_busy(False)`)
  — if `LUMI_INPUT_BUFFER` on and the queue is non-empty and not `_busy`: pop the left, schedule
  `_handle_input_line(text, echoed=True)` as the next worker. It naturally re-enters the drain when
  that turn finishes, so the queue empties one turn per message.
- `_run_turn` / `_handle_input_line` honor `echoed=True` → do **not** re-render the user row (LUMI-181
  already showed it); everything else (mirror to Telegram, store, emotion render) is unchanged.
- Ordering: strictly FIFO; a line enqueued during the drain (you keep typing) goes to the back.

**Dependencies:** **LUMI-181** (the queue + echo marker).

**Expected result:**
Send "перше" then "друге" while she's answering "нульове": after "нульове" finishes she answers
"перше", then "друге" — three separate replies, in order, none merged.

**Acceptance criteria:**
- [ ] On turn completion the next queued line runs; the queue empties one turn per message, FIFO.
- [ ] Each queued line gets its own reply (**never merged**); a queued command/directive routes correctly.
- [ ] The user row is shown once (no duplicate) for a queued line.
- [ ] A line typed mid-drain is appended to the back and answered after the rest; **no paid calls**.

### LUMI-183 — Proactive-worker guards, pending-count surfacing, config, off-pin

**Description:**
The finish. Give the proactive workers (idle nudge / `%think` / scheduler / inbox-drain) a
**queue-empty** guard so a proactive turn **never jumps ahead** of your pending lines. Surface the
backlog: the status line shows a pending count (e.g. `⋯2`). Confirm the whole feature is
**byte-identical off**.

**What needs to be done:**
- Add `and not self._input_queue` to the guards in `_maybe_nudge`, `_maybe_think`, `_sched_tick`,
  `_poll_inbox` (they already check `_busy`) — a proactive act waits until the queue is empty.
- Status line: a compact pending indicator (`⋯{n}` when the queue is non-empty), cleared as it drains.
- The `.env` documents `LUMI_INPUT_BUFFER`; a short operator note (README/docs) of what changes.
- **Off-pin:** a test that with the flag off the widget disables during a turn, a submit-while-busy is
  dropped (today's behavior), no queue is used, and the status carries no pending marker.

**Dependencies:** **LUMI-182** (the drain it guards + surfaces).

**Expected result:**
While messages are queued, no idle nudge / scheduled act cuts in; the status shows `⋯2`; with the
flag off everything behaves exactly as before v1.2.

**Acceptance criteria:**
- [ ] A proactive worker (nudge/think/scheduler/inbox) is skipped while the queue is non-empty.
- [ ] The status shows the pending count and clears as the queue drains.
- [ ] **Off → byte-identical**: widget disabled during a turn, submit-while-busy dropped, no queue, no pending marker (pinned).
- [ ] `.env` + a short operator note document `LUMI_INPUT_BUFFER`; **no paid calls**.

## Notes

- **Sizes:** LUMI-180 **M** (decouple + refactor), LUMI-181 **M** (queue + echo), LUMI-182 **M** (the
  drain — the payoff), LUMI-183 **S** (guards + status + docs + off-pin).
- **No core change, no contract change** — this is a pure TUI/interface phase; `{reply, emotion,
  intensity}`, per-user isolation, and the memory layers are untouched.
- **The model stays serialized** — `_busy` remains the single-turn mutex; the phase decouples the
  *widget*, not the turns. **Each queued message gets its own reply — never merged** (unlike the v0.13
  Telegram burst-consolidation).
- **Off (`LUMI_INPUT_BUFFER`) → byte-identical** — the input locks during a turn exactly as today
  (pinned per issue).
- **Build order:** LUMI-180 → LUMI-181 → LUMI-182 → LUMI-183 (strictly sequential).
