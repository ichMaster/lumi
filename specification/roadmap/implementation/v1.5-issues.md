### LUMI-192 — S1: async POST — the ordered post-turn worker queue

**Description:**
`reply()` currently runs ~4.5–6.5 s of local work **after** the model call (two `append_message`s,
`_index_messages` embeds, the closeness write) before returning — the user stares at a finished
reply that hasn't appeared. Make `reply()` return the validated `EmotionState` **the instant
`validate()` passes**; everything after moves onto a **single ordered background worker queue**.
Ordering is the correctness key: the **next turn's prompt-build drains the queue first** (the next
prompt needs the prior messages persisted), and a clean shutdown / session close **drains fully** —
nothing is lost. Gated on `LUMI_ASYNC_POST` (default off → today's synchronous path, byte-identical).

**What needs to be done:**
- A single ordered worker (one thread + FIFO queue) owned by `Core`: `enqueue(fn)` for the post-turn
  work; `drain()` blocks until empty.
- `reply()` splits: validate → **return state**; the persist/index/closeness tail becomes one queued
  job (keeping today's exact order within the job).
- **Drain points:** the start of the next `reply()`/prompt-build; `end_session`; a `close()`/atexit
  hook for clean shutdown. S0 timing: `post_ms` then measures only the enqueue (the drain cost shows
  up in the next turn's `pre_ms` — note it in `/latency`).
- Config: `LUMI_ASYNC_POST` in `Config` + `load_config` + `build_core` (default off).
- Errors inside a queued job: log + never crash the worker (the next job still runs); a failed
  persist is retried once on the next drain.

**Dependencies:** none (v0.2 `Repository`, v0.16/0.17 index path, v0.3 gate — all shipped).

**Expected result:**
With the flag on, `reply()` returns in ≤ the model-call time; the next turn still sees the previous
turn's messages; a clean exit loses nothing.

**Acceptance criteria:**
- [ ] Unit: the queue is FIFO; `drain()` empties it; drain-on-demand + drain-on-exit (fake clock, no real sleeps).
- [ ] Integration: a full turn returns **before** a (mocked, delayed) persist completes; the record lands after; the next turn's prompt equals the sync path's (drain correctness).
- [ ] A simulated abort mid-window loses at most the un-drained turn; the next start recovers.
- [ ] A queued-job failure is logged, never crashes the worker, retries once on the next drain.
- [ ] **Off (default) → byte-identical** to the synchronous path (pinned).
- [ ] **No paid calls** (mock model + embedder).

### LUMI-193 — S2: incremental store — SQLite message journal behind `Repository`

**Description:**
Every persisted message currently rewrites the whole 12.9 MB `store.json` — twice per turn, O(history)
and growing. Add a **SQLite backend** behind the same `Repository` contract (the ARCHITECTURE-named
next backend): messages insert **O(1)**; the rest of the store (sessions, summaries, facts, closeness,
thoughts…) persists on close/interval as today. **The core does not change** — the swap lives entirely
under the seam. Selected via `LUMI_STORE_BACKEND=json|sqlite` (default `json` → byte-identical).

**What needs to be done:**
- A `SqliteRepository` implementing the full `Repository` contract; messages in an append-only table
  (`user_id`, `session_id`, `ts`, `role`, `text`, …) — `INSERT` per message, indexed reads.
- One DB file beside the store (e.g. `store.db`), one connection layer (WAL mode) — shared later by
  LUMI-194's vectors table.
- The non-message aggregates may stay JSON-serialized (in SQLite or the existing file) — persisting
  them on close/interval, NOT per message; the isolation/staleness invariants carry `user_id` through.
- Config: `LUMI_STORE_BACKEND` in `Config`/`load_config`/the composition root (default `json`).
- A `store.json` **export** path for back-compat (the migration script itself is LUMI-195).

**Dependencies:** none (parallel with LUMI-192).

**Expected result:**
With `sqlite`, one message persists in O(1) — no full-file rewrite — and the per-message persist cost
no longer grows with history.

**Acceptance criteria:**
- [ ] `SqliteRepository` passes the **same contract tests** as the JSON store (round-trip, isolation, clear/forget) — the suite runs against both backends.
- [ ] One message persist = one `INSERT` (no full rewrite); persist cost measured flat as history grows (unit, fake data).
- [ ] Per-user isolation pinned (A's records never readable as B) on the SQLite backend.
- [ ] `LUMI_STORE_BACKEND=json` (default) → **byte-identical** store file + behavior (pinned).
- [ ] **No paid calls.**

### LUMI-194 — S2b: the vector store on SQLite (float32 BLOB, no re-embed)

**Description:**
`store.vectors.jsonl` is **~450 MB**, loads **entirely into RAM at startup**, and appends per turn on
the POST path — the bigger hog than `store.json`. Move the `VectorStore` (already behind `Repository`)
to a **SQLite table** in LUMI-193's DB: keyed by `user_id`, the vector a **float32 `BLOB`**
(~4–8× smaller than JSON-text floats), `INSERT` per message, queried per search (no full-file load),
pruning a plain `DELETE … WHERE ts < ?`. **App-side cosine stays** (brute-force is instant at this
scale); `sqlite-vec` is the later/at-scale upgrade. The migration **re-packs the existing `.jsonl`
rows without re-embedding** — no paid Voyage re-index. The `Embedder` seam and the
`search_vectors(…, kind=…)` contract are untouched.

**What needs to be done:**
- A vectors table (`user_id`, `msg_id`, `kind`, `ts`, `role`, `text`, `parent_msg_id`, `chunk_index`,
  `vector BLOB` float32 little-endian) + `vectors_model` metadata; add/search/reset per the existing
  `VectorStore` contract.
- Search: load the requesting user's candidate vectors (optionally `kind`-scoped) → numpy cosine as
  today; results identical to the JSONL path.
- The in-place `.jsonl` → SQLite re-pack (float32 conversion, batched, idempotent) — invoked by the
  LUMI-195 migration script; `reset_vectors` (model change) drops rows for re-index as today.
- Backfill/id-dedup semantics unchanged (content-addressed ids keep it idempotent).

**Dependencies:** **LUMI-193** (the DB file + connection layer).

**Expected result:**
Vectors persist incrementally (`INSERT`), on-disk size drops ~4–8×, startup no longer loads 450 MB
into RAM, and the migration costs zero embedder calls.

**Acceptance criteria:**
- [ ] The SQLite `VectorStore` passes the **same contract tests** as the JSONL one (add/search/`kind` scope/isolation/reset).
- [ ] Migration round-trip: `.jsonl` → rows → **identical search results**, embedder called **zero** times (mock embedder asserts).
- [ ] float32 `BLOB` round-trips to the same cosine ranking (tolerance-tested against float64 JSON).
- [ ] Per-user isolation pinned on the vectors table.
- [ ] **No paid calls** (mock embedder everywhere).

### LUMI-195 — Off-pin, integration, the migration script, docs

**Description:**
The proof + the guard + the operator path. A full-turn integration across the three pieces, the
off-pins, the one-command migration script (`store.json` + `store.vectors.jsonl` → SQLite, with a
backup and a `store.json` export for rollback), the docs, and the measured before/after recorded in
LATENCY.md (S0 gives the numbers).

**What needs to be done:**
- **Integration:** a full turn with `LUMI_ASYNC_POST=on` + `LUMI_STORE_BACKEND=sqlite`: returns before
  the delayed persist, record + vector land, next-turn prompt correct, abort recovery works.
- **Migration script** (`scripts/migrate_store.py`): store.json → SQLite + vectors re-pack (LUMI-194's
  path), **backs up first**, idempotent, prints counts; a `--export-json` reverse path for rollback.
- **Off-pins:** `LUMI_ASYNC_POST=off` → synchronous, byte-identical; `LUMI_STORE_BACKEND=json` → the
  JSON store untouched; both defaults.
- **Docs:** `.env.example` (both flags + migration how-to); LATENCY.md — S1/S2/S2b marked shipped with
  the measured POST before/after from `/latency`; README current-version note comes with the release.

**Dependencies:** **LUMI-192, LUMI-193, LUMI-194.**

**Expected result:**
One command migrates an existing installation safely; off is provably inert; the POST drop is measured
and recorded.

**Acceptance criteria:**
- [ ] Integration: turn returns ≤ model-call time (mocked delayed persist); records land; next prompt correct; abort → next start recovers.
- [ ] The migration script: backup → migrate → identical reads (messages + search results), idempotent on re-run, `--export-json` round-trips.
- [ ] Both off-pins hold (defaults → byte-identical store file + payloads).
- [ ] Docs cover `LUMI_ASYNC_POST`, `LUMI_STORE_BACKEND`, the migration; LATENCY.md carries the measured numbers.
- [ ] **No paid calls.**

## Notes

- **Sizes:** LUMI-192 **L** (the queue + drain points in `core`), LUMI-193 **L** (the second
  `Repository` backend), LUMI-194 **M** (one table + re-pack on 193's foundation), LUMI-195 **M**
  (integration + script + docs).
- **S0 already shipped** (v1.3.1): `pre_ms`/`llm_ms`/`post_ms` + `/latency` — use it for the
  before/after in LUMI-195. Note: with S1 on, the drained work surfaces in the *next* turn's `pre_ms`.
- **Byte-identical off:** both flags default off/`json`; every issue pins its off-path.
- **No re-embedding anywhere** — the vector migration is a format re-pack; the mock embedder asserts
  zero calls (Voyage is paid).
- **`sqlite-vec` explicitly out of scope** — app-side cosine stays; the KNN index is the at-scale/server
  upgrade named in ARCHITECTURE.
- **Build order:** (LUMI-192 ∥ LUMI-193) → LUMI-194 → LUMI-195.
