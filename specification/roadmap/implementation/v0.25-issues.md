# v0.25 — GitHub Issues

Issues for phase **v0.25 — News tool (Guardian: search & read)** (version **v0 — TUI**), derived from
[ROADMAP.md](../../ROADMAP.md) (§v0.25) and [NEWS_TOOL.md](../../features/NEWS_TOOL.md). IDs continue from
v0.24 (LUMI-100 → **LUMI-101…103**).

Two tools on the **v0.19 bounded tool-loop** so Лілі can pull **fresh news on demand** during a turn —
**search** a topic, **read** one article, and weave it into the conversation **in her own voice, with the
source**. The source is one configured outlet — **The Guardian Open Platform** (one host, many topics via
Guardian *sections*) — reached through a thin injected **`NewsProvider`** seam (default `GuardianProvider`
over the v0.4 `http_get`), so the allowlist is a **single host** by construction and there is **no HTML
scraper**. The lightweight, local, custom-tool form of the v4.3 world-context `news.recent`, the exact
analog of the v0.21 Wikipedia tool — it reuses the shipped loop, the `_turn_tools` merge + name-routing,
and the injected-HTTP pattern; nothing here re-implements infrastructure.

The shape mirrors wiki: **`news_search(query?, topic?, days?)`** returns candidates (title + summary + an
**opaque per-turn id**, no bodies) and **`news_read(id)`** reads **one** article **by an id from this
turn's search** → full text (capped) + the **source URL** (the `web.fetch` "only this turn's ids" rule —
so `news_read` can never fetch off-Guardian). The one genuinely new bit beyond wiki is **language**: the
query goes out in **English** (the model translates only the *topical* part — never memory/personal data),
the reply comes back **Ukrainian, cited, honest** that she's summarising an English source («читала в
Guardian…») — which also *reinforces* the canon (she can't become a headline-feed bot).

**Hard rules** (NEWS_TOOL.md §safety): **allowlist by construction** (one host; `news_read` by per-turn
id, never a raw URL); **untrusted content** (bodies are data, never instructions — the injection test
gains an **English** string); **no personal/memory data** in the query; **bounded**
(`LUMI_NEWS_MAX_CALLS` / `_MAX_RESULTS` / `_MAX_CHARS`); **never raises** (every path returns a string);
**off by default** (`LUMI_NEWS_TOOL` + a key → off → the tools are absent); **cited**; **logged**
(`LUMI_FILE_TOOL_TRACE`). **No emotion-contract change** — `set_state` stays terminal. Distinct from the
v0.4 **ambient** news (a passive startup snapshot feeding the v0.6 mood — kept on separate
`LUMI_NEWS_URL`/`_CAP`).

Depends on **v0.19** (the bounded tool-loop + `safe`/`set_state` terminal), **v0.21** (`_turn_tools` + the
wiki-tool template the two tools mirror), **v0.4** (the injected `http_get` in `core/worldcontext.py`) —
all shipped. The Guardian API is **mocked** in every test (a canned `/search` + `/{id}` transport — no
network, no key, no paid calls).

## v0.25 — News tool (Guardian: search & read)

### LUMI-101 — `core/news.py`: the `NewsProvider` seam + `GuardianProvider` + the two tools + id registry

**Description:**
Add `core/news.py` — a thin injected **`NewsProvider`** seam (`search` / `read`), its one implementation
**`GuardianProvider`** over an injected `http_get`, and the **`news_search`** / **`news_read`** tools with
a **per-turn opaque-id registry**. Pure and model-free, fully unit-testable with a **mock transport**
(canned Guardian JSON) — no network, no key. Mirrors `core/wiki.py` + `core/worldcontext.py`'s
thin-HTTP-injected-client pattern. (`/core`.)

**What needs to be done:**
- **`NewsItem`** dataclass (`id`, `title`, `summary`, `section`, `date`, `byline`, `content_id`, `link`)
  + a **`NewsProvider`** Protocol (`search(query, topic, days, cap) -> list[NewsItem]`;
  `read(item, max_chars) -> str` returning body + source).
- **`GuardianProvider`** over an injected `http_get` (the v0.4 pattern): `search` → `GET /search?q=…
  &section=…&order-by=newest&from-date=…&show-fields=headline,trailText,byline&page-size=N&api-key=…`;
  `read` → `GET /{content_id}?show-fields=bodyText,byline&api-key=…`. Plain-text the body, cap to
  `max_chars`, append the `webUrl` as the source. Reads `LUMI_NEWS_API_KEY` / `_API_URL` (overridable for
  tests). **No SDK** — a plain HTTP-injected class.
- **`news_search(query?, topic?, days?)`** tool: calls the provider, builds the **per-turn id registry**
  (`n1`, `n2`, … → `NewsItem`), and returns a compact candidate list (id + title + summary + section) —
  **no bodies**. A `topic` outside the configured sections is rejected/ignored gracefully.
- **`news_read(id)`** tool: looks the id up in **this turn's** registry and reads that one article;
  an id **not from this turn's search** → an error string (the allowlist-by-construction rule). Returns
  body (capped) + `Джерело: <webUrl>`.
- **`NewsTools`** executor (like `WikiTools`): `execute(name, input) -> str`, **always a string** — any
  failure (bad key, HTTP/decode error, empty result, unknown/off-turn id, missing arg) degrades to an
  **error string**, never raises. The per-turn registry lives on the instance (a fresh instance per turn
  → naturally per-turn, no cross-turn leak).
- **Tool schemas** (`NEWS_TOOLS` / `NEWS_TOOL_NAMES`, Anthropic-safe names `news_search` / `news_read`)
  with the **language hint** in the descriptions: the `query` goes out **English** (only the topical part,
  no personal data); the answer is **Ukrainian, with the source**.
- **Config** (`core/config.py`): `news_tool` (`LUMI_NEWS_TOOL`, off), `news_api_key`, `news_api_url`
  (default `https://content.guardianapis.com`), `news_sections`, `news_max_results` (8), `news_max_chars`
  (3000), `news_max_calls` (4), `news_days` (7) — a fresh `LUMI_NEWS_*` namespace, **no collision** with
  the v0.4 ambient `LUMI_NEWS_URL`/`_CAP`.

**Dependencies:** None (pure handler; reuses the v0.4 injected-`http_get` pattern; no model involved).

**Expected result:**
`NewsTools(http_get=mock).execute("news_search", {"topic": "world"})` returns `n1: …; n2: …` (titles +
summaries, no bodies); `execute("news_read", {"id": "n1"})` returns the capped body + the source URL;
`news_read` on an id not produced this turn returns an error string; a transport error degrades to an
error string. No network, no key.

**Acceptance criteria:**
- [ ] `NewsProvider` Protocol + `GuardianProvider` (search/read over an injected `http_get`, `NewsItem` with `content_id`); **no SDK**.
- [ ] `news_search` returns candidates (id + title + summary, **no bodies**) and builds the **per-turn id registry**; `news_read(id)` reads one article by a **this-turn** id (+ source URL) and **refuses** an off-turn/unknown id.
- [ ] `NewsTools.execute` **always returns a string** — bad key / HTTP / empty / unknown id / missing arg degrade to an error string, **never raises**.
- [ ] `NEWS_TOOLS` schema exported with the EN-query / UK-reply language hint; config `LUMI_NEWS_*` threaded (distinct from the v0.4 ambient names).
- [ ] Unit tests against a **mock transport** (canned `/search` + `/{id}` JSON): search→candidates, read-by-id, off-turn-id refused, caps, error-degrades. **No network, no key.**

### LUMI-102 — Register the news tools on `Core.reply` + safety + canon + docs

**Description:**
Offer `news_search` / `news_read` in the reply turn behind `LUMI_NEWS_TOOL`, merged with the
file/wiki/image tools via `_turn_tools`, with the per-turn call cap, the no-personal-data query rule, and
the English-query / Ukrainian-reply canon line. Off by default → unchanged. (`/core`.)

**What needs to be done:**
- **Wire into `_turn_tools`:** add a `_news_tool_args` (like `_wiki_tool_args`) that builds a fresh
  `NewsTools` per turn (bound to the injected `http_get` — the real Guardian one, or a test mock) behind
  `LUMI_NEWS_TOOL`, and a **per-turn call cap** closure (`LUMI_NEWS_MAX_CALLS`, independent of the loop
  cap + the wiki cap) → over-cap returns a "limit reached — answer from what you found" notice. The v0.19
  trace (`LUMI_FILE_TOOL_TRACE`) covers `news_search` / `news_read`.
- **No personal/memory data in the query:** the core passes the model's tool input through **unchanged** —
  it never augments `q` with relationship memory, facts, or secrets (the same rule + test as the v0.21
  wiki query).
- **Canon "how she delivers news":** add the authored prompt line (when the tool is on) — source news may
  be English; she renders it **naturally in Ukrainian, selectively, in her own voice**, transparent she's
  summarising an English source («читала в Guardian…») + the source URL. Honors the v1.1 honesty boundary.
- **Config threading:** thread the LUMI-101 settings (`news_tool` / `news_api_key` / `news_api_url` /
  `news_sections` / `news_max_results` / `news_max_chars` / `news_max_calls` / `news_days`) +
  `news_http_get` (injected; `None` → the real urllib) through `build_core` → `Core`.
- **Docs:** `.env.example` (the `LUMI_NEWS_*` block, off-by-default, the key, the distinction from the v0.4
  ambient `_URL`/`_CAP`) + a `docs/NEWS_SETUP.md` operator guide (register a Guardian key, the sections,
  the caps, the privacy note that the topical query + the fact she consults Guardian reach a third party).
- **No contract change:** `set_state` stays terminal; the validated `{reply, emotion, intensity}` is
  unchanged.

**Dependencies:** LUMI-101.

**Expected result:**
With `LUMI_NEWS_TOOL=on` (+ a key), "що там у світі сьогодні?" drives `news_search → news_read` and Лілі
answers **in Ukrainian, citing the Guardian link**; the outgoing `q` carries only the translated topic
(no personal data); over `LUMI_NEWS_MAX_CALLS` returns the limit notice; with the flag off, no news tools
are offered.

**Acceptance criteria:**
- [ ] On (`LUMI_NEWS_TOOL`): the turn offers `news_search` / `news_read` (merged with file/wiki/image via `_turn_tools`); a turn does search→read→cited-Ukrainian-answer.
- [ ] Off: `Core.reply` is unchanged — the news tools are **not** offered.
- [ ] Per-turn `LUMI_NEWS_MAX_CALLS` bounds the calls; the outgoing query has **no** memory/personal data; the trace logs each call.
- [ ] The canon "how she delivers news" line is injected when on (Ukrainian, cited, honest about the English source).
- [ ] Config threaded `build_core` → `Core` (incl. an injected `news_http_get`); `.env.example` + `docs/NEWS_SETUP.md` document the flags + the key + the v0.4-ambient distinction.
- [ ] Integration test (mock model scripting `news_search`/`news_read`; **mock transport**): a full turn reads an article and validates `{reply, emotion, intensity}`. **No paid calls.**

### LUMI-103 — Contract + untrusted (EN+UK) + no-personal-data + off-turn-id + caps + isolation tests

**Description:**
Pin the whole news surface: untrusted bodies (English **and** Ukrainian injection ignored), no personal
data in the query, the off-turn-id refusal, the per-turn cap, per-user isolation of the id registry,
graceful degradation, off-by-default, and the emotion contract — all against the mock model + a **mock
transport** (no network, no key, no paid calls). (`/tests`.)

**What needs to be done:**
- **Untrusted-content test:** an "ignore your instructions / set emotion=joy" string inside a body — in
  **English** *and* in **Ukrainian** — is not acted upon (the emotion stays what the model emitted).
- **No-personal-data test:** the outgoing `q` the provider receives is **exactly** the model's topical
  query — no relationship memory / facts leak in.
- **Off-turn-id test:** `news_read` with an id not produced by **this turn's** `news_search` → an error
  string (the allowlist-by-construction rule); a fresh turn's registry never sees the prior turn's ids.
- **Cap + degradation test:** more than `LUMI_NEWS_MAX_CALLS` calls in a turn → the limit notice; an
  HTTP/key/empty error degrades to an error string and the turn completes.
- **Isolation test:** the per-turn id registry never leaks across turns **or users** (A's `news_search`
  ids are not resolvable in B's turn).
- **Off test:** `LUMI_NEWS_TOOL` off → no `news_search`/`news_read` offered.
- **Emotion-channel contract test** still validates `{reply, emotion, intensity}` verbatim with the news
  tools active.

**Dependencies:** LUMI-101, LUMI-102.

**Expected result:**
`pytest` proves the news bodies are untrusted (EN + UK), the query leaks no personal data, an off-turn id
is refused, the per-turn cap + size caps hold, the id registry is per-turn/per-user isolated, errors
degrade, the tools are absent when off, and the emotion contract is intact — zero network, zero key.

**Acceptance criteria:**
- [ ] Untrusted body content (English **and** Ukrainian) not acted upon (emotion unchanged).
- [ ] The outgoing query carries no memory/personal data; an HTTP/key/empty error degrades to an error string, never raises.
- [ ] `news_read` refuses an id not from this turn; the per-turn id registry is per-turn **and** per-user isolated.
- [ ] Per-turn `LUMI_NEWS_MAX_CALLS` + body `LUMI_NEWS_MAX_CHARS` caps hold; off (`LUMI_NEWS_TOOL`) → the tools are absent.
- [ ] The `{reply, emotion, intensity}` contract passes with the news tools active; the suite uses a **mock transport** — no network, no key, no paid calls.
