# Version v0.37 — Issue Breakdown

**Phase:** v0.37 — OpenAI engine: tool-loop + runtime model toggle (GPT-5.5 ↔ Opus 4.8)

Since v0.18 the model lives behind one `LLMClient` seam, but switching to a non-Anthropic provider
**silently drops the bounded tool-loop** (it's Anthropic-only — `core/llm.py:726`) and **never passes
`reasoning_effort`** — so a frontier reasoning model like **GPT-5.5** (or DeepSeek-V4-Pro) runs **blind**
(no file/wiki/news/web/journal/image tools, no `%`-thought-tools) and at **default** reasoning, and any
switch needs a restart. This phase makes a non-Anthropic frontier model a **real Opus 4.8 alternative**:
port the tool-loop to **OpenAI function-calling**, pass **`reasoning_effort`**, and add a **`/model` runtime
toggle**. The Anthropic path is **untouched**; off → unchanged. Code design:
[GPT55_SWITCH_AND_TOOL_LOOP.md](../../../docs/GPT55_SWITCH_AND_TOOL_LOOP.md); operator guide:
[MODELS_SETUP.md](../../../docs/MODELS_SETUP.md).

## Dependency tree

```
LUMI-146 (OpenAI tool-loop)     ── the enabler: tools on non-Anthropic (independent)
LUMI-147 (reasoning_effort)     ── independent (same client, separate concern; can land in parallel)
LUMI-148 (/model toggle)        ── independent mechanically; most USEFUL after 146 (so GPT-5.5 keeps tools)
```

**Parallelization:** all three are largely independent and can land in parallel. **LUMI-146** is the
headline (it's what makes GPT-5.5 usable with tools); **LUMI-148** is best demoed after 146 (otherwise a
`/model gpt-5.5` turn runs tool-less). It's a **port, not new design** — the loop already exists and is
proven (`AnthropicClient._tool_loop`); only the wire format differs.

### LUMI-146 — OpenAI tool-loop (function-calling port)

**Description:**
Give `OpenAICompatibleClient` its own bounded tool-loop using **OpenAI function calling**, mirroring
`AnthropicClient._tool_loop` (`core/llm.py:456`), so the file / wiki / news / web-lookup / journal / image
tools — and the `%`-thought-tools that ride them — **work on GPT-5.5 / DeepSeek-V4-Pro**. Today
`reply_structured` accepts `tools`/`tool_executor` and **throws them away** (`llm.py:726`).

**What needs to be done:**
- **`_to_openai_tools`** — convert Lumi's Anthropic-shaped tools (`{name, description, input_schema}`) to the
  OpenAI form (`{"type":"function","function":{name, description, parameters}}`).
- **`_tool_loop`** (function-calling): `tool_choice="auto"` + the tools on intermediate rounds; on the
  **final round** force `tool_choice="none"` + `response_format={"type":"json_object"}` so the turn always
  terminates. **Terminal** = the first message with **no `tool_calls`** → parse its content with
  `parse_emotion_json` → `{reply, emotion, intensity}` (the v0.3 gate validates). Non-terminal `tool_calls`
  run via `tool_executor`; feed each back as a `role:"tool"` message with the **untrusted** prefix (recall →
  the **recollection** prefix). Handle **parallel `tool_calls`** (execute all) and the **image-result
  divergence** (`view_image` → a follow-up `role:"user"` image turn, since `role:"tool"` can't carry an
  image). Per-round stats so `.lumi/cache-log.jsonl` still tags `kind` per call.
- **`_text_tool_loop`** twin for the think path (text terminal — returns the message text, no JSON parse).
- **Wire** `reply` / `reply_structured` to use the loops when `tools` + `tool_executor` are present; the
  **no-tools path stays byte-identical**.
- **Docs:** update `MODELS_SETUP.md` — tools now work on OpenAI (remove/qualify the "tool-loop goes dark" line).

**Dependencies:** none new (reuses v0.18 `OpenAICompatibleClient`, v0.19 the loop it ports).

**Expected result:**
On `LUMI_PROVIDER=openai` + `LUMI_MODEL=gpt-5.5`, a turn that needs a file/wiki tool **fires the tool** and
the bounded loop terminates with a valid `{reply, emotion, intensity}`. DeepSeek-V4-Pro works via the same
adapter for free.

**Acceptance criteria:**
- [ ] The OpenAI function-calling loop **executes tools** (`tool_executor` called with the right name/args) and feeds results back as `role:"tool"` with the **untrusted** prefix (recall → **recollection**).
- [ ] **Terminal** parses to a valid `EmotionState`; the **forced final round** (`tool_choice:"none"` + `json_object`) always terminates; **parallel `tool_calls`** all execute; the **image-result divergence** is handled.
- [ ] The **no-tools** path is byte-identical to today; the **Anthropic path is unchanged**.
- [ ] **No paid calls** (a mock OpenAI transport scripts the responses).

### LUMI-147 — `reasoning_effort` passthrough

**Description:**
Pass `LUMI_EFFORT` to OpenAI/DeepSeek reasoning models (GPT-5 family, DeepSeek-V4-Pro). Today the adapter
**ignores** thinking/effort (`llm.py:636`), so a "Thinking" model runs at its **default** depth, untunable.

**What needs to be done:**
- Thread `effort` into the `OpenAICompatibleClient` constructor + the builder (`build_llm`/`build_core`,
  from `cfg.effort`).
- An **`_OPENAI_EFFORT`** map (Lumi `low|medium|high|xhigh|max` → OpenAI `low|medium|high`; `xhigh`/`max` →
  `high`).
- Add `reasoning_effort` to **every** OpenAI request (the tool-loop rounds + the single non-tool calls) when
  `effort` is set; omit it when unset.
- **Docs:** note in `MODELS_SETUP.md` that `LUMI_EFFORT` is now honored on OpenAI/DeepSeek.

**Dependencies:** none (independent; composes with LUMI-146 — same client).

**Expected result:**
With `LUMI_EFFORT=high` on `LUMI_PROVIDER=openai`, requests carry `reasoning_effort="high"`; unset → omitted.

**Acceptance criteria:**
- [ ] With `LUMI_EFFORT=high`, `reasoning_effort="high"` appears in the OpenAI request kwargs; unset → the key is absent.
- [ ] The mapping clamps `xhigh`/`max` → `high`; invalid values are dropped safely.
- [ ] The **Anthropic path is unchanged**; **no paid calls** (mock transport).

### LUMI-148 — `/model` runtime engine toggle (Opus 4.8 ↔ GPT-5.5)

**Description:**
A TUI command to swap the **active engine mid-session** without a restart — ideal for the engine A/B the
cost analysis recommends (flip back and forth on the same conversation). Today switching means editing `.env`
+ restart.

**What needs to be done:**
- **`Core.switch_model(provider, model)`** — rebuild the `LLMClient` from the (already-loaded) config keys +
  re-point `self._model`. (The v0.38 per-op tiers still resolve via `_model_for()`; this sets the default /
  reply engine.)
- A **`/model`** TUI command (beside `/mood`/`/core`): `/model` prints the active engine; `/model opus` /
  `/model gpt-5.5` (or a full id) swaps it. Aliases (`opus` → `anthropic`/`claude-opus-4-8`, `gpt-5.5` →
  `openai`/`gpt-5.5`) read from config so ids aren't hard-coded. The status bar already reflects `self._model`.
- Both keys in `.env` already supported; a configured alt model id/alias.

**Dependencies:** none mechanical; **soft on LUMI-146** (so a `/model gpt-5.5` turn keeps its tools).

**Expected result:**
`/model gpt-5.5` then `/model opus` swaps the engine **on the fly, no restart**; the status bar updates; the
conversation continues (history is just messages).

**Acceptance criteria:**
- [ ] `/model` with no arg prints the active engine; `/model opus` / `/model gpt-5.5` swaps it **without a restart**; the status bar updates.
- [ ] `switch_model` rebuilds the client from config keys + re-points the model; **per-user isolation + the `{reply, emotion, intensity}` contract are untouched**; a mid-session switch is safe (cold cache on the first turn, one-off).
- [ ] Aliases come from config (not hard-coded); an unknown alias is rejected with a clear message.
- [ ] **No paid calls** (mock client).

## Notes

- **Sizes:** LUMI-146 **L** (the port — the bulk), LUMI-147 **S**, LUMI-148 **M**.
- **No contract change** — the `{reply, emotion, intensity}` channel + memory record shapes are untouched; the
  v0.3 validation gate already covers JSON-mode/structured output across providers. Off → byte-identical.
- **Anthropic path untouched** in all three (a regression test pins it).
- **DeepSeek-V4-Pro** rides the same OpenAI-compatible adapter — LUMI-146/147 cover it for free (but the
  China-server privacy caveat still applies to the intimate `reply`; see MODELS_SETUP.md).
- **Composes with v0.38 (model routing):** `/model` (LUMI-148) sets the *engine*; v0.38 sets which *tiers*
  run the cheap ops. They stack.
- Build order: **LUMI-146 first** (tools), then **LUMI-148** (the toggle is useful with tools); **LUMI-147**
  any time (independent).
