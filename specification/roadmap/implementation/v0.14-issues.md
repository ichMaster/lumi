# v0.14 ‚Äî GitHub Issues

Issues for phase **v0.14 ‚Äî Local voice (ElevenLabs)** (version **v0 ‚Äî TUI**), derived from
[ROADMAP.md](../../ROADMAP.md) (¬ßv0.14) and [VOICE_LOCAL.md](../../features/VOICE_LOCAL.md). IDs
continue from v0.13 (LUMI-056 ‚Üí **LUMI-057‚Ä¶059**).

Hear –õ—ñ–ª—ñ ‚Äî a separate local app that voices her replies in her ElevenLabs voice, no server. The
voicer is the **twin of the v0.13 `outbox‚Üítelegram` daemon** (here `outbox ‚Üí speaker`): it **reuses
the v0.13 file bus** (`outbox.jsonl` + `state/fifo.py`, records `{id, text, emotion, ts, kind}`),
reads new ids via `fifo.read_since` from a **`spoken` pointer**, **voices only `kind="lili"`** (her
replies ‚Äî your mirrored `kind="user"` keyboard lines are **skipped, never voiced**), one at a time
in ascending order, retries on failure, and **skips the pre-existing backlog on first run**. The
**only** core/TUI change is a one-line outbox gate (write on **voice OR bridge**). ElevenLabs is a
second cloud dependency (`ELEVENLABS_API_KEY` + internet) ‚Äî **optional/toggle**, `elevenlabs` an
optional dep, **mock TTS** in tests (no paid calls). Depends on v0.13 (the bus), v0.1, v0.3.

**Description:**
The voicer reads `outbox.jsonl`, but today the TUI writes it **only** when `LUMI_BRIDGE=on`.
Generalize the gate so voice gets the outbox too, and add the voice config. (`/tui`, `/core`.)

**What needs to be done:**
- **Config** (`core/config.py`): `voice` (`LUMI_VOICE`, off by default), `elevenlabs_api_key`
  (`ELEVENLABS_API_KEY` ‚Äî a **secret**, never logged/committed), `voice_id` (`LUMI_VOICE_ID`),

  `kind="user"` regardless).
- **Test isolation** (`tests/conftest.py`): force `LUMI_VOICE=off` (alongside `LUMI_BRIDGE=off`) so

**Dependencies:** None (builds on the v0.13 outbox).

### LUMI-059 ‚Äî The voicer app (`voice/voicer.py`)

**Description:**
The voicer process ‚Äî daemon 2's twin ‚Äî voicing **only –õ—ñ–ª—ñ's lines** from the outbox, in order,
exactly once each, resumable. (`/voice`.)

**What needs to be done:**
- `voice/voicer.py`, the **pure, tested** core: `voice_pending(outbox_path, spoken_path, tts, play)
  -> int` ‚Äî `fifo.read_since` from the `spoken` pointer; for each record ascending: **`kind="user"`
  ‚Üí skip** (advance the pointer past it, don't voice); **`kind="lili"` ‚Üí voice** (`play(tts.synth(text,
  emotion=‚Ä¶))`, advance on success); on a synth/play **failure ‚Üí stop** (leave the pointer before the
  failed id ‚Äî retry next loop). **First-run skip:** if no `spoken` pointer yet, initialize it to the
  current last outbox id (don't replay the backlog).
- `run()` ‚Äî the glue (`python -m voice.voicer`, like the telegram daemons): build `ElevenLabsTTS`
  + a local `play`, do the first-run skip, loop `voice_pending` with a small sleep; lazy
  `elevenlabs`; `# pragma: no cover`. The `spoken` pointer = `outbox_path.with_suffix(".spoken")`.

**Dependencies:** LUMI-058 (the TTS adapter).

**Expected result:**
With the voicer running, –õ—ñ–ª—ñ's replies are spoken in order, exactly once each; your keyboard lines
are never voiced; the pre-existing outbox is not replayed on first start.

**Acceptance criteria:**
- [ ] Voices `kind="lili"` ascending, one at a time; **`kind="user"` skipped** (pointer still advances).
- [ ] **Retry-on-failure**: a failed synth/play does **not** advance the pointer ‚Üí re-voiced, never lost/doubled.
- [ ] **First-run skip**: a fresh voicer starts from the current tail (no backlog replay).
- [ ] **Resume**: the `spoken` pointer persists across a restart.
- [ ] Unit tests via `MockTTS` + a mock `play` (no network, no real sleeps, no paid calls).

### LUMI-060 ‚Äî Telegram voice messages (daemon 2 voice mode)

**Description:**
Optionally send –õ—ñ–ª—ñ's replies to Telegram as **voice messages** (the voice bubble), not just text ‚Äî
composing daemon 2 (v0.13) with the v0.14 TTS adapter. (`/telegram`, reusing `/voice`.)

**What needs to be done:**
- **Config:** `LUMI_TELEGRAM_VOICE` (off by default). When on, daemon 2 voices instead of plain text.
- **daemon 2 voice branch** (`telegram/outbound.py`): for each `kind="lili"` record, synth via
  `voice/tts.ElevenLabsTTS`, **convert MP3 ‚Üí OGG/OPUS** (ffmpeg ‚Äî already required), and
  `bot.send_voice(chat, BufferedInputFile(ogg), caption=text + emoji)` ‚Äî so you can **read or listen**.
  In voice mode it's **one voice message per reply** (no N-batch ‚Äî voice can't be merged); the
  catch-up cap still applies; `kind="user"` still skipped; optional face photo is mutually exclusive.
- **`mp3_to_ogg(mp3_bytes) -> bytes`** helper (ffmpeg `libopus`, `# pragma: no cover` ‚Äî subprocess).
- **Resilience:** a synth/convert/send failure for one reply logs and **does not advance the pointer**
  (retry next loop), same as the text path ‚Äî never wedges the daemon.

**Dependencies:** LUMI-058 (the TTS adapter); builds on the shipped daemon 2 (v0.13).

**Expected result:**
With `LUMI_TELEGRAM_VOICE=on`, each new –õ—ñ–ª—ñ reply arrives in Telegram as a voice message (text as
caption); your lines are never voiced.

**Acceptance criteria:**
- [ ] `LUMI_TELEGRAM_VOICE` config (off by default); off ‚Üí unchanged text behavior.
- [ ] In voice mode, daemon 2 sends each `kind="lili"` reply via `send_voice` (caption = text + emoji); `kind="user"` skipped.
- [ ] MP3 ‚Üí OGG/OPUS conversion (ffmpeg); one voice message per reply (no batching).
- [ ] A failed synth/convert/send doesn't advance the pointer (retry) and doesn't crash the daemon.
- [ ] Unit tests: voice-mode selection routes to `send_voice` with the caption; `kind="user"` skipped ‚Äî via **mock TTS + a fake bot** (no network, no paid calls).
