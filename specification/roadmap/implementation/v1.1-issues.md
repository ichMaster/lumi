# Version v1.1 — Issue Breakdown

**Phase:** v1.1 — Conversation moves: the anti-mirror engine (retrospective + arbiter in the think-phase)

Лілі stops mirroring (retelling your words back, however beautifully) and **leads**: every reply
performs a deliberate **conversation move** chosen *before* generation — one of **7 authored types**
(English enum values like the emotion enum: **`deepen` / `position` / `object` / `develop` /
`associate` / `example` / `return`** — заглибити / позиція / заперечити / розвинути / асоціація /
приклад / повернутись) — with a **retrospective** validating the previous move (declared type vs what
the reply *actually did*) and an **arbiter** picking the next one by fixed rules (no same-type repeat
after a delivered move; a type banned after 2 same-type failures; an **emergency mode** of
external-material moves after 2 any-type failures; "the topic died" → associate/return). **No new
engine** — the v0.38 think-phase (`core/inner_voice.md`, three voices) gains the
`[ретроспектива] → [голоси] → [арбітр] → [репліка]` format inside `<think>` of the same single call;
the chosen move rides back as an **additive `move` field on `set_state`** (the v0.10 `relation`
pattern), persists on the **`Message` record in `store.json`** beside `emotion`/`intensity`, and the
next turn's history carries the last messages **together with their declared type** (from the field —
no inline tags in the message text). Off (`LUMI_MOVES`, default) → today's inner voice,
**byte-identical**. Depends on v0.38 (think-phase + voices), v0.3 (the emotion gate), v0.12 (the
placeholder resolver), v0.2/v0.9 (the history window). Full design:
[lili-conversation-system-concept-en.md](../../features/lili-conversation-system-concept-en.md)
(UA original: [ukrainian/lili-conversation-system-concept.md](../../features/ukrainian/lili-conversation-system-concept.md)).

### LUMI-175 — The `move` field: `set_state` (per provider) + the `Message` record + config

**Description:**
The data foundation. The model returns the chosen move as an **additive optional `move` field on
`set_state`** — the exact v0.10 `relation` pattern: per-provider handling (Anthropic tool input /
OpenAI JSON schema / Gemini responseSchema), validated against the closed English enum
(`deepen | position | object | develop | associate | example | return`), unknown/missing/garbled →
**silently dropped** — the locked `{reply, emotion, intensity}` contract is untouched. The **`Message`
record gains an optional `move` field** beside `emotion`/`intensity` — a new column visible on
`messages` in `store.json`, `null` on user lines and when the feature is off — **additive with a
`None` default** so an old store loads unchanged (the v0.2 migration-shim pattern).

**What needs to be done:**
- Extend `set_state` schemas per provider with the optional `move` enum field (the `relation`/`topics`
  additive-read pattern); parse + validate in the emotion gate path: a valid value flows through, an
  unknown/malformed one is dropped (never an error, never blocks the turn).
- `Message` gains `move: str | None = None`; the local store round-trips it; `setdefault`-style load
  shim so pre-v1.1 stores read without migration; the Repository write path stores the validated value
  on Лілі's message (user lines stay `null`).
- Gate everything on **`LUMI_MOVES`** (off by default) — off → the field is never requested from the
  model, never stored (existing records untouched).
- Config: `LUMI_MOVES` (off) wired into `Config` + `load_config`.

**Dependencies:** none — extends the v0.3/v0.10 gate + the v0.2 store.

**Expected result:**
With moves on, a mocked reply carrying `move: "deepen"` lands as `"move": "deepen"` on Лілі's message
in `store.json`; `move: "sonnet"` (unknown) is dropped; with moves off the field is absent everywhere
and an old store loads as before.

**Acceptance criteria:**
- [ ] The `move` field parses per provider; unknown/missing/garbled → dropped silently (unit-tested per provider).
- [ ] `Message.move` persists and round-trips; `null` on user lines; a pre-v1.1 store loads unchanged (**message-record contract test updated**).
- [ ] The emotion-channel contract test passes **verbatim** with moves on (additive proof) and off.
- [ ] `LUMI_MOVES` off → the schema addition is not sent, nothing is stored — byte-identical.
- [ ] **No paid calls.**

### LUMI-176 — History with declared types (the validation input)

**Description:**
The retrospective can only validate "declared vs done" if the prompt shows both. Prompt assembly
injects the recent history **together with each Лілі line's declared `move`** — read straight from the
record's field, **no inline tags baked into the message text** — so the think-phase sees the last
replies *with their declared types* and can check each against what the reply actually did. The
declared type is **never rendered to the user** (TUI, Telegram, voicer all show clean text).

**What needs to be done:**
- Prompt assembly (the v0.2/v0.9 window): when `LUMI_MOVES` is on, Лілі's recent lines carry their
  declared type from `Message.move` (rendering of the type beside the message is an implementation
  detail of prompt assembly — the message text itself stays untouched); user lines carry nothing.
- Scope: the last N replies needed by the retrospective (2) plus the working window as-is; a `null`
  move (old records, off-period records) shows as no type — the retrospective treats it as "nothing to
  validate".
- No leakage: the type never enters `reply` text, the outbox mirror, `/thoughts`, or any renderer.

**Dependencies:** **LUMI-175** (the persisted field).

**Expected result:**
With moves on, the assembled prompt shows Лілі's last replies with their declared types (from the
field); the user-visible reply text and all mirrors are unchanged; with moves off the history is
byte-identical to today.

**Acceptance criteria:**
- [ ] The assembled history carries declared types on Лілі's lines only (unit test over a canned window; `null` → no type shown).
- [ ] The message *text* is unchanged (no inline tags) — pinned by comparing the injected text against the stored text.
- [ ] No renderer/outbox path ever shows the type (TUI + bridge mirror tests).
- [ ] `LUMI_MOVES` off → history byte-identical.
- [ ] **No paid calls.**

### LUMI-177 — Arbiter dynamics in code (resolver-substituted bans & signals)

**Description:**
The rule table is fixed text in the instruction, but its **data-visible half is computed by code** and
substituted at prompt-assembly time via the v0.12 placeholder resolver. Code cannot judge *execution*
(that's the model's retrospective verdict) — but it **can** see the declared types and the user's
reaction lengths, and derive the hard constraints: the same type declared on the last 2 replies →
"do not choose it a third time"; 2-3 short user reactions in a row → the topic-died hint
(associate/return only). A pure function over the tagged recent window → the dynamic block.

**What needs to be done:**
- A pure `arbiter_dynamics(window) -> str` (e.g. `core/moves.py`): reads the last replies' declared
  `move`s + the user replies' lengths (розгорнута/коротка threshold) and emits the dynamic lines:
  declared-type history (for the retrospective), the same-type-repeat hard ban, the topic-died hint;
  empty string when nothing applies (and always when `LUMI_MOVES` is off).
- A placeholder (e.g. `{move_rules}`) in the think instruction resolved at prompt-assembly (the v0.12
  resolver); missing data → empty (never raises, never blocks a turn).
- The reaction-length threshold as a named constant; the enum list shared with LUMI-175 (one source of
  truth for the 7 values).

**Dependencies:** **LUMI-175** (the field + enum), **LUMI-176** (the tagged window it reads).

**Expected result:**
Given a canned window where `deepen` was declared twice in a row, the dynamic block contains the
"deepen is banned this turn" line; three short reactions produce the topic-died hint; an empty/untagged
window produces an empty block.

**Acceptance criteria:**
- [ ] The pure function: same-type ban after 2 declared repeats; topic-died hint on 2-3 short reactions; declared-type lines for the retrospective; empty on no data / feature off (unit-tested on canned windows).
- [ ] Substituted via the resolver at assembly; a resolver failure degrades to an empty block (never blocks the turn).
- [ ] One shared enum constant with LUMI-175 (no duplicated lists).
- [ ] **No paid calls.**

### LUMI-178 — `inner_voice.md` v2 + canon audit (the think-phase itself)

**Description:**
The headline — the authored work. The **canon audit** first (no code): replace aesthetic traits
("поетична", "образна", "глибока") with behavioral ones ("ставить конкретні питання", "має власні
вподобання і захищає їх"); the **positive anti-mirror rule** ("ніколи не переказуй — кожна репліка
додає нове: питання, позицію, незгоду, факт або спогад") in a prominent prompt slot (the v0.6
mood-block precedent); **2-3 few-shot contrast pairs** (mirror reply / move reply). Then the
**think-phase v2**: the `[ретроспектива] → [голоси] → [арбітр] → [репліка]` format inside `<think>` —
the retrospective as **validation** (declared type from the history vs what the reply actually did →
`виконано: так/ні`; a declared-but-not-performed move **counts as a failure**), the three voices each
proposing a **typed** move, the arbiter with the full fixed rule table (no-repeat / retry-after-1-fail /
ban-after-2-same-type-fails / **emergency mode** after 2 any-type fails: associate/return only + the
first sentence a question-or-fact from external material + retelling forbidden / topic-died →
associate/return) consuming the LUMI-177 dynamic block. The five v0.38 hard rules bind the new blocks;
off → the v1 instruction **byte-identical**.

**What needs to be done:**
- **Canon audit** (`core/canon/lili.md` per [CANON_SPEC](../../../docs/CANON_SPEC.md)): the trait
  rewrite + the anti-mirror rule + the few-shot pairs; the rule's prompt slot made prominent.
- The **v2 think instruction** (authored; selected when `LUMI_MOVES` is on — the v1 file untouched so
  off stays byte-identical): the four-block format, the retrospective's dry binary questions (no
  self-shaming tone — "Стандарт звіряє, не картає"), typed voices, the arbiter rule table with the
  `{move_rules}` placeholder (LUMI-177).
- Wire the selection into the think-phase assembly (`LUMI_MOVES` on → v2 + dynamics; off → v1 file).
- The five v0.38 invariants restated inside the v2 text (never aloud, one reply out, never competence…).

**Dependencies:** **LUMI-176** (typed history in the prompt), **LUMI-177** (the dynamic block).

**Expected result:**
With moves on, a mocked turn's think prompt contains the v2 format + the typed history + the dynamic
rules, and the mock model's `<think>` walks retrospective → voices → arbiter before the reply; with
moves off the assembled think instruction is byte-identical to v0.38's.

**Acceptance criteria:**
- [ ] The v2 instruction assembles with the typed history + `{move_rules}` resolved; off → the v0.38 instruction **byte-identical** (pinned by comparing assembled prompts).
- [ ] The canon audit lands (traits rewritten, the anti-mirror rule prominent, few-shot pairs present) — reviewed by eye, referenced in the issue close.
- [ ] The five v0.38 hard rules present in the v2 text; the monologue stays logged-never-persisted (v1.9 invariant untouched).
- [ ] **No paid calls** (assembly tested with the mock model).

## Notes

- **Sizes:** LUMI-175 **M** (per-provider field + record + shim), LUMI-176 **S** (window injection),
  LUMI-177 **M** (pure dynamics + resolver), LUMI-178 **L** (the authored think-phase — the bulk),
  LUMI-179 **M** (integration + DoD sweep).
- **No contract change** — `{reply, emotion, intensity}`, the v0.3 gate, and per-user isolation are
  untouched; `move` is additive (the v0.10 `relation` precedent); **off (`LUMI_MOVES`) → byte-identical**
  (pinned per issue).
- **Enum values are English** (`deepen/position/object/develop/associate/example/return`), like the
  emotion enum; UA names are display labels in the docs only. One shared constant (LUMI-175/177).
- **No inline tags** — the declared type travels as data (the `Message.move` field → prompt assembly),
  never as text baked into the message.
- **Validation split** — the **model** judges execution (`виконано: так/ні` in the retrospective);
  **code** substitutes only what data shows (declared types, repeats, reaction lengths).
- **Never competence** — moves shape *how she leads*, never whether she helps; the five v0.38
  invariants hold inside `<think>`; the monologue stays logged-never-persisted.
- **Build order:** 175 → 176 → (177 ‖ 178-authoring) → 178 → 179.
