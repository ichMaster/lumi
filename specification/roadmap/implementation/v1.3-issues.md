# Version v1.3 — Issue Breakdown

**Phase:** v1.3 — Explicit Gemini prompt cache (TTL 1h): warm prefill across pauses

On the Gemini engine the prompt cache stops being best-effort: the stable prefix (the v0.15
cacheable head, ~12 K tok measured) is held in an **explicit server-side `cachedContent`** with a
**chosen TTL** (`LUMI_PROMPT_CACHE_TTL`, `1h`), so the first reply **after a pause** starts from a
warm prefill instead of a cold ~12 K re-read. A **speed** phase (the target is TTFT — the
after-a-pause latency spike; implicit hits decay within minutes: measured 2026-07-14, a 104 s gap
degraded the hit 12,094 → 8,064). Today [core/llm.py `GeminiClient`](../../../core/llm.py) only
*reads* `cachedContentTokenCount` — it never creates or references a cache. Scope: the **reply
channel** only (the think channel's head isn't byte-stable — the PROMPT_OPTIMIZATION_III T5
candidate, out of scope). Off (`LUMI_GEMINI_EXPLICIT_CACHE`, default) → today's implicit-only
requests, **byte-identical**. Full design: [ROADMAP §v1.3](../../ROADMAP.md); background:
[docs/PROMPT_OPTIMIZATION_III.md](../../../docs/PROMPT_OPTIMIZATION_III.md),
[features/LATENCY.md](../../features/LATENCY.md).

### LUMI-184 — Probe: verify `cachedContents` on the active Gemini model

**Description:**
The de-risk step. `gemini-3.1-pro-preview` is a **preview** id, and explicit-cache support (plus its
exact constraints) must be verified **before** building the lifecycle. Extend
[scripts/gemini_probe.py](../../../scripts/gemini_probe.py) (the v0.39 dev script — manual, paid,
never CI) with a cache probe that answers, against the live API: does `cachedContents.create` accept
this model id; what is the **minimum cacheable size**; may the cached object carry the
`systemInstruction` (and must the request then omit its own); does `generateContent` with
`cached_content` report the hit in `cachedContentTokenCount`; does a TTL update work; does delete work.

**What needs to be done:**
- A `--cache` mode in the probe: create a `cachedContent` (a realistic ~12 K-tok system text,
  `ttl: "3600s"`) → call `generateContent` referencing it → assert `cachedContentTokenCount` ≈ the
  cached size → patch the TTL → delete. Print each step's verdict.
- Probe the **constraint** that shapes LUMI-186: a request that carries **both** `cached_content`
  and its own `systemInstruction` — record whether the API rejects it (expected: it does; the
  volatile tail must then ride in the leading `contents`).
- Record the verdict (supported / min size / constraints / prices if surfaced) in this file under a
  **Probe result** note and in the LUMI-186 issue before starting it.

**Dependencies:** none.

**Expected result:**
A one-command probe that prints a clear supported/unsupported verdict and the constraint answers;
the LUMI-186 design confirmed or amended before any core code is written.

**Acceptance criteria:**
- [ ] `uv run python scripts/gemini_probe.py --cache` runs the create → reference → hit-check → TTL-update → delete sequence and prints each verdict.
- [ ] The `systemInstruction`-beside-`cached_content` constraint is answered explicitly.
- [ ] The verdict is recorded (this file + the LUMI-186 issue); **manual + paid — never CI** (cents; no test depends on it).

### LUMI-185 — `latency_ms` on every cache-log record (all providers)

**Description:**
The [LATENCY](../../features/LATENCY.md) **S0 slice** this phase needs to prove itself: today the
core tracks a turn's `latency_ms` in `last_stats` but **persists it nowhere**, so the cached-vs-cold
TTFT delta — the whole point of v1.3 — can't be measured from history. Add per-call wall-clock to
the per-call cache-log record ([core/cache_log.py](../../../core/cache_log.py)), for **every**
provider and channel (`reply`/`think`/`mood`/housekeeping).

**What needs to be done:**
- Measure wall-clock around each model call (`time.perf_counter()` at the call site that already
  feeds `_register_usage`) and thread it into the `CacheEvent` → the JSONL record (`latency_ms`,
  int; absent → `null`-safe readers).
- All providers (Anthropic / OpenAI / Gemini / DeepSeek) get it for free if measured at the shared
  call seam — measure once, not per client.
- The cache report's per-channel table gains a median-latency column (best-effort; old records
  without the field are skipped, never crash the report).

**Dependencies:** none.

**Expected result:**
Every new cache-log line carries `latency_ms`; the report can answer "what does a warm vs cold turn
cost in seconds" from real data.

**Acceptance criteria:**
- [ ] Every model call appends `latency_ms` to its cache-log record (unit: mocked client, the field present and plausible).
- [ ] Old records without the field still parse; the report renders (backward-compatible).
- [ ] The report shows per-channel median latency; **no paid calls** (mock model).

### LUMI-186 — Explicit-cache lifecycle in `GeminiClient`

**Description:**
The core of the phase. When `LUMI_GEMINI_EXPLICIT_CACHE` is on (and the provider is Gemini), the
client maintains **one explicit `cachedContent` per (model, prefix-fingerprint)**: create it from
the `cache_prefix` (the v0.15 stable head) with the TTL mapped from `LUMI_PROMPT_CACHE_TTL`
(`5m`→`"300s"` / `1h`→`"3600s"`), **reference it on every reply call**, **recreate when the prefix
fingerprint changes** (a mood flip or digest rebuild → once, not per turn), best-effort **delete**
the stale object, and on **any** API error (or an unsupported model, per the LUMI-184 verdict)
**degrade to today's implicit path** — one log line per session, never a failed turn. **Exact
model-id binding**: a `/model` switch drops the handle and recreates lazily. Per the probe: the
request referencing a cache carries **no own `systemInstruction`** — the volatile tail rides as the
leading `contents` block.

**What needs to be done:**
- A small cache-handle state on the client: `{name, model, fingerprint, expires_hint}`; the
  fingerprint reuses the v0.15 cache-monitor prefix hash ([core/cache_log.py](../../../core/cache_log.py)).
- Lifecycle: no handle / fingerprint changed / model changed → `cachedContents.create` (prefix as
  the cached `systemInstruction`, TTL from config) + best-effort delete of the old; then reference
  via `cached_content` and send the **tail + messages** only (no request `systemInstruction`).
- Fallback: create/reference failure → drop the handle, log once (`explicit cache unavailable →
  implicit`), run the call exactly as today. The turn **never** fails because of caching.
- Config: `LUMI_GEMINI_EXPLICIT_CACHE` (off by default) in `Config` + `load_config`; TTL reuses
  `LUMI_PROMPT_CACHE_TTL` (no new TTL knob).
- Usage accounting unchanged: `cachedContentTokenCount` already maps to `cache_read_tokens`.

**Dependencies:** **LUMI-184** (the probe verdict — supported + the systemInstruction constraint).

**Expected result:**
With the flag on and TTL `1h`, a reply after a 20-minute pause still reports `cache_read` ≈ the full
prefix (warm prefill — no cold re-read); a mood flip recreates the cache exactly once; pulling the
network or a bad model id degrades to implicit with a single log line and the reply still arrives.

**Acceptance criteria:**
- [ ] First flagged call creates the cache (`ttl="3600s"` for `1h`); subsequent calls reference it and carry **no request `systemInstruction`** (mocked API asserts the payload shape).
- [ ] Prefix change → exactly **one** recreate + best-effort delete of the stale object; model switch → lazy recreate.
- [ ] Any cache-API failure → implicit fallback, logged once, the turn succeeds (mocked failure test).
- [ ] **Off → byte-identical request payloads** to today (pinned); the `{reply, emotion, intensity}` contract and the prompt text untouched.
- [ ] **No paid calls** in tests (mocked Gemini caches API).

### LUMI-187 — Observability, continuity report, docs, off-pin

**Description:**
The proof. Log the explicit-cache lifecycle as first-class cache events (`created` / `recreated` /
`fallback`, with the cause), teach the cache report a **continuity view** — do post-gap turns stay
warm? — and record the measured before/after (post-pause `latency_ms` with the flag off vs on, the
LUMI-185 field) in PROMPT_OPTIMIZATION_III. Pin the off-path.

**What needs to be done:**
- Cache-log events for the lifecycle (kind `cache-mgmt` or a field on the reply event): `created` /
  `recreated(cause: prefix|model)` / `fallback(cause)`.
- The cache report (`.lumi/cache-report.md`): an explicit-cache section — creates/recreates per
  session, and the **gap table**: turns after a > 5-min gap with their `cache_read` vs the prefix
  size and their `latency_ms` (the continuity proof).
- Docs: PROMPT_OPTIMIZATION_III gains the measured before/after; `.env.example`/operator note for
  `LUMI_GEMINI_EXPLICIT_CACHE`; the ROADMAP phase marked with the probe verdict.
- **Off-pin test:** flag off → no caches API call ever, request payloads unchanged, no new log events.

**Dependencies:** **LUMI-186** (the lifecycle it observes), **LUMI-185** (the latency field it reports).

**Expected result:**
The cache report shows explicit-cache continuity across gaps (post-pause `cache_read` ≈ prefix) and
the latency delta; the docs carry the measured numbers; off is provably inert.

**Acceptance criteria:**
- [ ] Lifecycle events (`created`/`recreated`/`fallback` + cause) appear in the cache log and render in the report.
- [ ] The report's gap table shows post-gap turns with `cache_read`, prefix size, and `latency_ms`.
- [ ] PROMPT_OPTIMIZATION_III records the measured before/after; the operator docs cover the flag.
- [ ] **Off → no caches API call, unchanged payloads, no new events** (pinned); **no paid calls** (mock).
