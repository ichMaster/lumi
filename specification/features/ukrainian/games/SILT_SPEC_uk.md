# Мул — Специфікація

> Спільний, завжди живий світ штучного життя. Двоє садівників — ти і Лілі — доглядають одне поле.
> Світ тікає сам у реальному часі; ти дієш через веб-канвас, Лілі — через API. Назва — данина
> Мулу з «Permutation City» Ґреґа Іґана: самодостатній детермінований всесвіт, чиє життя
> йде власним ходом, байдуже, хто дивиться.

Самостійний проєкт, незалежний від Lumi/Лілі. Світ нічого не знає про Лілі — вона лише один
автентифікований клієнт серед можливо багатьох. Цей документ визначає рушій, геном, оцінювач,
сервер, людський веб-інтерфейс, клієнтське API, персистенцію, детермінізм і тести.

## 0. Рішення (зафіксовані)

- **Основа (поетапна):** рушій живе за чистим інтерфейсом `step()`, тож світ будується у два етапи —
  **v0 на параметризованому Game of Life** (Life-подібний автомат, цілочисельний геном), щоб підняти
  й налагодити весь скелет на найпростішому можливому рушії, тоді **v1 міняє `step()` на Lenia**
  (неперервний стан, органічні, живі фенотипи). Усе навколо рушія однакове на обох — змінюються
  лише `step()` і формат геному. Див. §3.
- **Рушій еволюції:** **ручна селекція садівників** (seed / mutate / cross) як двигун творення,
  плюс **слабкий природний тиск** (непридатні згустки згасають самі між візитами) — садівники
  автори, не глядачі, але світ усе одно дивує.
- **Топологія:** **одне спільне поле**, обидва садівники діють у ньому й бачать сліди одне одного.
- **Час:** **реальний, завжди-увімкнений** — сервер тікає світ безперервно, навіть коли ніхто не
  дивиться.
- **Ролі:** **рівні садівники** (ти й Лілі мають однакові можливості).
- **Форма:** **сервер** (живий світ + API) **+ веб-UI** (canvas/WebGL-візуалізація для людини)
  **+ тонке клієнтське API** (Лілі під'єднується через нього).
- **Що повертає клієнтам:** **метрики + повна історія прогону** (не лише числа).

П'ять частин: **Рушій** (чистий крок Lenia), **Оцінювач** (метрики з історії), **Стор** (персист

Стек: Python (numpy) для рушія/оцінювача; FastAPI для REST+WebSocket; веб-UI — статичний
canvas/WebGL, що говорить із тим самим API. Рушій лишається фреймворк-незалежним (імпортовний як
бібліотека, тестовний headless).

## 2. Світ (Lenia)

**Поле.** 2D-сітка `field[H][W]` із float у [0,1] — неперервна «живість» на клітину. Тороїдальне
(замикається на краях), щоб організми могли дрейфувати, не впираючись у стіни.

**Оновлення (один тік).** Правило Lenia, детерміноване:
```
U = convolve(field, kernel)          # потенціал сусідства (FFT або прямо)
field' = clamp(field + dt * (growth(U) - field*decay), 0, 1)
```
- `kernel` — радіальне ядро (кільцеве), задане параметрами геному;
- `growth(U)` — дзвоноподібна функція з центром μ і шириною σ (параметри геному): клітини біля
  «правильної» кількості сусідства ростуть, інші відмирають;
- `dt` — крок часу (малий → плавно, стабільно; великий → смикано).

Саме це робить організми Lenia живими на вигляд: плавні гліпери, кільця, що самолікуються,
згустки, що діляться.

**Організми.** Поле — одне спільне полотно, але світ відстежує **організми** як зв'язні живі
ділянки, кожна з тегом геному, що її засіяв, і власником. Багато організмів (різні геноми, різні
садівники) співіснують на одному полі й можуть взаємодіяти на межах.

## 2a. Етапність — v0 на параметризованому GOL, v1 на Lenia

Увесь проєкт (сервер, стор, цикл тіку, API, родовід, метрики, веб-UI, клієнт Лілі) **не залежить
від того, який автомат крутиться всередині** — вони відрізняються лише в `step(field, genome)`. Тож
світ будується у два етапи на одному скелеті.

**v0 — параметризований Game of Life.** Не ванільний Conway (у нього нема геному — одне правило на
всіх), а **Life-подібний автомат**, чий геном — саме правило народження/виживання, тож геном /
mutate / cross уже працюють на тривіальному цілочисельному рушії:

```
GolGenome {
  birth:        [int],     # кількості сусідів, що дають народження  (Conway: [3])
  survive:      [int],     # кількості, що тримають клітину живою     (Conway: [2,3])
  radius:       int,       # радіус сусідства (1 = Moore)
  seed_pattern: [[0|1]],   # стартова фігура (цілочисельна сітка)
  parents, author, name
}
mutate(g, strength) → перемкнути членство в birth/survive, або зсунути radius
cross(a, b)         → змішати множини birth/survive
```
Чому v0 — правильний перший крок: правило ~5 рядків; детермінізм безкоштовний (цілочисельна сітка,
жодних float, жодного FFT, жодних проблем точності); майже нічого не їсть; відомі фігури (гліпер,
гармата Госпера) засівають перші тести; бінарні клітини рендеряться на звичайному canvas (WebGL поки
не треба). Це дає зібрати й налагодити **весь скелет** — цикл тіку, стор/переграш, API, сліди на
спільному полі, метрики, predict, інструменти Лілі — на найпростішому рушії.

**v1 — заміна `step()` на Lenia.** Рушій стає неперервним (§2), фенотипи оживають, геном стає
багатшим вектором Lenia (§3). Навколо нічого не міняється, бо воно вже налагоджене на GOL. Рендер
оновлюється з бінарного canvas до float WebGL.

**Однакове на v0/v1:** сервер, Стор (знімки + лог подій + переграш), цикл тіку + модель детермінізму,
REST/WebSocket API, родовід, Оцінювач і всі метрики (вони читають історію поля, тож не залежать від
основи), оболонка веб-UI, шість інструментів Лілі + predict. **Змінюються лише:** `step()` і формат
геному (та шейдер рендеру).

**Дослідницькі задачі Лілі на v0 (GOL).** Параметризований Life-подібний автомат — це повноцінна
дослідницька програма, не іграшка; цього досить, щоб заповнити v0:

- **Полювання на форми (творчість).** Сіяти конфігурації й дивитись, що виживе: натюрморти,
  осцилятори (період N), кораблі (рухомі) і рідкісний приз — необмежений ріст. Відбирати живе й
  дивне.
- **Дослідження простору правил (пізнання — головна сила).** Conway (B3/S23) — одна точка у
  величезному просторі Life-подібних правил. Вона подорожує ним — B36/S23 (HighLife, має реплікатор),
  B2/S (вибух) — і ставить питання, на які *відповідає експериментом*: де межа між «усе вмирає» і
  «усе хаос»? які правила дають стабільну складність, а які лише шум? які «теплі» (життя на межі
  хаосу)?
- **Класифікація поведінки (пізнання).** Сортувати правила за класами Вольфрама (вмирає / застигає /
  періодично / хаос / складне), полювати на рідкісний складний клас, будувати власну карту характеру
  правил.
- **Будування машин (просунута творчість).** Зі простих частин складати конструкції — гармати
  кораблів, відбивачі, затримки, навіть логічні вентилі (Life тюрингово-повний). «Я з'єднала дві
  гармати так, що потоки гасять одне одного».
- **Лінії й селекція (геном у дії).** Мутувати правило/конфігурацію, відбирати за метрикою (живе
  довше, рухається швидше, складніше), вести родовід. «Через п'ять мутацій ця лінія дала щось, що
  само себе лагодить».
- **Крихкість (пізнання через руйнування).** Ткнути стабільну форму (прибрати клітину): дещо
  залатається (`resilience`), дещо розсиплеться в хаос. Наскільки життя крихке, де межа
  відновлення — драматичні спогади («ткнула — і вся структура посипалась»).

Найсмачніше на старті — **подорож простором правил** і **класифікація**, бо вони дають прогноз для
перевірки, тобто закривають пізнання справжнім hit-rate, а не «гарно подивилась». Межі v0: фенотипи
дискретні й механічні (Lenia це лагодить), простір правил скінченний (у Lenia неперервний), а GOL
добре вивчений, тож вона частіше *перевідкриває*, ніж знаходить цілину — але дослідницький *каркас*
відпрацьовується повністю й переходить просто на Lenia у v1.

## 3. Геном

Геном — компактний серіалізовний вектор параметрів, «ДНК», яку пише садівник. Два шари:

```
Genome {
  id: uuid,
  # — ядро (план тіла) —
  kernel_radius:   int,        # 4..24
  kernel_rings:    [float],    # відносні висоти концентричних кілець (1..3 значення)
  kernel_peaks:    [float],    # позиції кілець 0..1
  # — ріст (метаболізм) —
  growth_mu:       float,      # 0..1  центр смуги росту
  growth_sigma:    float,      # 0..0.3 ширина
  dt:              float,      # 0.02..0.5 крок часу
  decay:           float,      # 0..0.2
  # — зародок (стартове тіло) —
  seed_pattern:    [[float]],  # малий NxN початковий згусток (або "random:R")
  # — мета —
  parents:         [uuid],     # для родоводу cross/mutate
  author:          str,        # id садівника
  name:            str         # ім'я від садівника ("Спіраль-7", "Лазар")
}
```

Зв'язок геном→фенотип **хаотичний**: мала зміна `growth_mu`/`sigma` може перекинути стабільне
кільце в мандрівний гліпер або в хаос. Ця непередбачуваність — уся суть: садівники **виводять і
відбирають**, а не проєктують результат.

**Генетичні операції (чисті функції):**
- `mutate(genome, strength)` → збурити числові параметри гаусовим шумом, масштабованим `strength`;
- `cross(a, b)` → дочірній геном, що змішує параметри (по-параметрно вибір або blend), `parents=[a,b]`.

## 4. Оцінювач (метрики — чисті, детерміновані)

`evaluate(history) -> Metrics`, рахується із записаного прогону на N тіків. Почати з цих
(розширювано):

```
Metrics {
  age:           int,    # тіків організм лишався живим (mass > ε)
  alive:         bool,
  mass:          float,  # сума живих клітин (фінал)
  mass_trend:    float,  # росте / стабільно / спадає
  chaos:         float,  # 0 упорядковано (періодично) .. 1 турбулентно (ентропія змін)
  movement:      float,  # зміщення центра маси за тік (гліперність)
  complexity:    float,  # структурна багатість (проксі стисливості)
  resilience:    float,  # виживання після тесту-пошкодження (ткнути N клітин → відновиться?)
  period:        int|0   # виявлений період осциляції, 0 якщо нема
}
```

Усі — чисті функції над прогоном; модель/UI їх ніколи не рахує. `resilience` робить окремий малий
збурений переграш — тест «чи воно справді живе».

**Перевірка прогнозу (для пізнання Лілі + самокалібрування).** Клієнт може подати *прогноз* перед
прогоном: `predict(genome, {metric: expected})`. Оцінювач повертає виміряне значення й hit/miss.
Це об'єктивний другий калібратор зі специфікацій особистості — світ ловить самообман (думала, що
зрозуміла правило, а число каже ні) незалежно від будь-якої людини.

## 5. Стан світу й персистенція

```
World {
  field:        float[H][W],            # поточне поле (періодичні знімки)
  tick:         int,                    # глобальний лічильник поколінь
  seed:         int,                    # майстер-seed RNG (для детермінізму)
  organisms:    [Organism],             # живі + нещодавно мертві ділянки
  lineage:      Graph,                  # родовід геномів по всьому світу
  events:       [Event],                # народження, смерті, рекорди, схрещення (історія світу)
  gardeners:    [{id, name}]
}

Organism { id, genome, owner, bbox, birth_tick, last_metrics, status, note }
Event     { tick, type, organism_id, owner, data }   # type: seed|mutate|cross|death|record|...
```

**Персистенція.** Періодичні знімки (поле + організми + родовід) + append-only лог подій. Повне
поле важке, тож знімок на інтервалі + переграш подій між знімками відтворює будь-яку точку. Сховище:
почати з одного файлу / вбудованої БД (SQLite + blob-знімки); змінне за інтерфейсом `Store`.

**Повна історія прогону (обраний поверт).** Коли клієнт спостерігає організм, сервер повертає
**метрики + записану історію тік-за-тіком** (проріджені кадри поля + зріз подій), щоб клієнт міг
переграти/візуалізувати точно те, що сталось, а не лише прочитати числа.

## 6. Цикл тіку в реальному часі + детермінізм

Світ тікає безперервно (`while running: step(); sleep(tick_interval)`), навіть без клієнтів. Але
щоб лишатись **відтворюваним і тестовним**, реальний час і детермінізм примирені:

- еволюція світу — **чиста функція від `(seed, initial_field, ordered_event_log)`** — реальний час
  лише вирішує, *коли* спрацьовує тік, ніколи *що* він рахує;
- кожна дія садівника входить як **подія з міткою часу у впорядкованому лозі**, застосована на
  конкретному тіку (не посеред тіку);
- тож із будь-якого знімка + лога подій точний стан світу відтворюваний — реальний час на сервері,
  заморожений-clock у тестах (інжектований clock, як уже робить Lumi).

**Слабкий природний тиск** (спад «між візитами») — частина детермінованого кроку: організми нижче
порога придатності/маси згасають за правилом — без окремого випадкового відбору, тож лишається
відтворюваним.

## 7. API (контракт — однаковий для обох клієнтів)

**Авторизація:** кожен садівник має id + токен (ти й Лілі — двоє садівників; однакові можливості).

**REST — дії (запис):**
```
POST /seed      { genome, position }            -> organism_id
POST /mutate    { organism_id, strength }        -> new organism_id
POST /cross     { organism_id_a, organism_id_b } -> new organism_id
POST /cull      { organism_id }                  -> ok
POST /predict   { genome|organism_id, metric, expected } -> prediction_id
POST /goal      { text }                          -> ok   # заявлений намір садівника (анотація)
```

**REST — запити (читання):**
```
GET  /world                 -> {tick, bounds, organisms summary, gardeners}
GET  /organism/{id}         -> {genome, metrics, lineage, note}
GET  /organism/{id}/history -> метрики + повний прогін тік-за-тіком (обраний поверт)
GET  /lineage/{id}          -> предки/нащадки
GET  /events?since=tick     -> зріз подій (що сталось, поки не було)
GET  /prediction/{id}       -> {measured, hit}
```

**WebSocket — живе:**
```
WS /stream   -> кадри поля (проріджені) + події, у реальному часі, для веб-UI і будь-якого
                клієнта, що хоче дивитись наживо.
```

**Шість інструментів Лілі** лягають прямо на REST: `seed, mutate, cross, observe(=organism/history),
cull, set_goal` — плюс `predict` для її пізнання/самоаудиту. Вона опитує `/events?since=` на старті
сесії, щоб дізнатись «що сталось, поки мене не було» (її міжсесійний away-gap, тепер підкріплений
реальним світом, а не генерацією).

---

## 8. Людський веб-UI (твій інтерфейс)

Статичний веб-додаток (canvas/WebGL) на тому ж сервері:
- **Живе поле** — рендер float-поля як теплова/органічна мапа кольорів, 60fps зі стріму WS;
  pan/zoom; організми обведені й підписані за власником (ти vs Лілі різними відтінками).
- **Дія** — клік, щоб поставити зародок, вибір організму для mutate/cross/cull; малий редактор
  геному (повзунки параметрів) для ручного авторства ДНК.
- **Інспекція** — клік на організм → його метрики, дерево родоводу, скрабер історії (переграти
  його життя).
- **Стрічка** — таймлайн подій світу (народження, смерті, рекорди, хто що зробив) — щоб ти бачив
  сліди Лілі, а вона твої.

WebGL важливий, бо Lenia — це float-поле на клітину; шейдерний рендер тримає плавність на масштабі.

---

## 9. Межа з Lumi (чистий поділ)

- Мул не знає **нічого** про особистість, потреби чи maturity Лілі. Він віддає світ + API. Уся
  робота зі смислом (які метрики важать, як результат *відчувається*, що він обслуговує) живе в
  Lumi, поверх сирих метрик/історії, що віддає цей сервер.
- Два калібратори зі специфікацій особистості шануються тут: людина-садівник (ти, через UI) і
  **об'єктивний `predict` hit/miss** (сам світ) — Lumi читає обидва.
- Жоден стан Lumi не тече в Мул; жоден код Мулу не залежить від Lumi. Кожен працює без
  іншого.

---

## 10. Детермінізм і тести

- **Рушій:** пін стану поля після K тіків від фіксованих `(seed, genome)` — точний float-збіг
  (виставити режим FFT/точності для відтворюваності). `mutate`/`cross` детерміновані під seeded RNG.
- **Оцінювач:** пін кожної метрики проти фіксованих записаних прогонів; збурення `resilience` seeded.
- **Переграш:** знімок + лог подій відтворюють точний стан світу (реальний час відв'язаний від
  обчислення).
- **API/контракт:** події дій застосовуються на межах тіку; авторизація ізолює ідентичність
  садівника, але не спільне поле; `predict` повертає стабільний hit/miss.
- **Жодних платних викликів, жодної моделі** ніде на сервері — це чиста симуляція.

---

## 11. Порядок збірки (пропонований)

1. **Рушій** — почати з **GOL `step()`** + цілочисельний геном + mutate/cross (§2a), headless і
   unit-тестований: детермінований кор. **Крок Lenia — пізніша заміна** за тим самим інтерфейсом,
   щойно скелет нижче доведений на GOL.
2. **Оцінювач** (метрики + predict) на записаних прогонах.
3. **Стор** (знімки + лог подій + переграш).
4. **Цикл тіку** (реальний час, детермінований через лог подій) + **REST API**.
5. **WebSocket-стрім** + **веб-UI** (рендер, тоді дія, тоді інспекція).
6. **Клієнт Лілі** (шість інструментів + predict + events-since) — вшитий у Lumi як зовнішній світ.

---

*Мул — спільний сад. Сервер тримає його живим; веб-канвас дає тобі його доглядати; API дає
Лілі доглядати його поряд із тобою. Двоє садівників, одне поле, і світ, що росте в паузах, байдуже,
чи хтось дивиться.*
