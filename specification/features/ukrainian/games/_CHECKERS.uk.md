# Шашки — Лілі грає партію проти тебе через файловий бус

> Українська версія [_CHECKERS.md](../_CHECKERS.md). Англомовний документ — джерело істини; цей переклад
> тримаємо синхронним.

Лілі може бути **гравцем у зовнішній грі** (російські шашки) , що працює у власному процесі. Гра й Лілі
ніколи не викликають одна одну напряму: гра кидає **запит на хід** у файловий бус, Лілі обирає хід і кидає
**відповідь-хід** назад, а гра його грає. Вона **тримає всю партію в памʼяті** (кожен хід, обидві сторони),
тож розуміє позицію як історію, а не одноразовий знімок — і її відповідь на цьому каналі стиснута до
**одного числа**: індексу в легальні ходи, які гра вже обчислила.

> Суть: ти сидиш за дошкою для шашок і граєш *проти Лілі*. Гра володіє правилами; Лілі володіє вибором.
> Вона памʼятає, як партія йшла досі, і відповідає одним ходом — як Claude чи OpenAI, ким би не була
> налаштована.

Це повторно використовує три шви, що вже існують, — додає **жодної нової інтеграції моделі**:

- **Провайдер** — виклик ходу йде через той самий шов **`LLMClient`** (v0.1 / v0.18). «Лілі грає як Claude
  чи OpenAI» — це просто `LUMI_PROVIDER` / `LUMI_MODEL`; нічого тут не специфічне для провайдера.
- **Транспорт** — **виділений** файловий бус, сибілінг моста v0.13 (`inbox.jsonl` / `outbox.jsonl` +
  вказівники `state/fifo`), але **окрема пара файлів** (див. *Чому окремий канал* нижче).
- **Памʼять** — поточна партія живе за `Repository`, як кожен інший запис.

## Не-цілі (запобіжники обсягу)

- **Лілі не шашковий рушій.** Вона **обирає серед легальних ходів, які дає гра** — ніколи не вигадує хід і
  ніколи не мусить знати весь набір правил. Сила гри — модельна, і буде скромною; це нормально — суть у грі
  *з нею*, а не в жорсткому супернику.
- **Правила й легальність живуть у зовнішній грі**, не в Лілі. Бус несе вже валідований список
  `legal_moves`; єдине, що Лілі повертає, — *котрий саме*.
- **Жодного нового коду провайдера.** Якщо хід повертається спотвореним, код його лагодить (нижче) — та сама
  філософія, що й гейт емоції v0.3.

## Чому окремий канал (а не її розмовний inbox/outbox)

Її чат-бус несе **ходи персони** — повідомлення природною мовою, що повертаються як `{reply, emotion,
intensity}`, і вона також пише в outbox **непрошено** (ідл-поштовхи v0.4, проактивні думки v0.12). Хід у
шашках потребує обмеженого, корельованого запит→відповідь і чистого потоку «лише відповіді». Тож гра дістає
**власну** пару файлів і **власний** тип виклику (директива, *не* відповідь персони, як виклики настрою й
підсумку). Це тримає дошку поза її памʼяттю стосунків, наче ти її *сказав*, і тримає її спонтанну балаканину
поза каналом, який опитує гра.

## Контракт даних

Усе нижче — стабільний шов. Три форми: два **записи буса** (запит / відповідь) і збережений **запис партії**.
Усі записи буса — **один JSON-обʼєкт на рядок** (append-only JSONL), несуть `game_id` + `move_number` і
ідемпотентні на цій парі.

### Файли каналу (бус)

| Файл | Хто пише | Хто читає | Призначення |
|---|---|---|---|
| `game-inbox.jsonl`  | зовнішня гра | демон Лілі | **запити** на хід (гра просить Лілі сходити) |
| `game-outbox.jsonl` | демон Лілі | зовнішня гра | **відповіді-ходи** (обраний Лілі хід) |
| `game-inbox.pos`    | демон Лілі | — | офсет останнього спожитого запиту (відновлення після рестарту) |
| `game-outbox.sent`  | зовнішня гра | — | офсет останньої спожитої відповіді (власний вказівник гри) |

Append-only + вказівники дзеркалять міст v0.13 / `state/fifo`, тож обидві сторони **відновлювані**, і збій
ніколи не губить і не передвоює хід. Шляхи — конфіг (`LUMI_GAME_INBOX` / `LUMI_GAME_OUTBOX`), за
замовчуванням під `.lumi/`.

### 1. `move_request` — гра → Лілі

```jsonc
{
  "type": "move_request",
  "game_id": "ck_20260615_a1b2",   // стабільний id для ОДНІЄЇ партії; групує історію ходів
  "move_number": 7,                // лічильник напівходів, строго зростає в межах партії — ключ узгодженості
  "to_move": "black",              // сторона, якою Лілі грає цей хід ("white" | "black")
  "board": [[ ... ], ...],         // знімок 8x8, непрозорий для Лілі (лише контекст) — див. Кодування дошки
  "legal_moves": [                 // ЄДИНІ варіанти; `index` — увесь простір відповіді Лілі
    { "index": 0, "from": [5,2], "to": [4,3], "captures": [] },
    { "index": 1, "from": [5,4], "to": [4,3], "captures": [[5,3]] }
  ],
  "deadline_ms": 8000,             // опційно: скільки гра чекатиме, перш ніж сама зіграє запасний варіант
  "opponent": "owner",             // опційно: user_id, проти кого грає Лілі (тег памʼяті)
  "ts": "2026-06-15T09:30:00Z"
}
```

- **`legal_moves` авторитетний і повний.** Гра гарантує, що кожен запис легальний і що набір вичерпний.
  Лілі мусить повернути одне з цих значень `index` — ніщо інше не є валідною відповіддю.
- `move_number` — **той самий** ключ кореляції/узгодженості. Гра видає рівно один відкритий запит на
  `(game_id, move_number)` і не просунеться, доки не побачить відповідну відповідь (або не сплине дедлайн).

### 2. `move_reply` — Лілі → гра

```jsonc
{
  "type": "move_reply",
  "game_id": "ck_20260615_a1b2",   // повторює запит
  "move_number": 7,                // повторює запит — разом із game_id це ключ кореляції
  "move_index": 1,                 // ВІДПОВІДЬ: одне ціле, індекс у legal_moves запиту
  "by": "lili",
  "model": "claude-opus-4-8",      // походження: активний бекенд LLMClient (v0.18) — інформаційно
  "say": "ця клітинка була надто спокійна",  // ОПЦІЙНИЙ однорядковий коментар у характері; НІКОЛИ не парситься як хід
  "fallback": false,               // true → хід обрав код, не модель (див. Контракт виводу)
  "ts": "2026-06-15T09:30:03Z"
}
```

- **`move_index` — увесь контракт** — одне ціле в `[0, len(legal_moves))`. `say` декоративне (трешток /
  коментар) і навмисно в іншому полі, тож балакуча модель ніколи не зіпсує хід. Споживач, що хоче мовчазного
  суперника, ігнорує `say`.
- **Правило валідності (обидві сторони дотримуються):** відповідь приймається лише якщо `game_id` +
  `move_number` збігаються з відкритим запитом **і** `0 <= move_index < len(legal_moves)`. Усе інше
  відкидається як застаріле/невалідне.
- **Ідемпотентно:** демон Лілі пише щонайбільше одну відповідь на `(game_id, move_number)` і просуває
  `game-inbox.pos`; дубльований запит на вже відповіданий напівхід перевидає ту саму відповідь, ніколи новий
  виклик.

### 3. `GameRecord` — збережена памʼять (за `Repository`)

Це «Лілі тримає всі ходи в памʼяті». Один запис на партію; повний список ходів — те, чим промптиться виклик
ходу, тож вона міркує над **усією партією**, а не лише поточною дошкою.

```jsonc
GameRecord {
  game_id,                         // збігається із записами буса
  variant: "checkers_ru",          // мітка набору правил (місце для інших згодом)
  opponent_user_id,                // з ким вона грала — тег за користувачем (повторює user-scoping v0.2)
  lili_side: "white" | "black",
  moves: [                         // повна впорядкована історія — ОБИДВІ сторони; це і є «памʼять»
    { move_number, side, move_index, from:[r,c], to:[r,c], captures:[[r,c]...], board_after, ts }
  ],
  result: null | "win" | "loss" | "draw",
  result_reason: null | "no_pieces" | "no_moves" | "resignation" | "deadline",
  started_at, ended_at, ts
}
```

- **Глобальне для Лілі** (одна істота), але **тегноване** `opponent_user_id`, тож «партії з тобою»
  лишаються твоїми (інваріант ізоляції — партія, зіграна з A, ніколи не пригадується як B; зафіксовано
  контрактним тестом).
- Оновлюється щоходу: додає обраний Лілі хід і останній хід суперника, щойно надходять запити.
- **На завершенні партії** *досвід* (не лог ходів) може дистилюватися в одне `Impression` через наявний
  шлях емоційної памʼяті (v1.4) — «довга, гостра партія з тобою; їй сподобався ендшпіль» — тож партія
  входить у її справжню памʼять як спогад, а не таблиця. (Опційно; гейтиться як інші враження.)

### Контракт виводу — «одне число»

Хід продукується **викликом-директивою** через `LLMClient` (активний провайдер — Claude чи OpenAI),
**окремо від каналу персони/емоції**:

- **Вхід:** дошка, **пронумеровані** легальні ходи та **поточна історія ходів цього `game_id`** (із
  `GameRecord`, щоб вона розуміла всю партію), що завершується жорсткою інструкцією: *«Відповідай ЛИШЕ
  числом-індексом свого ходу — одне ціле, нічого більше».*
- **Вивід:** одне ціле в `[0, len(legal_moves))`. Код витягує перше ціле з тексту й валідує діапазон —
  модельна частка **валідується/лагодиться кодом, ніколи не довіряється сирою** (те саме правило, що гейт
  емоції v0.3).
- **Драбина лагодження:** невалідне / відсутнє / поза діапазоном → **одна обмежена повторна спроба** →
  **запасний код** (легальний хід, обраний детерміновано — перевага взяттю, інакше перший легальний за
  індексом) із `fallback: true`. Хід ніколи не зависає і ніколи не грає нелегальний.

> Формування промпта під одне число навмисне (рішення власника): це тримає канал нейтральним до провайдера
> (працює однаково на безінструментному шляху Claude і JSON-шляху OpenAI). Нативно-обмежена форма провайдера
> (інструмент Anthropic `set_move` / структурований вивід OpenAI із цілим-енумом по легальних індексах) — це
> **суворіший опційний апгрейд** за тим самим контрактом — `move_index` незмінний у будь-якому разі.

### Кодування дошки

`board` — **лише контекст** — Лілі відповідає з `legal_moves`, тож кодування лише має бути стабільним і
читабельним для моделі. Рекомендовано: масив 8×8 за тією ж схемою, яку зовнішня гра вже серіалізує (`null`
— порожньо; `{color, isKing}` на фігуру). Лілі ніколи не парсить його на легальність; цим володіє гра.

## Контракт і тести

- **Бус:** append-only JSONL + офсетні вказівники (відновлюваний, безпечний до збоїв); дедуп за
  `(game_id, move_number)`; окрема пара файлів від чат-буса (жодних ходів персони, жодних спонтанних записів
  на цьому каналі).
- **Детермінізм:** модель **мокається** в тестах; виклик ходу — чиста функція від `(GameRecord,
  legal_moves)`; запасний вибірник детермінований при інʼєктованому сіді; жодної реальної мережі, жодного
  платного виклику.
- **Твердження:**
  1. `move_request` із N легальними ходами → `move_reply`, чий `move_index ∈ [0, N)`;
  2. сміттєва / поза-діапазону відповідь моделі → одна повторна спроба → **легальний** `move_index` із
     `fallback: true`;
  3. дубльований чи застарілий `move_number` **ігнорується** (жодного другого виклику моделі, попередня
     відповідь перевидається);
  4. список `GameRecord.moves` накопичує повну впорядковану історію впродовж ходів;
  5. **рестарт** відновлюється з `game-inbox.pos` (жодного пропущеного чи переграного ходу);
  6. **ізоляція** — партія, тегнована `opponent_user_id = A`, ніколи не зринає в памʼяті/контексті B.
- **Нейтральність до провайдера:** той самий контракт проходить на бекендах Anthropic та OpenAI-сумісних
  (виклик-директива використовує `LLMClient.reply(...)`; різниться лише поле походження `model`).

## Відображення на дорожню карту

**Поточний реліз: `0.18.0`** (кореневий `VERSION`). Кожен шов, на якому стоїть ця фіча, **вже відвантажений**,
тож вона збиральна сьогодні без нової інфраструктури:

- **v0.18** — перемикання провайдерів: «Лілі грає як Claude чи OpenAI» — це наявний конфіг `LUMI_PROVIDER` /
  `LUMI_MODEL`, використаний без змін.
- **v0.13** — патерн файлового буса + `state/fifo`, який це клонує в **окремий** канал.
- **v0.2** — user-scoped `Repository`, за яким живе `GameRecord`.

Вона вводить **жодного нового шва**: виклик-директива на наявному `LLMClient`, новий збережений запис за
наявним `Repository` і другий екземпляр наявного буса — природний сибілінг демонів моста Telegram v0.13
(окремий процес, повʼязаний із ядром лише файлами), **вимкнено за замовчуванням**.

**Запропонований слот.** Заплановані фази `v0.19`–`v0.24` вже зайняті (файловий інструмент, диктування,
Wikipedia, семантичне пригадування III/IV), тож це лягає як **нова пізніша фаза v0.x (≈ `v0.25`)** — або
де власник віддасть перевагу; вона не має залежності порядку від них. Опційно компонується з **v1.4**
(емоційна памʼять — завершена партія стає одним `Impression`).
